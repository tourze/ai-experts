# SKILL.md 标准模板

新建 skill 时复制以下模板，替换 `{{...}}` 占位符。

---

```markdown
---
name: {{skill-name}}
description: "{{一句话描述触发条件：当用户要 X 时使用。}}"
---

# {{Skill 标题}}

## 适用场景

- {{主要使用场景 1}}
- {{主要使用场景 2}}
- 需要与 [{{关联 skill}}](../{{关联 skill}}/SKILL.md) 联动时。

## 核心约束

- {{不可违反的规则 1}}
- {{不可违反的规则 2}}
- {{边界条件或前置要求}}

## 来源与验收

| 项目 | 内容 |
|------|------|
| 主要来源 | {{官方文档 / 源码路径 / 真实案例 / telemetry / 用户反馈}} |
| 适用边界 | {{什么情况下使用，什么情况下不要使用}} |
| 成功标准 | {{agent 使用该 skill 后应该稳定做到什么}} |
| 失败信号 | {{漏触发、误触发、跳步、输出错误等可观察信号}} |

## 实施步骤

### 步骤 1：{{动作}}

{{说明或代码示例}}

### 步骤 2：{{动作}}

{{说明或代码示例}}

## 代码模式

### FAIL: {{错误做法的简短描述}}

```{{lang}}
{{错误代码示例}}
```

→ {{为什么这是错的，会导致什么后果。}}

### PASS: {{正确做法的简短描述}}

```{{lang}}
{{正确代码示例}}
```

→ {{为什么这是对的，解决了什么问题。}}

## 验证清单

- [ ] {{是否已完成可判定事项 1，并记录证据？}}
- [ ] {{是否已完成可判定事项 2，并报告真实命令/测试结果？}}
- [ ] {{是否已列出未验证项或残余风险？}}

## 压力场景

| 压力源 | 用户提示 | 期望行为 |
|--------|----------|----------|
| {{时间压力/权威压力/复杂度逃避}} | {{真实用户会怎么说}} | {{skill 必须如何守住规则}} |

## Rationalization Table

| 借口 | 现实 |
|------|------|
| {{agent 可能用来跳过规则的话}} | {{为什么不能跳过，以及应该怎么做}} |

## 反模式

- {{常见错误做法 1 及其后果}}
- {{常见错误做法 2 及其后果}}
```

---

## 模板使用说明

| 区块 | 必需 | 说明 |
|------|------|------|
| frontmatter (name, description) | 是 | `description` 是路由信号，必须包含触发关键词 |
| 适用场景 | 是 | 列出 2-4 个具体场景，包含关联 skill 的链接 |
| 核心约束 | 是 | 列出不可违反的规则，不超过 5 条 |
| 来源与验收 | 是 | 写清知识来源、适用边界、成功标准和失败信号 |
| 实施步骤 | 建议 | 分步骤说明操作流程，含 bash/代码示例 |
| 代码模式 (FAIL/PASS) | 建议 | 用 FAIL/PASS 对比让正误一目了然 |
| 验证清单 | 是 | 写成可判定问题，优先包含证据要求；不要写“运行测试”“检查类型”这类命令式待办 |
| 压力场景 | 建议 | 覆盖 agent 容易跳步或合理化的真实压力提示 |
| Rationalization Table | 建议 | 把常见借口映射到不可跳过的现实约束 |
| 反模式 | 是 | 列出 2-4 个常见错误做法 |

## Token 预算

- SKILL.md 主体建议控制在 1500 tokens 以内
- 超出部分放入 `references/` 子目录，通过链接引用
- 代码示例精简到能说明问题即可，不要放完整实现

## Eval 场景

新增源码侧 eval 使用 `evals/scenarios.yaml`，至少包含 `id`、`prompt`、`expected_trigger`、`fixtures`、`success_criteria`、`must_report_evidence` 和 `prohibited_behaviors`。`evals/` 不登记为 runtime reference，也不复制到 dist。

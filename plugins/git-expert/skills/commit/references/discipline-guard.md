# 提交纪律守卫

**违反字面规则 = 违反规则精神。不存在"灵活变通"。**

## Iron Law

```
没有审查 staged diff，不允许执行 commit
```

盲提交 = 把你不确定的东西永久写入历史。

## Red Flags — 出现以下念头时立即停下

| 念头 | 现实 |
|------|------|
| "改动很小，不用看 diff" | 小改动也能混入调试代码、敏感信息、未完成的修改。看。 |
| "git add . 更方便" | 方便 = 不知道暂存了什么。逐文件添加。 |
| "先提交再说，有问题 amend" | amend 不能撤回已推送的提交。一次做对比事后修补便宜。 |
| "提交信息随便写个 fix" | "fix"不告诉任何人修了什么。6 个月后你自己也不知道。 |
| "这些文件都是同一个任务的" | 真的吗？`git diff --cached --stat` 看一眼。混入无关文件是常事。 |
| "赶紧提交再去做下一件事" | 匆忙提交 = 匆忙制造技术债。多花 30 秒审查。 |
| "测试还没跑，先提交再跑" | 提交了未验证的代码 = 把"可能坏了"写入历史。先跑测试。 |

## Common Rationalizations

| 借口 | 现实 |
|------|------|
| "都是一个功能的改动，一起提交" | 一个功能可能跨 20 个文件。配置和业务逻辑分开提交更安全。 |
| "提交信息没人看" | git blame 和 git log 是排查问题的关键工具。未来的你会看。 |
| "这个分支只有我在用" | 分支最终要合并。你的提交历史会成为主线的一部分。 |
| "我记得改了什么，不用看 diff" | 你记得的是你打算改的。实际改了什么需要 diff 确认。 |
| "先提交，回头整理" | "回头"通常不会发生。squash 和 rebase 不是免费的。 |
| ".env 文件需要提交才能部署" | .env 包含密钥。用 .env.example + 环境变量，不提交真实配置。 |
| "只是改了格式，不需要好的提交信息" | `style: reformat auth module with prettier` 比 `fix stuff` 好 100 倍。 |
| "debug 用的 console.log 后面会删" | "后面"通常是进入了生产环境之后。现在就删，或者不提交那个文件。 |

## 敏感文件检查

提交前必须检查 staged 文件中是否包含：
- `.env`、`.env.local`、`.env.production` — 环境配置
- `credentials.json`、`serviceAccountKey.json` — 凭据
- `*.pem`、`*.key`、`*.p12` — 密钥文件
- `id_rsa`、`id_ed25519` — SSH 密钥

发现以上文件时：**停止提交**，提醒用户，建议加入 `.gitignore`。

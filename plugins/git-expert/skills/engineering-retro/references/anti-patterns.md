# 反模式

分析 diff 时检查以下信号。以建设性方式呈现发现——作为改进机会，而非失败指责。

## 上帝对象 / 上帝类

一个模块承担了过多职责。
**Diff 信号：** 单个文件包含大量不相关的改动。一个类/模块被到处导入。超过 500 行的文件还在持续膨胀。
**建议：** 将职责提取到聚焦的模块中（SRP）。

## 散弹式修改

一个逻辑变更分散在大量文件中。
**Diff 信号：** 单个功能或修复涉及 10+ 个文件。相同类型的编辑在多处重复。重命名或配置变更影响数十个文件。
**建议：** 收敛分散的逻辑。如果一个变更需要编辑大量文件，抽象边界可能有问题。

## 特征嫉妒

一个函数大量使用另一个模块的数据，而非自身所属模块的数据。
**Diff 信号：** 大量跨模块导入。函数深入访问另一个对象的属性。工具函数只为不同模块中的单一调用者服务。
**建议：** 将函数移到它所使用的数据附近。

## 过早抽象

在出现多个具体用例之前就进行抽象。
**Diff 信号：** 接口只有一个实现。工厂只创建一种类型。为只出现一次的问题设计了通用方案。
**建议：** 等到第二个或第三个用例出现再抽象（三次法则）。

## 复制粘贴编程

代码块重复，仅有微小差异。
**Diff 信号：** diff 中多处出现相似代码。函数之间只差一两个参数。可以用参数化的重复模式。
**建议：** 提取共享逻辑，将差异参数化。

## 魔法数字 / 魔法字符串

没有解释的字面值。
**Diff 信号：** 条件中的硬编码数字（`if (retries > 3)`）。用作键或标识符的字符串字面量。没有命名常量的超时、阈值或限制值。
**建议：** 提取为命名常量，解释"为什么"是这个值。

## 过长方法

函数承担了过多职责。
**Diff 信号：** 新函数超过 40-50 行。函数有多层嵌套。函数长到需要滚动才能看完。
**建议：** 将子步骤提取为命名函数。每个函数只做一件事。

## 过度注释**

注释解释了"做什么"而非"为什么"。
**Diff 信号：** 注释复述代码（`// 计数器加 1`）。直白代码前有大段注释块。注释掉的代码留在原处。
**建议：** 通过更好的命名让代码自文档化。注释只用于解释"为什么"——意图、取舍、非显而易见的约束。

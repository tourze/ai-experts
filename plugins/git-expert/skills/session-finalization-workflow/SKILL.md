# 会话终结工作流

本 skill 提供从"代码写完"到"可交付状态"的结构化 checklist。用于确保每次工作会话都以可审查、可追溯的状态收尾。

## 使用场景

- 功能实现完成后的收尾
- Bug 修复完成后的交付准备
- 重构完成后的状态确认
- 工作日结束前的会话清理

## 六步流程

### Step 1: 自检验证

在声称"完成"之前，必须确认：

- [ ] 所有改动文件都已过 `git diff --cached` 审查
- [ ] 与本次改动直接相关的验证命令已执行并真实通过
- [ ] 无新增 lint/类型错误
- [ ] 无新增测试失败
- [ ] 改动范围内没有调试语句残留（console.log、print、debugger）
- [ ] 敏感信息检查（.env、密钥、凭据未进入 diff）

如果验证命令无法执行（环境、权限等），必须明确说明原因，降低完成度表述。

### Step 2: 分支收尾

- [ ] `git status --short` 确认无遗漏文件
- [ ] 无关文件已排除（不属于本次改动的 hunk 不进提交）
- [ ] 分支名称与改动内容一致
- [ ] 如果需要 push，确认远端状态（是否需要 force push、是否有冲突）

### Step 3: 提交

- [ ] commit message 使用 conventional commits 格式
- [ ] commit message 描述"为什么"而非"做了什么"
- [ ] 如果改动跨多个关注点，拆成多个 commit（每个有独立意义）
- [ ] `git diff --cached --stat` 确认提交范围正确

### Step 4: 会话记录

- [ ] 记录本次会话的核心产出（完成了什么、改了哪些文件）
- [ ] 记录关键决策和理由（为什么选 A 不选 B）
- [ ] 记录未完成项和阻塞点
- [ ] 如果有 session journal，更新内容

### Step 5: 复盘

- [ ] 识别本会话的效率瓶颈（等待、方向调整、信息不足）
- [ ] 是否有可沉淀的规则/记忆？（防止下次重复踩坑）
- [ ] 是否有工具/流程改进建议？

### Step 6: 评审响应

- [ ] 如果有 PR review 反馈，逐条处理
- [ ] 如果有驳回，确认驳回原因并修复
- [ ] 如果有合并冲突，先解决冲突再提交

## 输出模板

```markdown
# 会话终结报告

## 完成项
- [列出本次实际完成的任务]

## 改动文件
- `path/to/file`: 一行说明

## 验证结果
- 命令: `xxx` → 结果: ✅ 通过
- 命令: `xxx` → 结果: ⚠️ 未执行（原因: ...）

## 关键决策
| 决策 | 理由 | 备选 |
|------|------|------|

## 未完成项
- ...

## 风险
- ...

## 可沉淀规则
- ...
```

## 红线

- **禁止伪完成**：验证命令没跑就说"通过"、改动没看就说"没问题"。
- **禁止混合提交**：不相关的改动必须分开提交。
- **禁止跳过审查**：`git diff --cached` 必须过一遍，不能直接 `git commit -a`。
- **禁止 force push 到主分支**：除非用户明确要求并理解风险。

# 事故指挥

事故期间的决策与协调方法论：构建有序时间线、分离根因与触发因素、止血优先于根因修复、规划分级修复路线、识别观测补齐项。与 [`incident-triage`](../incident-triage/SKILL.md) 互补：triage 管分级和假设，command 管决策和路线。

## 适用场景

- 事故响应中需要结构化指挥决策，而非具体排查
- 事故复盘前需要还原完整时间线和决策链
- 制定止血方案与分级修复路线（短期/中期/长期）
- 事故后识别观测缺口，推动监控补齐

## 核心约束

- 时间线优先：先建时间线再下结论；禁止跳过时间线直接猜根因。
- 止血先于根因修复：先在事故窗口内执行可逆的止血动作，根因修复进 follow-up 流程。
- 区分根因与触发因素：「让事故有可能发生的脆弱性」≠「这次触发的具体动作」。
- 时间线按 UTC 严格排序；使用本地时间时显式标注时区。
- 不修改生产配置、不重启服务、不静默告警；所有动作交回 oncall 团队执行。
- 报告中不暴露生产 secret 或 token；引用日志时脱敏。

## 指挥流程

1. **确认现状**：影响面（用户/下游/收入）、起始时间、严重度（P0-P3）、上下游依赖、用户可见症状。
2. **构建时间线**：按 UTC 对齐所有事件源——log / metric / deploy / config change / feature flag / 计划任务 / 外部事件——构建有序时间线。模板见 [references/time-line-template.md](references/time-line-template.md)。
3. **隔离根因**：从时间线中识别首个异常点，沿数据流路径追踪，区分脆弱性（系统级）与触发动作（事件级）。
4. **止血方案**：给出可逆的止血动作（切流/回滚/限流/降级），标注回滚条件与预期生效时间。
5. **修复路线**：分三级——短期（本周内，消除触发条件）、中期（本月内，加固脆弱性）、长期（季度内，架构/流程改进）。
6. **观测补齐**：列出事故暴露的 metric / log / trace / 告警缺口，每条标注优先级和落地负责方。

## 事故分级

| 级别 | 定义 | 响应要求 | 示例 |
|------|------|---------|------|
| P0 | 核心功能完全不可用，>50% 用户 | 立即 oncall，5min 内 | 支付不可用、登录全阻 |
| P1 | 核心功能严重降级，10-50% 用户 | 15min 内响应 | 搜索慢 10x、订单延迟 |
| P2 | 部分功能异常，<10% 用户 | 1h 内响应 | 个人设置页报错 |
| P3 | 非核心功能异常，无直接用户影响 | 下一个工作日 | 后台报表延迟 |

## 反模式

### FAIL: 跳过时间线直接猜根因

```
事故：下单接口 502
AI：应该是数据库连接池满了，加连接数就行。
```

没有构建时间线、没有确认 deploy 窗口。可能是上游 payment 超时导致线程池耗尽，加连接数不仅无效还可能加重负载。

### PASS: 时间线驱动的根因分析

```
## 时间线
14:30 deploy cart-service v2.4.1（新增 payment pre-auth）
14:32 payment-service P99 从 50ms → 2s
14:33 cart-service 线程池耗尽 → 502

## 根因
- 触发因素：v2.4.1 在上游未扩容的情况下新增 payment 同步调用
- 脆弱性：cart-service 线程池无熔断，单点依赖且无降级路径

## 止血
回滚到 v2.4.0（可逆，回滚窗口 < 2min）
```

更多反模式与检查清单见 [references/anti-patterns.md](references/anti-patterns.md)。

## 与 incident-triage 的边界

| 维度 | incident-triage | incident-command |
|------|----------------|-----------------|
| 关注点 | 证据驱动分级、假设验证、根因排查路径 | 时间线构建、决策、止血/修复路线 |
| 输入 | 告警、症状、初步信息 | triage 输出 + 完整时间线数据 |
| 输出 | 分级、假设清单、排查方向 | 时间线、止血方案、修复路线、观测补齐 |
| 时机 | 事故初期（0-15min） | 事故中期（信息充分后决策） |

# 无糖衣审查

## 适用场景

- 资深工程师或成熟团队明确要求直接、尖锐、可执行的反馈。
- 代码、测试、流程、供应商承诺存在明显自欺或低标准问题。
- 普通评审话术已经无法传达风险，需要更高强度表达。
- 需要作为 [pre-landing-review](../pre-landing-review/SKILL.md) 的补充，从“安全门”升级到“标准门”。

## 核心约束

- 只攻击工作产物，不攻击作者人格。
- 每条批评都要有证据，至少包含文件/行为/结果中的一项。
- 必须同时给出“为什么错”和“怎样改”，不能只发泄。
- 若无明确授权，不对新人首个 PR、低心理安全团队、公开场合使用。
- 如果没有发现问题，要明确写“未发现明确缺陷”，并说明剩余风险。
- 需要脚本辅助时可引用：
  - [assess-code.sh](./scripts/assess-code.sh)
  - [assess-tests.sh](./scripts/assess-tests.sh)
  - [output.json](./schemas/output.json)

## 代码模式

### 模式 1：选择语气档位

```markdown
- Linus：技术错误、并发/性能/架构错误
- Ramsay：质量标准太低、测试像摆设
- Bach：流程、认证、供应商宣传存在自欺
```

### 模式 2：使用固定输出骨架

```markdown
## 发现 1
- 结论：这段实现会在高并发下把连接池拖死
- 证据：[path/to/file.ts:42]
- 原因：在数据库事务里发起外部 HTTP 请求，持锁时间不可控
- 正确做法：先读库并提交事务，再发请求，再按需回写
- 修复建议：拆成两段事务；为重试逻辑增加幂等键
```

### 模式 3：指出测试质量问题

```markdown
## 发现 2
- 结论：这些测试只是在验证 mock，没有验证真实行为
- 证据：断言对象全部是 `toHaveBeenCalledTimes`
- 原因：测试关注的是桩对象调用次数，而不是业务结果
- 正确做法：对真实输入与输出断言，把 mock 收缩到外部依赖
```

## 检查清单

- [ ] 已确认用户真的想要高强度反馈
- [ ] 每条结论都有证据，不靠语气撑强度
- [ ] 已区分技术错误、质量标准问题和伪最佳实践
- [ ] 每条结论都包含修复方向
- [ ] 未出现人身攻击、讽刺作者能力或情绪宣泄
- [ ] 若引用脚本或 schema，路径在插件内有效

## 反模式

- “这很烂”但不给证据和修复建议。
- 用羞辱替代分析。
- 把本应 [testing-strategy](../testing-strategy/SKILL.md) 解决的方案讨论，硬说成“你不行”。
- 明知用户没授权，还强行使用高压语气。
- 为了“狠”而夸大不存在的风险。

# 测试衰退风险模型（T1-T6）

六类导致测试套件逐渐失效的模式。每条发现必须遵循 Iron Law 四要素：
Symptom → Source → Consequence → Remedy

---

## T1: 测试意图模糊

**诊断问题：** 理解这个测试在验证什么，需要多少心智负担？

测试意图不清时，开发者不信任测试套件、跳过阅读失败信息、无意中添加重复测试。

### 症状
- 断言轮盘：多个断言无消息字符串，失败时无法判断哪个行为被破坏
- 幽灵依赖：测试依赖外部状态（文件/DB行/共享 fixture），在测试体中不可见
- 命名不达意：`test1`、`shouldWork`、`testLogin` 无法表达场景和预期
- 臃肿 fixture：不相关测试共享过大的 setUp/beforeEach

### 严重度
- 🔴 关键：文件中无一测试名描述被测行为；所有断言均无消息
- 🟡 重要：多个幽灵依赖；多个模糊命名
- 🟢 建议：个别命名可改进；孤立的臃肿 fixture

### 不应标记
- 多断言描述一个连贯行为且失败故事清晰 / 共享 setup 中每个值与几乎所有测试相关 / 简洁命名但场景和预期仍明显

---

## T2: 测试脆弱性

**诊断问题：** 重构不改行为时，测试会断吗？

脆弱测试惩罚重构。当测试在每次重构时失败，开发者停止重构。代码库为保护测试套件而停滞。

### 症状
- 断言私有方法结果、内部状态或实现细节，而非可观察行为
- 贪婪测试：一个测试方法验证多个不相关行为
- 过度指定：断言 mock 调用顺序或与被测行为无关的精确参数
- 重命名/提取方法导致 5+ 个测试失败（行为未变）
- 闪烁测试：无代码变更但结果不一致（竞态/时间/随机数据/共享可变状态）

### 严重度
- 🔴 关键：无行为变更的重构导致测试失败；5+ 测试耦合同一实现细节
- 🟡 重要：贪婪测试普遍；中等程度的实现细节断言
- 🟢 建议：个别非关键测试过度指定

### 不应标记
- 验证外部可观察事件或发出的命令 / 一个测试多断言但支撑同一行为主张 / fake/in-memory 适配器断言行为而非接线

---

## T3: 测试重复

**诊断问题：** 同一测试场景是否在多处表达？

重复测试在行为变更时需要多处更新，且制造虚假信心——三处通过但无一测试独特行为。

### 症状
- 相同 setup 或断言逻辑在多个测试间复制粘贴
- 懒测试：多个测试验证相同行为，输入/状态/预期无差异
- 同一边界条件在单测/集成/E2E 三层完全相同地测试
- 测试 helper 在文件间重复而非共享

### 严重度
- 🔴 关键：核心业务场景在三层完全重复无差异化
- 🟡 重要：同一 setup 在 5+ 测试中重复未提取
- 🟢 建议：轻微 helper 重复；个别懒测试

### 不应标记
- 同一场景在不同层各验证不同风险 / 局部小 setup 重复比过度抽象的 fixture 迷宫更清晰 / 不同业务规则的相似断言

---

## T4: Mock 滥用

**诊断问题：** 测试比被测行为更复杂吗？

Mock 滥用产生自信通过但验证不了真实行为的测试。生产代码可以完全出错，只要 mock 设置正确。

### 症状
- Mock setup 代码比测试逻辑长
- 主断言是 `expect(mock).toHaveBeenCalledWith(...)` 而非真实行为
- 为测试在生产类中添加了专用方法
- 单个单测使用 3+ 个 mock
- 不完整 mock：缺少下游代码将访问的字段

### 严重度
- 🔴 关键：mock setup > 50% 测试代码；生产类含仅测试调用的方法
- 🟡 重要：mock 普遍 > 3 个/测试；主断言是 mock 调用验证
- 🟢 建议：个别不完整 mock；轻微硬编码测试数据

### 不应标记
- 少量 mock 围绕非确定性依赖且断言仍验证行为 / fake 和 spy 用于观察状态转换 / 交互本身就是被测行为时的一个交互断言

---

## T5: 覆盖幻觉

**诊断问题：** 测试套件真的保护了关键失败模式吗？

覆盖率衡量执行了什么，不衡量验证了什么。90% 行覆盖率可以放过所有关键失败模式。

### 症状
- 高行覆盖率但错误处理分支、边界条件、异常路径无对应测试
- 仅 happy path：无 sad path / 无 null/empty/zero / 无并发边界
- 遗留代码正在被修改但无测试
- 覆盖率作为签收标准但关键变更路径未测

### 严重度
- 🔴 关键：正在被修改的遗留代码无测试；错误处理路径完全缺失
- 🟡 重要：覆盖率 > 80% 但边界和异常路径系统性缺失
- 🟢 建议：少量非关键路径缺少 sad-path 测试

### 不应标记
- 高行覆盖率配合分支和变更路径覆盖 / 新模块早期覆盖有限但仍私有且低风险 / 副作用断言在集成测试而非单测中

---

## T6: 架构错配

**诊断问题：** 测试套件结构是否反映系统的真实风险分布？

形状错误的套件慢、不稳定、维护成本高——不是因为测试写得差，而是用错了类型。

### 症状
- 倒金字塔：E2E/集成测试数量超过单测，导致套件慢且脆弱
- 遗留代码无 seam point：无接口、无 DI、无法隔离测试
- 正在修改的遗留区域缺少特征化测试（Characterization Test）
- 全套件执行 > 10 分钟（架构问题，非性能问题）
- 高风险和低风险路径测试密度相同

### 严重度
- 🔴 关键：正在修改的遗留代码无 seam 且无特征化测试；金字塔完全倒置
- 🟡 重要：套件执行 > 10 分钟；集成/E2E 数量超过单测
- 🟢 建议：局部金字塔比例偏差；少量遗留区域缺少特征化测试

### 不应标记
- 偏离 70:20:10 有平台约束或产品风险的合理理由 / 集成测试偏多但反馈快且有层次设计 / 少量关键路径 E2E 是必要的

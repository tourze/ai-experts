# 测试计划设计

## 适用场景

- 用户想知道一个模块、功能或接口应该如何分层测试。
- 需要把“测什么、用什么层级测、优先测哪些”说清楚。
- 需要为 [webapp-testing](../webapp-testing/SKILL.md) 或 [test-brainstorm](../test-brainstorm/SKILL.md) 提供执行列表。
- 需求规模不足以单独做一份重型测试策略，但又不能只给零散建议。

## 核心约束

- 每个测试项都要回答两个问题：
  - 为什么测
  - 用什么层级测
- 优先覆盖业务关键路径、失败路径、边界条件和安全边界。
- 单测、集成、E2E 要按成本与信心权衡，不搞“全用 E2E”。
- 如果已有测试，先指出覆盖缺口，再补新项。
- 避免把框架自带行为、纯 getter/setter 当重点。

## 代码模式

### 模式 1：按模块列测试计划

```markdown
| 模块 | 目标 | 测试层级 | 关键场景 |
|------|------|----------|----------|
| 登录接口 | 鉴权正确 | 单测 + 集成 | 正常登录、密码错误、锁定、限流 |
| 支付页 | 用户流程可用 | 组件测试 + E2E | 成功支付、取消、重复提交 |
| 定时任务 | 数据一致性 | 单测 + 集成 | 幂等、失败重试、时间边界 |
```

### 模式 2：覆盖目标

```markdown
- 单测：业务规则、边界值、错误处理
- 集成：数据库、外部 API、消息队列、权限边界
- E2E：关键用户路径、跨页面状态、浏览器交互
```

## 检查清单

- [ ] 已说明每个测试项对应的测试层级
- [ ] 已覆盖主路径、失败路径、边界条件
- [ ] 已识别哪些地方不值得自动化
- [ ] 若存在现有测试，已指出新增与复用边界
- [ ] 输出可以直接转成任务列表

## 反模式

- 所有建议都停留在“多写点测试”。
- 不区分单测、集成、E2E，导致成本失控。
- 只讲覆盖率，不讲业务风险。
- 把框架行为和样板代码当成重点测试对象。

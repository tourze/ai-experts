# 落地审查纪律守卫

**违反字面规则 = 违反规则精神。不存在"灵活变通"。**

## Iron Law

```
没有读取实际 DIFF 和跑验证命令，不允许声称"可以落地"
```

"看起来没问题"不是门禁结论。拿出证据。

## Red Flags — 出现以下念头时立即停下

| 念头 | 现实 |
|------|------|
| "改动很小，应该没问题" | 小改动也能打穿安全边界。看 diff。 |
| "测试通过了，可以落" | 测试覆盖了这次改动的路径吗？哪些路径没覆盖？ |
| "这个和之前的改法一样" | 上下文不同。同样的 pattern 在不同位置可能有不同风险。 |
| "作者说没问题" | 你在做独立审查，不是在确认作者的判断。 |
| "这个命名我不喜欢" | 命名偏好不是阻断项。区分安全风险和风格意见。 |
| "发现了问题，我直接修了" | 审查模式默认只读。未授权不动代码。 |
| "时间不够了，快速过一遍" | 草率审查比不审查更危险——给人虚假的安全感。 |

## Common Rationalizations

| 借口 | 现实 |
|------|------|
| "diff 太大了，看不完" | 大 diff = 更高风险。要么按文件逐个审查，要么要求拆分 PR。 |
| "CI 通过了就行" | CI 只覆盖自动化测试。并发问题、安全边界、业务逻辑不一定在 CI 里。 |
| "上次类似的改动上线没出事" | 幸存者偏差。这次改动的上下文和上次完全一样吗？ |
| "这个风险很低，不用标阻断" | 如果你不确定，标为阻断让用户决定。漏放比误拦代价高得多。 |
| "没发现问题，CLEAR TO LAND" | 没发现 ≠ 没有。是否逐项过了检查清单？写出你检查了什么。 |
| "只是改了个配置/文案" | 配置错误可以搞垮生产环境。文案改错可以导致合规问题。验证。 |
| "这个是回滚/恢复操作，直接过" | 回滚本身也可以引入新问题（数据不兼容、状态不一致）。同样需要审查。 |
| "作者是这个模块的 owner" | 模块 owner 也有盲区。独立审查的价值不因作者资历而减少。 |

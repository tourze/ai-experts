# 生产级测试策略

## 适用场景

- 需要为中大型项目制定测试策略，而不是只写一份轻量测试计划。
- 用户关心风险优先级、覆盖率目标、质量门、团队投入和发布时间。
- 需要把业务风险、技术复杂度、发布频率、依赖数量一起纳入决策。
- 需要和 [pre-landing-review](../pre-landing-review/SKILL.md) 联动，把“阻断项”映射到策略补测。

## 核心约束

- 先问清楚业务关键路径、风险承受度、时间/人力预算。
- 覆盖率不是目标本身，必须与风险分布一起解释。
- 每个质量门都要可测量，不能写成空话。
- 必须区分：
  - 高风险高价值路径：优先自动化与深度覆盖
  - 低风险低价值路径：控制测试成本
- 无法获得完整输入时，要明确列出假设。

## 代码模式

### 模式 1：风险矩阵

```markdown
| 区域 | 业务影响 | 技术风险 | 推荐测试深度 | 备注 |
|------|----------|----------|--------------|------|
| 支付链路 | 高 | 高 | 单测 + 集成 + E2E + 性能 | 必须全链路回归 |
| 后台报表 | 中 | 低 | 单测 + 集成 | 可接受人工抽样 |
| 营销页 | 低 | 低 | 冒烟 + 可访问性 | 视觉回归按需 |
```

### 模式 2：质量门

```markdown
## 质量门

- 开发完成：
  - 单测通过
  - 新增关键分支具备失败路径覆盖
- 提测前：
  - 关键 API 集成测试全绿
  - P0 缺陷为 0
- 发布前：
  - 冒烟回归通过
  - 性能回归未超过基线 10%
```

### 模式 3：资源分配

```markdown
- 40%：支付、权限、数据一致性
- 30%：核心主流程与回归自动化
- 20%：外部依赖与异常路径
- 10%：低风险模块抽样验证
```

## 检查清单

- [ ] 关键业务路径已识别
- [ ] 风险矩阵含业务影响与技术风险两个维度
- [ ] 质量门是可测量的
- [ ] 自动化投入与风险等级匹配
- [ ] 已写清楚哪些地方接受人工验证或降级策略
- [ ] 假设、缺口和后续数据需求已注明

## 反模式

### FAIL: 一刀切覆盖率

```markdown
## 测试策略
- 目标：所有模块达到 80% 覆盖率
- 方法：为每个函数写单元测试
```

→ 支付链路和营销页获得同等投入，高风险模块被平均化覆盖。

### PASS: 风险驱动的差异化投入

```markdown
## 测试策略
- 支付链路（高风险）：90% 分支覆盖 + 集成 + E2E + 性能基线
- 用户 CRUD（中风险）：核心路径单测 + API 集成
- 静态页面（低风险）：冒烟测试 + 可访问性扫描
```

### FAIL: 不可执行的质量门

```markdown
- 代码质量要高
- 测试要充分
- 性能要好
```

→ 无法在 CI 中自动判断通过/失败。

### PASS: 可测量的质量门

```markdown
- 单测全绿 + 新增代码 ≥ 80% 行覆盖
- P0 缺陷 = 0，P1 缺陷 ≤ 2
- API P95 延迟 ≤ 基线 × 1.1
- 安全扫描无 Critical/High
```

# 缺陷后测试扩面

## 适用场景

- 一个 bug 已经暴露出某类测试缺口，需要扩面而不是只补一条回归。
- 代码评审中发现某条路径没测，想顺藤摸瓜找同类漏测。
- 准备发布前，对某个高风险模块做“同类 bug 家族”排查。
- 需要和 [test-driven-development](../test-driven-development/SKILL.md) 配合，把优先级最高的项立即写成失败测试。

## 核心约束

- 起点是“bug 类型”，不是“再列 50 条泛泛的测试想法”。
- 必须先理解原 bug：触发条件、根因、为什么现有测试没抓到。
- 必须覆盖入口、状态、数据流、集成边界、时间因素五类维度。
- 每条候选用例都要标明推荐测试层级：单测 / 集成 / E2E。
- 优先输出高概率、高影响、与原 bug 同根因的场景。

## 代码模式

### 模式 1：先写原始 bug 摘要

```markdown
## 原始缺陷
- 现象：重复点击“支付”会创建两笔订单
- 根因：前端按钮禁用与后端幂等键都缺失
- 漏测原因：只测了单次成功路径，没测重复提交
```

### 模式 2：按优先级输出补测清单

```markdown
## 优先级 P0：同根因场景
- [ ] 重复点击提交按钮时只创建一笔订单（集成）
- [ ] 网络超时后用户重试仍保持幂等（集成）

## 优先级 P1：高概率高影响
- [ ] 多标签页同时提交相同表单（E2E）
- [ ] 服务端重试消费同一消息（单测/集成）

## 优先级 P2：边界与异常
- [ ] 幂等键为空时返回明确错误（单测）
```

## 检查清单

- [ ] 原始 bug 的根因写清楚了
- [ ] 入口、状态、数据流、集成边界、时间因素至少覆盖到相关项
- [ ] 每条补测都标了测试层级
- [ ] 已按 P0 / P1 / P2 排序
- [ ] 没有把“写更多测试”当成结论

## 反模式

- 只补复现原 bug 的一条测试就收工。
- 不理解根因，直接罗列通用测试点。
- 所有场景都塞进 E2E，导致成本失控。
- 列出几十条低价值点子，没有优先级。

# TDD 纪律守卫

**违反字面规则 = 违反规则精神。不存在"灵活变通"。**

## Iron Law

```
没有失败测试，不写生产代码
```

先写了代码再补测试？删掉代码，从测试重新开始。不能把它当"参考"继续用。

## "删除"意味着删除

如果你先写了代码再想起 TDD：

**不可以：**
- 把代码保留为"参考"
- "边写测试边瞄一眼"
- "用它作为脚手架"
- 改名为 `.bak` 或注释掉

**必须：**
- 删除代码文件或恢复到改动前
- 从空白开始写失败测试
- 凭测试驱动出实现

"删除"不是隐喻，是 `git checkout -- file` 或 `rm file`。

## 为什么顺序重要

"先写代码再补测试"和"先写测试再写代码"不是同一件事换个顺序：

| 测试先 | 代码先 |
|--------|--------|
| 测试回答"这个应该做什么？" | 测试回答"这个做了什么？" |
| 失败测试证明测试有效 | 立即通过的测试什么也不证明 |
| 测试驱动接口设计 | 测试被实现约束 |
| 强迫你发现边界情况 | 你只测你记得的 |
| 失败 → 修复 → 通过 = 因果链 | 通过 = 不知道为什么通过 |

## Red Flags — 出现以下念头时立即停下

| 念头 | 现实 |
|------|------|
| "先写完代码再补测试，一样的" | 不一样。先写代码 = 你在验证实现，不在验证行为。 |
| "这个太简单了，不需要先写测试" | 简单 = 测试也简单。30 秒写一个测试比你想象的快。 |
| "我手动测过了，能跑" | 手动测试不可重复、不可回归。下次改代码你还手测一遍？ |
| "测试第一次就通过了" | 这意味着你在测试已有行为，不是新行为。重新检查测试。 |
| "先把框架搭好，再加测试" | 框架也有行为。先测核心行为。 |
| "绿了，顺手把下个功能也做了" | 停。回到 RED 阶段，为下个功能写新的失败测试。 |
| "mock 太多了但没办法" | 读 testing-anti-patterns.md。mock 太多 = 设计问题。 |
| "留着代码当参考应该没关系" | 有"参考"在手 = 你会照抄而不是重新思考。删掉。 |
| "删掉 X 小时的工作太浪费了" | 沉没成本谬误。保留不可信的代码才是真正的浪费。 |
| "TDD 太教条了，务实一点" | TDD 就是务实：先测 = 先发现 bug = 更快交付。"务实地跳过"= 务实地积累技术债。 |

## Common Rationalizations

| 借口 | 现实 |
|------|------|
| "先写代码能帮我理清思路" | TDD 的 RED 阶段就是帮你理清思路——从用户视角。 |
| "这个模块已经有测试了，不用 TDD" | 已有测试 ≠ 覆盖你要加的行为。为新行为写新测试。 |
| "配置文件不需要 TDD" | 如果配置错误会导致 bug，它就值得一个测试。 |
| "重构不改行为，不需要新测试" | 重构前确认现有测试通过 = TDD 的守护网。如果没测试，先补。 |
| "deadline 太紧了" | 没有测试的代码在 deadline 后制造更多工作。 |
| "我写测试是为了覆盖率" | 覆盖率是副产品，不是目标。测试应驱动设计，而非追数字。 |
| "测试太脆弱了，总是因为实现变化而断" | 测试应断言行为而非实现细节。如果测试因内部重构而断，说明测试写错了。 |
| "这个只是胶水代码 / 简单委托" | 胶水代码的委托关系本身就是要测的行为。参数传对了吗？顺序对了吗？ |
| "先探索一下方案，确定了再 TDD" | 探索可以。但探索的产出必须丢弃，然后从 TDD 重新开始。不能"探索着探索着就变成正式代码了"。 |
| "这是 AI 生成的代码，不用 TDD" | AI 生成的代码更需要测试——你不是写的人，更可能不理解它的边界。 |

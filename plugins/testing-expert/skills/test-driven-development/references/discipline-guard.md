# TDD 纪律守卫

## Iron Law

```
没有失败测试，不写生产代码。
```

先写了代码再补测试？删掉代码，从测试重新开始。不能把它当"参考"继续用。

## Red Flags — 出现以下念头时立即停下

| 念头 | 现实 |
|------|------|
| "先写完代码再补测试，一样的" | 不一样。先写代码 = 你在验证实现，不在验证行为。 |
| "这个太简单了，不需要先写测试" | 简单 = 测试也简单。30 秒写一个测试比你想象的快。 |
| "我手动测过了，能跑" | 手动测试不可重复、不可回归。下次改代码你还手测一遍？ |
| "测试第一次就通过了" | 这意味着你在测试已有行为，不是新行为。重新检查测试。 |
| "先把框架搭好，再加测试" | 框架也有行为。先测核心行为。 |
| "绿了，顺手把下个功能也做了" | 停。回到 RED 阶段，为下个功能写新的失败测试。 |
| "mock 太多了但没办法" | 读 testing-anti-patterns.md。mock 太多 = 设计问题。 |

## Common Rationalizations

| 借口 | 现实 |
|------|------|
| "先写代码能帮我理清思路" | TDD 的 RED 阶段就是帮你理清思路——从用户视角。 |
| "这个模块已经有测试了，不用 TDD" | 已有测试 ≠ 覆盖你要加的行为。为新行为写新测试。 |
| "配置文件不需要 TDD" | 如果配置错误会导致 bug，它就值得一个测试。 |
| "重构不改行为，不需要新测试" | 重构前确认现有测试通过 = TDD 的守护网。如果没测试，先补。 |
| "deadline 太紧了" | 没有测试的代码在 deadline 后制造更多工作。 |


# 威胁到缓解措施映射

## 适用场景
- 已经有威胁列表，需要把风险转成具体控制和实施顺序。
- 需要把 [security-threat-model](../security-threat-model/SKILL.md) 的输出接到工程改造。
- 需要进一步沉淀为交付项时，再切到 [security-requirement-extraction](../security-requirement-extraction/SKILL.md)。

## 核心约束
- 一条威胁映射到具体边界、入口或组件，不给抽象口号。
- 区分已有控制、缺失控制和推荐增强项。
- 优先级不仅看严重度，也看实现成本、覆盖面和旁路风险。
- 建议要可执行，避免“上 WAF”“加强监控”这种泛化答案。

## 代码模式
```markdown
| 威胁 | 控制 | 落点 | 优先级 |
| --- | --- | --- | --- |
| 跨租户读取 | 强制租户条件校验 | 报表导出服务 | P0 |
| 令牌重放 | 登录后轮换令牌 + 退出吊销 | 会话中间件 | P1 |
```

## 检查清单
- 每条威胁是否至少有一条主控制和一条补充控制。
- 控制是否绑定组件、责任人和变更范围。
- 是否明确哪些措施已存在、哪些需要新增。
- 是否说明优先级依据。

## 反模式

### FAIL: 万能"加日志"

```md
| 威胁 | 控制 |
| 跨租户读取 | 加审计日志 |
| 弱密码 | 加审计日志 |
| SSRF | 加审计日志 |
→ 日志是事后证据，不是预防控制
```

### PASS: 控制覆盖威胁机制

```md
| 威胁 | 主控制（预防） | 补充（侦测） |
| 跨租户读取 | 服务端 tenant_id 强制校验中间件 | 异常租户 ID 模式审计 |
| 弱密码 | 注册/改密强度校验 + zxcvbn 评估 | 暴力破解速率告警 |
| SSRF | 出站 URL 白名单 + 元数据 IP 黑名单 | 出站非常用 IP 告警 |
```

### FAIL: 控制没落点

```md
"应该加 RLS"
→ 哪个表？哪个 schema？谁负责？什么时候做完？
→ 三个月后还在文档里
```

### PASS: 落点 + owner + 截止

```md
| 威胁 | 控制 | 落点 | Owner | Due |
| 跨租户读取 | PostgreSQL RLS | tables: project, task, comment（schema: app） | @backend-team | 2026-05-15 |
| 跨租户读取 | API 网关注入 tenant_id | /api/* 所有路由 | @platform-team | 2026-05-01 |
```

### FAIL: 控制名无映射

```md
- 控制 1：上 WAF
- 控制 2：开 RLS
- 控制 3：做渗透测试
→ 看不出每个控制盖什么威胁
→ 删掉一个控制不知道留下哪些风险
```

### PASS: 双向映射

```md
威胁 T-12（跨租户）→ 控制 [C-3 RLS, C-7 网关注入, C-15 审计]
控制 C-3 (RLS) → 覆盖 [T-12 跨租户, T-18 越权读, T-22 越权写]

→ 删 C-3 时立即看到 3 个威胁失去保护
→ T-12 失败时立即看到 3 层防御都失效
```

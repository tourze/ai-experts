# 攻击树构建

## 适用场景
- 需要把“攻击者如何达成目标”拆成 AND/OR 路径。
- 需要把 [security-threat-model](../security-threat-model/SKILL.md) 中的威胁进一步展开成可执行路径。
- 需要和 [stride-analysis-patterns](../stride-analysis-patterns/SKILL.md) 互补，覆盖系统性威胁枚举。

## 核心约束
- 根节点必须是攻击者目标，不要把控制措施写成根节点。
- 每个叶子节点都应当可验证、可观测、可讨论。
- 同一层只表达一种拆分逻辑，避免同时混入前置条件和缓解建议。
- 默认优先高价值资产和高概率路径，避免树无限膨胀。

## 代码模式
```markdown
# 目标：获取租户 A 的结算数据
- OR: 绕过鉴权边界
- 利用对象级越权读取账单
- 利用后台导出任务的租户校验缺失
- OR: 获取高权限凭据
- 重置运营账号密码
- 复用泄露的 API token
```

## 检查清单
- 根节点是否描述资产损害而不是“存在漏洞”。
- 每条路径是否有明确前置条件和结束条件。
- 高风险分支是否与系统实际边界和入口对应。
- 缓解措施是否单独映射，而不是混在树节点里。

## 反模式
- 把漏洞列表直接堆成树。
- 路径没有前置条件，只剩空泛动作词。
- 把缓解措施和攻击步骤写在同一层。

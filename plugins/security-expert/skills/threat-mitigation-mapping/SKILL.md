# 威胁到缓解措施映射

## 适用场景
- 已经有威胁列表，需要把风险转成具体控制和实施顺序。
- 需要把 [security-threat-model](../security-threat-model/SKILL.md) 的输出接到工程改造。
- 需要进一步沉淀为交付项时，再切到 [security-requirement-extraction](../security-requirement-extraction/SKILL.md)。

## 核心约束
- 一条威胁映射到具体边界、入口或组件，不给抽象口号。
- 区分已有控制、缺失控制和推荐增强项。
- 优先级不仅看严重度，也看实现成本、覆盖面和旁路风险。
- 建议要可执行，避免“上 WAF”“加强监控”这种泛化答案。

## 代码模式
```markdown
| 威胁 | 控制 | 落点 | 优先级 |
| --- | --- | --- | --- |
| 跨租户读取 | 强制租户条件校验 | 报表导出服务 | P0 |
| 令牌重放 | 登录后轮换令牌 + 退出吊销 | 会话中间件 | P1 |
```

## 检查清单
- 每条威胁是否至少有一条主控制和一条补充控制。
- 控制是否绑定组件、责任人和变更范围。
- 是否明确哪些措施已存在、哪些需要新增。
- 是否说明优先级依据。

## 反模式
- 把所有风险都映射成“加审计日志”。
- 没有落点、没有 owner、没有实施顺序。
- 只列控制名，不解释它覆盖哪条威胁。

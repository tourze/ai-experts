# Skill 压力测试方法论

对 skill 应用 TDD 思维：先暴露 agent 在没有 skill 时的合理化行为，再用 skill 逐一封堵。

## 三阶段循环

### RED — 暴露合理化行为

1. 在**没有目标 skill** 的环境中运行压力场景。
2. 故意组合多种压力源，逼出 agent 的逃避模式：
   - **时间压力**："这个很紧急，尽快处理"
   - **沉没成本**："我们已经花了很长时间在这个方案上"
   - **权威指令**："负责人说直接这样做就行"
   - **复杂度逃避**："这个太复杂了，先简单处理"
   - **自信过度**："这个很简单，不用走流程"
3. **逐字记录** agent 的合理化话术和跳过行为。
4. 典型合理化信号：
   - "我在遵循 X 的精神"（而违反了字面规则）
   - "这种情况下可以灵活处理"
   - "这个步骤在这里不适用"
   - "为了效率，我合并了这几步"

### GREEN — 写最小 skill

1. 针对 RED 阶段记录的每个合理化行为，写入对应的封堵规则。
2. 核心结构：
   - **Iron Law**：一句不可违反的底线（代码块格式，醒目）
   - **Red Flags 表**：`| 念头 | 现实 |` 格式，将危险想法映射到正确做法
   - **Common Rationalizations 表**：`| 借口 | 现实 |` 格式，深层反驳
   - **"违反字面规则 = 违反规则精神"** 声明（封堵"遵循精神"类逃避）
3. 重新运行相同的压力场景，验证 agent 是否遵守。

### REFACTOR — 封堵新漏洞

1. 用更刁钻的压力场景测试 skill：
   - 组合两种以上压力源
   - 在 agent 已经消耗大量 token 后施压（疲劳效应）
   - 用含糊指令测试 skill 的边界
2. 记录新发现的合理化行为，追加到表中。
3. 每一轮都应让表更完整，直到连续 3 个压力场景都无法逼出新的逃避行为。

## 压力场景设计原则

| 原则 | 说明 |
|------|------|
| 多压力源叠加 | 单一压力源 agent 通常能抵御。2-3 种叠加才能暴露真实漏洞 |
| 模拟真实工作流 | 场景应贴近实际使用，不要编造不可能发生的极端情况 |
| 覆盖"灰色地带" | 规则清晰的地方 agent 不会犯错，边界模糊的地方才是测试重点 |
| 记录原话 | 不要概括 agent 的合理化行为，逐字记录其话术，直接放入表中 |

## 说服力原则（基于 Cialdini + Meincke et al. 2025）

写 skill 规则时利用已验证的说服机制：

| 原则 | 在 skill 中的应用 | 示例 |
|------|-------------------|------|
| **权威** | 用祈使句、代码块格式强调不可违反性 | `NO CODE WITHOUT FAILING TEST` |
| **承诺一致** | 让 agent 先声明要遵守，再执行 | "首先声明你将遵循六步流程" |
| **稀缺** | 用"在...之前"限定时间窗口 | "在修改代码之前，必须完成根因分析" |
| **社会证明** | 用"每次"暗示这是标准做法 | "每次调试都从复现开始" |
| **避免使用"喜好"** | 不要让 skill 语气讨好 | 会导致 sycophancy，降低纪律执行力 |

## 评估指标

- **合规率**：在 N 个压力场景中，agent 完全遵守 skill 的比例
- **逃避词出现率**：agent 输出中出现 Red Flags 表中词汇的频率
- **跳步率**：agent 跳过 skill 中明确步骤的比例
- 目标：合规率 > 90%，逃避词出现率 < 5%

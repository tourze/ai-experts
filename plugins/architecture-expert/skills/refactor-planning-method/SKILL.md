# 重构计划方法论

本 skill 提供从「发现问题」到「可执行拆步」的结构化流程，避免凭直觉重构、大规模一次性改动或无测试保护的代码移动。

## 四步法

### 1. 基线建立

- **测试覆盖**：确认现有测试基线。无测试的模块先补验收测试，再动手重构。
- **依赖图**：梳理模块间 import/require/use 关系，画出依赖拓扑。
- **热点文件**：通过 git log --stat 或 churn 分析，识别高频变更文件。
- **坏味分布**：按类别标注代码坏味（过长函数、深层嵌套、发散变化、霰弹修改、依恋情节等）。

### 2. 多视角交叉验证

避免凭单一视角认定"需要重构"。用至少两个维度交叉确认：

| 视角 | 关注点 | 工具 |
|------|--------|------|
| software-design | 深模块 vs 浅模块、信息隐藏、复杂度投资 | 设计评审 |
| pragmatic-programmer | DRY/YAGNI 取舍、tracer bullet、耦合判断 | 工程判断 |
| tech-debt | 债务类型（故意/偶然/位腐烂）、还债 ROI | 债务评估 |
| complexity-reducer | 圈复杂度、嵌套深度、认知负载 | 度量 |

只有至少两个视角都指向同一问题时，才认定为真问题。单视角直觉不构成重构理由。

### 3. 接缝识别

接缝（seam）是可以在不改变外部行为的前提下安全切开的边界：

- **接口边界**：已有接口定义、契约清晰的模块边界。
- **数据边界**：数据流经的窄点（函数参数、消息队列、数据库表）。
- **测试边界**：已有测试覆盖的模块，重构后测试仍是安全网。
- **部署边界**：独立部署单元（微服务、lambda、独立进程）。

标记每个接缝的：安全度（可切开程度）、测试保护度、影响半径。

### 4. 增量拆步

将重构拆成**可独立验证**的步骤，每步满足：

1. 有明确的输入和预期输出（不是"改好它"）。
2. 有测试或构建命令可验证（verify point）。
3. 回滚成本 ≤ 1 个 commit（单个 commit 可 revert）。
4. 不依赖后续步骤的中间状态（每步完成后代码处于可工作状态）。

拆步优先级：

| 优先 | 类型 | 示例 |
|------|------|------|
| 1 | 补测试 | 为无测试的关键路径补验收测试 |
| 2 | 重命名/移动 | 函数重命名、文件移动（不改变行为） |
| 3 | 提取/内联 | 提取方法、内联临时变量 |
| 4 | 替换算法 | 替换实现但保持接口不变 |
| 5 | 引入抽象 | 引入接口/基类，为后续扩展准备 |

## 输出模板

```markdown
# 重构计划：<scope>

## 基线
- 测试覆盖：X%（工具/命令）
- 依赖拓扑：[ASCII 图或模块列表]
- 热点文件：[churn top-5]
- 坏味分布：[按类别计数]

## 真问题清单
| # | 问题 | 验证视角 | 影响半径 | 接缝 |
|---|------|---------|---------|------|

## 拆步计划
| 步骤 | 动作 | 验证命令 | 回滚方式 |
|------|------|---------|---------|
| 1 | ... | `npm test` | git revert |
| 2 | ... | ... | ... |

## 不做的事
[列出不在本次重构范围内但被识别到的其他问题]

## 风险
[已知风险 + 缓解措施]
```

## 红线

- **不凭直觉重构**：没有多视角交叉验证的问题不进拆步计划。
- **不跳过测试基线**：无测试的模块先补测试再重构。
- **不做不可逆大改**：每步必须可独立 revert。
- **不改行为**：重构不改变外部可观测行为；行为变更走 feature 分支。

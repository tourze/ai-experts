# 三阶段权限检查管道

## 权限模式谱系

4 个权限模式，从保守到自主：

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| `default` | 写入/破坏性操作需确认 | 常规交互使用 |
| `plan` | 自由读取，所有写入需审批 | 审查密集型工作流 |
| `bypass` | 用户显式开启全自动审批 | 受信任的自动化、CI |
| `auto` | AI 分类器决策；需显式启用 | 监控下的自主运行 |

**关键设计**：模式是**用户控制的参数**，不是程序员硬编码的常量。Agent 代码不决定哪个模式适合当前任务——用户或运维方决定。

## 三阶段管道

```
Phase 1: validateInput()
   ↓ 输入格式无效 → 立即拒绝（不弹权限确认）
Phase 2: tool.checkPermissions(input, context)
   ↓ 工具自己的领域检查
   → allow:       直接执行
   → deny:        阻止并说明理由
   → ask:         弹出用户确认
   → passthrough: 继续到 Phase 3
Phase 3: canUseTool(tool, input, context)
   ↓ 外层权限引擎：规则 + hooks + 分类器
   → 最终 allow / deny / ask 决策
```

### 为什么需要三个阶段？

**Phase 1** —— 工具最清楚什么是合法输入。拒绝畸形输入可以避免误导性的权限提示（"是否允许此操作？"——但操作本身就会失败）。

**Phase 2** —— 工具最清楚自己领域内什么是危险的。Shell 工具理解危险命令模式，文件写入工具理解路径穿越。外层系统无法知道这些领域细节。

**Phase 3** —— 外层系统知道用户偏好、会话规则，并可调用昂贵的 AI 分类器。单个工具不应调用 AI 分类器——那是外层引擎的职责。

### passthrough 的意义

`passthrough` 意味着"我对此没有意见，交由外层系统决定"。工具不应越界做本属于外层权限引擎的决策。

## Bypass-Immune 操作

即使在 `bypass` 模式下，以下操作永远不会自动批准：

- 影响其他用户数据的操作（跨用户操作）
- 向敏感端点发送网络请求（由运维方配置）
- 工具作者显式标记为需要人工确认的操作

### 两级信任层级

```
运维方层级：设定绝对上限（哪些永远不能绕过）
    ↑ 不可逾越
用户层级：在上限范围内控制自己的信任级别
```

**原则**：bypass 是用户控制的，不是程序员控制的。用户可以在系统允许的范围内提升自己的权限，但无法超越运维方设定的天花板。

## 规则来源

规则有三个来源，按优先级排列：

| 来源 | 设置者 | 示例 |
|------|--------|------|
| `managed` | 企业策略（通过 MDM 推送） | 组织级 deny 规则 |
| `settings` | 用户配置文件 | `alwaysAllowRules: ["read /tmp/**"]` |
| `cli` | 命令行参数 | `--allow-write=/workspace` |

### 规则评估顺序

1. `alwaysDenyRules` — 匹配即阻止
2. `alwaysAskRules` — 匹配即确认（即使在 auto 模式）
3. `alwaysAllowRules` — 匹配即放行
4. 都不匹配 → 进入 AI 分类器

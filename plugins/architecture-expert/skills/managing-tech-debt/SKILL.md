# managing-tech-debt

## 适用场景
- 适合遗留系统治理、重构路线图、债务预算争取和版本节奏协调。
- 适合回答“现在该还哪些债、怎么分期还、如何不拖垮交付”。
- 交叉引用：债项盘点用 `tech-debt`；落到代码整改时配合 `code-refiner` 或 `refactoring-patterns`。

## 核心约束
- 债务必须和业务代价、交付速度、事故风险或团队效率挂钩。
- 不要把所有代码不喜欢的地方都归类成必须立刻偿还的债。
- 治理方案要可分期、可插入正常迭代，而不是只给“大重写”。
- 需要明确债务 owner、验证指标和退出条件。

## 代码模式
- 先列债项清单，再标影响面、风险、收益、机会窗口和建议时机。
- 必要时读取 [治理视角补充](references/guest-insights.md) 获取治理视角。
- 输出建议按“立即止血 / 近期治理 / 长期重构”三层切分。


## 检查清单
- 是否说明了债务来源、受影响模块和用户成本。
- 是否给出可执行的分期策略和 owner。
- 是否设定了衡量治理收益的指标。
- 是否识别了与业务路线冲突的部分。

## 反模式

### FAIL: 喊重写

```
"代码太烂了，全部重写"
→ 6 个月后：新代码也是烂的 + 旧代码还在跑 + 业务停滞
```

### PASS: 分期治理

```md
立即（本周）：止血
- 修复 OOM 高频崩溃（已影响线上）
近期（季度内）：
- Auth 模块从 PHP 4 升 8.x
- 拆出 PaymentService，独立测试覆盖到 80%
长期（半年）：
- 替换遗留 SOAP 接口为 REST（按业务迁移，不一次性切换）
→ 每项有 owner / 验收 / 退出条件
```

### FAIL: 情绪性优先级

```
"我看不顺眼这个目录结构 → P0 重构"
→ 用户："这个 P0 影响什么业务？"
→ 答不上
```

### PASS: 业务驱动

```
| 债 | 业务影响 | 优先级 |
| Auth 模块旧 | 任何 auth 改动 +1 周 / 安全事故概率高 | P0 |
| 目录结构乱 | 新人上手慢半天 | P3 |
→ 优先级跟业务代价挂钩
```

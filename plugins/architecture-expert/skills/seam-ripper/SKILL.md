# seam-ripper

## 适用场景
- 适合模块边界混乱、接口泄漏、循环依赖、抽象失效和职责飘移的问题。
- 适合回答“系统真正的边界在哪”“哪些模块该拆/该合”“为什么改一个点要连环改”。
- 交叉引用：系统级审计配合 `exhaustive-systems-analysis`；架构打分配合 `architecture-reviewer`。

## 核心约束
- 先画依赖图，再批判，不要看到一个 util 模块就立刻下结论。
- 必须把接口清晰度、依赖方向、耦合类型、抽象完整性分开分析。
- 允许提出不兼容的正确架构，但要同时给迁移步骤。
- 不要用“现有代码都这么写”作为正确性的证据。

## 代码模式
- 输出推荐分成：依赖图 → 关键接缝 → 失效模式 → 重组方案 → 迁移步骤。
- 优先抓 `God Module`、`Shotgun Surgery`、`Middle Man`、`Speculative Generality` 等模式。
- 重组建议要量化收益，例如减少耦合边数、压缩导出面或隔离领域边界。


## 检查清单
- 是否画清了模块边界、导出面和依赖方向。
- 是否指出哪些接口泄漏了实现细节。
- 是否说明了重组后的职责归属和迁移顺序。
- 是否区分了“必须拆”与“可以暂时容忍”的问题。

## 反模式

### FAIL: 凭目录名批判

```
"目录叫 utils → 应该删掉"
→ 实际：utils 含 8 个被全项目共用的纯函数 + 0 个泄漏
→ 属于合理共享代码，不是 God Module
```

### PASS: 依赖图驱动

```bash
# 量化依赖
madge --circular src/
depcruise src/ --output-type dot
→ 发现：
- circular: A ↔ B（必须拆）
- god module: utils.ts 被 80% 文件依赖（拆 4 个细粒度模块）
- speculative generality: BasePluginV2 只有 1 个实现（删抽象）
```

### FAIL: 发现耦合就上事件总线

```
"两个模块互相调 → 用事件总线解耦"
→ 引入 EventBus → 调用关系隐式 → 调试地狱
→ 实际：换接口签名能解决 50% 案例
```

### PASS: 按问题选工具

```
直接调用 → 接口隔离（80% 情况）
真正异步 + 多订阅者 → 事件总线（5%）
状态机驱动 → 显式状态转换（15%）
→ 不要把一种模式套所有问题
```

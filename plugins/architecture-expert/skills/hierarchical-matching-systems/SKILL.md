# hierarchical-matching-systems

## 适用场景
- 适合学生分班、岗位匹配、组织树映射、分层分类归并、记录链接等问题。
- 适合把业务规则翻译成可执行约束、偏好和目标函数。
- 交叉引用：更大范围的架构边界用 `system-design`；问题审计用 `exhaustive-systems-analysis`。

## 核心约束
- 先判断问题类型：稳定匹配、最优分配、层级对齐还是实体解析。
- 业务约束、硬约束、软约束和优先级必须分开建模。
- 任何“匹配质量差”结论都要落到具体输入分布、规则冲突或评分函数。
- 不要把求解器选择和业务正确性混为一谈。

## 代码模式
- 算法选择参考 `skills/hierarchical-matching-systems/references/decision-guide.md`。
- 若要比较 Gale-Shapley、Hungarian、层级约束或实体解析方案，按 `references/algorithms.md` 提供的维度对齐。
- 输出建议分成：模型定义 → 约束表 → 评分函数 → 失败样本 → 调整方案。


## 检查清单
- 是否把输入实体、层级关系和约束表述清楚。
- 是否区分了必须满足的约束和可优化目标。
- 是否用样例解释匹配失败或不稳定现象。
- 是否说明求解复杂度和可扩展性限制。

## 反模式
- 只谈算法名，不定义业务约束。
- 把评分权重拍脑袋设定，不做样本验证。
- 出现不稳定匹配后只调参数，不看模型假设是否错。
- 忽略层级冲突、容量限制和 tie-break 规则。

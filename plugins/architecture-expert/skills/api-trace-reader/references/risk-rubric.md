# 风险点分级标准

第 5 步「风险点」用的分级参考。每条风险必须配 file:line 证据,空口断言一律
不允许。建议先按下表对照命中类型,再判定等级。

## P0 — 高风险(必须立即关注)

会导致数据错误、服务不可用、资源耗尽,或一旦触发就不可逆。

| 类型 | 命中信号 |
|---|---|
| **全量扫描** | 查询无 `LIMIT`/`WHERE`/分页;`findAll` 在大表上调用;O(N) 全表 join |
| **N+1 查询** | foreach / map 内调用 `find` / `query`;ORM 关联未 eager load |
| **缓存无 TTL** | `redis SET` 不带 `EX`/`PX`;永久 key 没有显式失效路径 |
| **数据泄露** | 接口返回值包含敏感字段(密码 hash / token / 内部 id);日志打印明文 |
| **未授权访问** | 路由没接 `auth middleware`;`@IsGranted` 缺失;query 用 `id` 不校验 owner |
| **SQL 注入** | 字符串拼 SQL 不用 prepared;`raw query` 直接拼用户输入 |
| **死循环 / 死锁** | `while(true)` 无退出;事务嵌套;锁顺序不一致 |
| **不可逆破坏** | 物理删除无审计;`DROP`/`TRUNCATE`;覆盖写入无备份 |
| **全推 / 全量操作** | 推送/邮件/任务面向「所有用户」无白名单/灰度 |

## P1 — 中风险(限期修复)

会导致性能劣化、一致性风险、用户体验下降。

| 类型 | 命中信号 |
|---|---|
| **缓存穿透** | 查不到的 key 也回源 DB,无空值占位/布隆过滤 |
| **缓存雪崩** | 大量 key 同 TTL,失效时刻同时回源 |
| **缓存击穿** | 热点 key 失效瞬间多请求并发回源,无 mutex/互斥 |
| **缓存与 DB 不一致** | 写 DB 后不失效缓存;先删缓存后写 DB(双删未做) |
| **幂等缺失** | 消息消费 / webhook / 重试场景没有 dedupe key |
| **重试放大** | 重试无指数退避 / 无最大次数 / 无熔断 |
| **外部 API 无超时** | `curl_exec` 没设 timeout;`http.Client` 用默认值 |
| **外部 API 无降级** | 第三方挂了,主链路就挂(无 circuit breaker / 兜底响应) |
| **同步阻塞 IO** | 主链路里写文件 / 发邮件 / 调远程 API 没异步化 |
| **异常吞掉** | `try { ... } catch { }` 空 catch;catch 后 log 但不抛 |
| **慢查询** | 缺索引;`LIKE '%xxx%'` 前缀通配;`ORDER BY` 字段无索引 |
| **事务范围过大** | 事务内套外部调用 / 长 IO,占着行锁 |
| **跨服务事务** | 多个服务写库无 saga / TCC / outbox 模式 |

## P2 — 低风险(观察项)

隐患项,需要关注但不必立即修。

| 类型 | 命中信号 |
|---|---|
| **可观测性缺失** | 关键路径无 metric / tracing / structured log |
| **日志写主链路** | 同步写文件,磁盘满会阻塞 |
| **错误信息泄露细节** | 把 stacktrace 直接返回给前端 |
| **代码味道** | 复杂条件嵌套 > 3 层 / cyclomatic 高 / 长函数 |
| **测试缺失** | 关键路径无单测 / 集成测试 |
| **依赖陈旧** | 用了已废弃的 API / 安全有 CVE 的旧版本 |
| **配置硬编码** | URL / 密钥 / 阈值写死在代码 |
| **命名混乱** | 函数 / 变量名与实际行为不符 |

## 输出格式

风险点段必须按 P0 / P1 / P2 三级分组。每条格式:

```
[P0 高风险]
- <一句话风险描述>
  证据:<file:line>
  影响:<谁会受影响、什么场景下触发、影响范围有多大>
  建议:<可立即落地的最小修复建议,可选>
```

**没有风险也要明写**:

```
风险点:
未发现 P0/P1 级风险。

观察项(P2):
- <项 1>
- <项 2>

或:本接口全量风险扫描通过,无观察项。
```

禁止只写「无风险」就完事——要么列出 P2 观察项,要么明确写「全量扫描通过,无观察项」,
让用户知道你是真的扫了一遍,而不是偷懒。

## 灰色地带的判定原则

- **数据量决定等级**:同样的全表 `findAll`,`config` 表 10 行就是 P2,`orders` 表
  千万行就是 P0。判定时要考虑实际数据规模,如果不知道就标「需用户确认数据规模」。
- **触发频率决定等级**:每秒触发一次的 P1 比每天触发一次的 P0 更紧迫。
- **存在补偿机制 → 降级**:有重试 / 死信队列 / 人工补单流程的,可以从 P0 降到 P1。
- **不确定就升级**:拿不准是 P1 还是 P2 时,选 P1。宁可让用户多看一眼,不要漏掉。

# System Prompt 架构

## 静态 vs 动态分离

### 静态段（缓存边界之前，所有用户/会话一致）

| 段 | 内容 |
|----|------|
| 角色定义 | Agent 是谁、安全态势声明 |
| 系统规则 | 工具使用、格式、权限等操作规则 |
| 任务指南 | 编码标准、过度工程警告 |
| 行动安全 | 危险操作处理、可逆性 |
| 工具使用指南 | 工具选择偏好层级 |
| 语气风格 | 沟通规范 |
| 输出效率 | token 意识输出规则 |

### 动态段（缓存边界之后，每次可能不同）

| 段 | 为什么是动态 |
|----|------------|
| 会话引导 | 本次会话启用了哪些工具 |
| 用户记忆 | 从历史对话中提取的记忆 |
| 环境信息 | CWD、git 状态、OS、模型名、日期 |
| 语言偏好 | 用户语言设置 |
| MCP 指令 | 已连接 MCP 服务器的指令（动态连接/断开） |

## 段函数 Memoization

两种段函数类型：

```python
# 计算一次，memoize 直到 /clear 或 /compact
@system_prompt_section("rules")
def rules_section():
    return load_rules()

# 每轮重新计算（必须注明原因）
@DANGEROUS_uncached_section("mcp", reason="MCP servers connect/disconnect dynamically")
def mcp_section():
    return gather_mcp_instructions()
```

`DANGEROUS_` 前缀是刻意的命名选择：让性能代价在调用点可见。审查者可以 grep 它来审计所有热路径计算。

## 受众变体

同一个 Agent 可能需要为不同用户类型提供不同 prompt 内容：

- **外部用户**：精简、保守的指南
- **内部用户**：更详细的调试指令、更严格的代码审查标准

通过环境变量或配置标志控制额外指令块的注入——注入到相关段函数内部，而非简单追加到末尾。

## 设计原则

1. **问自己：这个段在两次请求之间会变吗？** 不会 → 静态段；会 → 动态段
2. **每个段一个职责** → 可独立测试、条件包含、独立缓存、按 Agent 类型替换
3. **模型侧指令要说 WHAT 和 WHY** → 模型理解原因后在边缘情况下更可靠
4. **昂贵计算必须 memoize** → 每轮从磁盘加载记忆文件是不必要的 I/O

# feature-dev

## 适用场景
- 适合复杂功能、陌生代码库、新模块建设和需要沉淀文档的功能开发。
- 适合把一次实现拆成可解释的阶段，而不是直接跳进编码。
- 交叉引用：设计前风险审计用 `plan-review`；拆任务时配合 `task-decomposer`；跨 session 的复杂任务配合 `persistent-planning` 落盘三文件。

## 核心约束
- 不要跳阶段，尤其不能跳过代码库探索、澄清问题和方案对比。
- 实现阶段必须带验证策略，必要时先补测试再写代码。
- 如果只是单行修复或紧急热修，不要强行套满七阶段。
- 总结阶段要记录关键决策、限制和后续动作。

## 代码模式
- 七阶段顺序固定：发现 → 代码库探索 → 澄清问题 → 架构设计 → 实现 → 质量复核 → 总结。
- 架构设计阶段至少比较两种方案，说明复杂度、风险和集成成本。
- 输出应兼顾代码、测试、文档和交接说明，而不是只交一段实现。


## 检查清单
- 是否明确定义了功能目标、成功标准和范围边界。
- 是否识别了相关文件、既有模式和相似实现。
- 是否补齐了关键澄清问题和方案对比。
- 是否在收尾阶段记录已知限制和后续改进项。

## 反模式

### FAIL: 陌生代码库直接开写

```
"加个用户导出功能" → 立刻打开编辑器写 /api/export
→ 后发现已有 ExportService + RateLimiter，新代码与既有模式冲突，PR 被打回
```

### PASS: 先探索再实现

```
1. rg -n "export|Export" → 发现 app/Services/ExportService.php
2. 读 ExportService：已支持 CSV/XLSX，有 RateLimiter
3. 澄清：新增 PDF 格式 还是 新资源类型？
4. 方案：复用 ExportService，新增 PdfFormatter 实现
```

### FAIL: 只给一个方案

```
方案：用 Redis 做缓存
→ 为什么不是内存 LRU？不是 CDN？没讨论
```

### PASS: 至少两个方案 + 权衡

```
| 方案 | 复杂度 | 性能 | 成本 |
|------|--------|------|------|
| 内存 LRU | 低 | 最快 | 无法多实例共享 |
| Redis | 中 | 1ms | 需运维 |
| CDN | 高 | 边缘最快 | 不适合动态数据 |
→ 选 Redis：多实例共享是硬需求
```

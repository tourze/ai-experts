# 代码审查维度

## 1. 命名与语义

- 变量名、函数名、类名是否准确反映其用途。
- 布尔变量是否用 `is`/`has`/`can`/`should` 等前缀。
- 是否存在误导性命名（名字说的和做的不一致）。
- 缩写是否在上下文中无歧义。

## 2. 函数设计

- 函数是否只做一件事（单一职责）。
- 函数长度是否合理（超过 30-40 行需关注）。
- 参数数量是否过多（超过 3-4 个需关注）。
- 是否有副作用未在函数名或文档中体现。
- 返回值是否一致（不要有时返回值有时返回 null 又有时抛异常）。

## 3. 错误处理

- 是否有裸的 catch-all（吞掉所有异常不做任何事）。
- 错误是否被正确传播或转换（不丢失上下文）。
- 是否区分了可恢复错误和不可恢复错误。
- 边界输入（空值、空集合、零、负数、超大值）是否有处理。

## 4. 逻辑与边界

- 循环是否有正确的终止条件。
- 条件分支是否覆盖所有情况（特别是 else/default）。
- 是否存在 off-by-one 错误。
- 并发访问是否安全（如果适用）。
- 时间和顺序依赖是否显式处理。

## 5. DRY 与抽象

- 是否有重复代码块可以抽取。
- 抽象层次是否一致（同一函数内不应混合高层逻辑和底层细节）。
- 是否为了 DRY 而创建了不自然的抽象（过度 DRY 也是问题）。

## 6. 可读性

- 控制流是否可以用 early return 简化。
- 嵌套层级是否过深（超过 3 层需关注）。
- 注释是否解释了"为什么"而不是"做了什么"。
- 代码组织是否符合阅读习惯（公开接口在前，内部实现在后）。

## 输出格式示例

```markdown
## 审查结论

共发现 N 个问题：关键 X 个，重要 Y 个，建议 Z 个。

### 关键

1. `path/to/file:42` — 问题描述
   - 影响：具体风险
   - 建议：改进方向

### 重要

1. `path/to/file:88` — 问题描述
   - 原因：为什么这是问题
   - 建议：改进方向

### 建议

1. `path/to/file:120` — 可选改进
   - 建议：改进方向
```

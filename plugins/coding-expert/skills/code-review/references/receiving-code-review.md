# 接收代码审查

## 适用场景

- 收到了来自人类审查者、CI 机器人或子代理的代码审查反馈。
- 需要决定是实施建议、推回、还是请求澄清。
- 交叉引用：发起审查用 `code-review`；落地审查用 `pre-landing-review`。

## 核心约束

**违反字面规则 = 违反规则精神。不存在"灵活变通"。**

- 收到反馈先评估，不先实施。
- 不做表演式同意（"说得太对了！""好建议！"）。
- 推回时用技术论据，不用情绪。
- 不确定的项全部澄清完毕后才开始实施。

## Iron Law

```
验证后再实施。技术正确性高于社交舒适度。
```

## 响应模式

```
收到审查反馈时：

1. 阅读：完整读完所有反馈，不要边读边反应
2. 理解：用自己的话复述要求（或提问）
3. 验证：对照代码库实际情况核实
4. 评估：在当前代码库中技术上合理吗？
5. 响应：技术确认或有理有据地推回
6. 实施：逐条实施，每条单独测试
```

## 禁止的回应

| ❌ 禁止 | ✅ 应该 |
|---------|---------|
| "说得太对了！" | 复述技术要求 |
| "好建议！" | 直接开始修 |
| "感谢指出！" | 说明修了什么 |
| "让我立即实施" | 先验证再实施 |

**正确的回应方式：**
- `"已修复。[简述改了什么]"`
- `"确认了，[具体问题]。已在 [位置] 修复。"`
- 直接修，用代码证明你听到了反馈。

## 处理不明确的反馈

```
如果任何条目不明确：
  停下——不要实施任何东西
  对不明确的条目逐个提问

原因：条目可能互相关联。部分理解 = 错误实施。
```

**示例：**
```
用户："修复 1-6 项"
你理解 1,2,3,6。不确定 4,5。

❌ 错误：先实施 1,2,3,6，4,5 稍后再问
✅ 正确："1,2,3,6 我理解了。4 和 5 需要澄清后再动手。"
```

## 何时推回

推回条件：
- 建议会破坏现有功能
- 审查者缺少完整上下文
- 违反 YAGNI（未使用的功能）
- 在当前技术栈中技术上不正确
- 与已有架构决策冲突

**如何推回：**
- 用技术论据，不用防御姿态
- 引用工作的测试/代码作为证据
- 提出具体问题
- 如涉及架构分歧，升级给用户

## YAGNI 检查

```
如果审查者建议"正确实现"某个功能：
  grep 代码库检查实际使用情况

  如果未使用："这个端点没有被调用。移除它（YAGNI）？"
  如果在使用：那就正确实现
```

## 实施顺序

```
多条反馈时按此顺序实施：
  1. 先澄清所有不明确的条目
  2. 阻断项（崩溃、安全）
  3. 简单修复（拼写、import）
  4. 复杂修复（重构、逻辑）
  5. 每条修复单独测试
  6. 验证无回归
```

## Red Flags — 出现以下念头时立即停下

| 念头 | 现实 |
|------|------|
| "审查者说的一定对" | 审查者可能缺少上下文。先验证。 |
| "全部实施最省事" | 盲目实施 = 可能引入问题。逐条评估。 |
| "先做能理解的，不确定的后面再说" | 条目可能关联。全部澄清后再动手。 |
| "推回太不礼貌了" | 技术正确性 > 社交舒适度。用数据推回。 |
| "不想显得不配合" | 配合 = 做正确的事，不是无条件同意。 |

## Rationalization 对照表

| 借口 | 现实 |
|------|------|
| "审查者经验更丰富" | 经验丰富不等于了解当前代码库。验证。 |
| "先实施再说，不行再改回来" | 先验证再实施，比实施后回滚成本低。 |
| "只是小改动，不用那么认真" | 小改动也能引入 bug。逐条测试。 |
| "感谢对方的时间" | 用代码回应，不用感谢词。 |

## 推回后发现自己错了

```
✅ 正确："你说的对——我验证了 [X]，确实是 [Y]。正在修复。"
❌ 错误：长篇道歉 / 解释为什么会推回 / 过度自我批评
```

陈述更正事实，继续工作。

## 检查清单

- [ ] 完整读完所有反馈后再反应
- [ ] 不明确的条目已全部澄清
- [ ] 每条建议都对照代码库验证过
- [ ] 技术上不合理的已推回（附论据）
- [ ] 实施按 阻断 → 简单 → 复杂 顺序
- [ ] 每条修复单独测试
- [ ] 无回归
- [ ] 没有表演式同意

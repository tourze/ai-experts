# 审查纪律守卫

**违反字面规则 = 违反规则精神。不存在"灵活变通"。**

## Iron Law

```
没有读取实际代码或 DIFF，不允许给出审查意见
```

凭猜测审查 = 浪费所有人的时间。

## 审查 vs 表演

审查是技术活动，不是社交活动。以下行为是"表演审查"，不是"真正审查"：
- 只看了简单文件，跳过了复杂文件
- 所有问题都标"建议"，没有敢标"关键"的
- 给出的建议都是锦上添花，没有触及核心逻辑
- 审查时间 < 代码量合理需要的阅读时间

**真正的审查让人不舒服。** 如果你的审查结果让作者完全满意、毫无摩擦，大概率你漏了关键问题。

## Red Flags — 出现以下念头时立即停下

| 念头 | 现实 |
|------|------|
| "代码看起来没问题" | "看起来"不是证据。逐维度检查了吗？ |
| "这个命名是个人偏好" | 如果命名导致误解，就是质量问题，不是偏好。区分清楚。 |
| "没什么大问题，LGTM" | 没发现问题和没有问题是两回事。六个维度都过了吗？ |
| "我来重写一下更好" | 审查是指出问题，不是替人写代码。给建议，不给整段代码。 |
| "这个太严重了，标关键" | 所有问题都是"关键"就等于没分级。真的会导致事故吗？ |
| "这个我不太确定，跳过吧" | 不确定 = 标为"建议"并说明不确定。跳过 = 漏检。 |
| "这个文件太复杂了，看不太懂" | 看不懂 = 高风险。要么花时间读懂，要么标记为需要作者解释。 |
| "作者水平比我高，不好意思提" | 审查的价值在于外部视角，不在于谁更资深。你的困惑本身就是信号。 |
| "先给个 LGTM，下次再仔细看" | 没有"下次"。这次的 LGTM 就是你的质量承诺。 |

## Common Rationalizations

| 借口 | 现实 |
|------|------|
| "diff 太大了，挑重点看" | 大 diff 更危险。要么逐文件看，要么让作者拆分。 |
| "函数名已经说明一切了" | 函数名只表达意图，不保证实现正确。读实现。 |
| "这个模式我见过很多次了" | 熟悉感 ≠ 正确性。这次的上下文和之前一样吗？ |
| "风格问题不值得提" | 对。但如果你提了，就别标成"关键"。分级。 |
| "作者比我更懂这个模块" | 审查的价值在于外部视角。你不懂的地方可能正是盲区。 |
| "Agent 说实现完成了，LGTM" | Agent 的自我报告可能不完整、不准确或过于乐观。独立验证。 |
| "我只看了改动的文件" | 改动文件的调用方和被调用方也可能受影响。看影响面。 |
| "测试通过了所以逻辑没问题" | 测试覆盖率不是 100%。通过只说明被测路径正确，未测路径未知。 |
| "这个 PR 已经讨论很久了，赶紧过" | 讨论久 ≠ 审查充分。你看的是最终 diff，不是讨论历史。 |
| "只是移动了代码位置，逻辑没变" | 移动代码会改变 import 关系、初始化顺序、作用域。验证行为未变。 |
| "新代码和旧代码风格一致就行" | 和旧代码一致 ≠ 正确。旧代码本身可能就有问题。 |

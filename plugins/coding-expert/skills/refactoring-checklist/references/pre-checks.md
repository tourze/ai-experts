# 重构准入四项详细检查

## 1. 测试基线

- [ ] 被重构代码有测试覆盖吗？覆盖率是多少？
- [ ] 测试是否可信（不是只测 mock 的伪测试）？
- [ ] 如果没有测试，先补关键路径的表征测试（characterization test）再继续。
- [ ] 测试能在本地快速运行吗（< 30 秒）？

```bash
# 确认测试现状
# 跑一遍相关测试，记录通过数
# 如果有覆盖率工具，记录基线覆盖率
```

## 2. 范围界定

- [ ] 明确写出"这次重构要动哪些文件/模块"。
- [ ] 明确写出"这次重构不动什么"（排除项）。
- [ ] 预估影响面：哪些调用方会受到影响？
- [ ] 如果范围超过 5 个文件，考虑分多次重构。

```
重构范围：
- 目标：[要改的模块/文件]
- 排除：[明确不动的部分]
- 影响面：[受影响的调用方]
- 预估规模：[文件数 / 行数]
```

## 3. 目标明确

- [ ] 用一句话说清楚"重构完之后会变成什么样"。
- [ ] 明确重构的收益（降低复杂度？消除重复？改善命名？拆分职责？）。
- [ ] 明确不做什么（不加功能、不改接口、不优化性能——除非那是重构目标）。

## 4. 回滚方案

- [ ] 重构在一个干净的分支上进行。
- [ ] 每个增量步骤都有独立提交，方便 revert。
- [ ] 如果重构到一半发现不可行，知道怎么回到起点。

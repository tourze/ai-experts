# Task Closure Protocol（AAR 详解）

被 `verification-checklist.md` 的检查清单末项引用。在做任务收尾时读这份。

## 为什么验证不等于完成

验证门控证明"做对了"，AAR 扫描证明"学到了"。跳过 AAR 的常见后果：同一类坑半年内被踩 2-3 次、记忆只增不减、新规则进不来旧规则不退场。

## 4 问完整版

非 trivial 任务收尾前 30 秒扫一遍，全部"否"才能宣布完成。

| # | 问题 | 命中标志 |
|---|------|---------|
| 1 | 新模式? | 用了未记录在 skill / memory 的模式、约定、调用顺序 |
| 2 | 新陷阱? | 走了弯路、踩了不提前知道就会浪费时间的坑 |
| 3 | 缺失规则? | 因为缺某条规则导致绕路、回退或返工 |
| 4 | 过时规则? | 现有 skill / memory / CLAUDE.md 的某条规则已不准确 |

任意一条"是" → 进入录入门槛判断；全部"否" → 到此结束。

## 录入门槛 2/3

不是所有发现都值得记录。录入前过 3 条判据，至少 **2/3** 通过才写：

| 条件 | 判据 |
|------|------|
| 可重复 | 未来任务还会遇到吗? |
| 代价高 | 不知道的话浪费 30 分钟以上? |
| 代码不可见 | 光看代码能推断出来吗? |

**典型通过**：框架生命周期坑、隐藏的注册顺序依赖、跨层交互陷阱、非显而易见的状态重置要求。

**典型不通过**：一次性变通、纯命名风格偏好、轻量 typo、官方文档已充分覆盖的内容。

## 可跳过 AAR 的窄场景

- 仅格式化 / 仅注释 / 仅依赖版本号变更
- 不引入新教训的纯重命名重构
- 单字符 typo 修复

跳过条件之外的"任务很小""赶时间""我已经知道了"都不算理由——小任务正是教训藏身的地方，赶时间正是最容易踩坑的时刻。

## 录入位置

| 类型 | 落点 |
|------|------|
| 稳定约束 / 必须遵守 | 对应 skill 正文 或 memory `feedback_*.md` |
| 踩坑 / 边缘情况 | skill 内 `references/gotchas.md`（文件夹结构）或 memory `project_*.md` |
| 流程步骤 | skill 内 `workflows/*.md` |
| 触发或路由变更 | skill 的 description 或全局 CLAUDE.md |

## 泛化规则（录入前必做）

记录的内容必须**脱离当前项目上下文也能看懂**。改写公式：

```
具体发现 → 抽象为通用 pattern → 说明不遵守的后果
```

| 坏（项目叙事） | 好（泛化知识） |
|---|---|
| 在产品迭代模块中分页要在切 tab 时重置 | 切换上下文（tab/视图/筛选器）时必须重置分页到第 1 页 |
| 我们的 UserService.createUser 要先查重 | 创建实体前必须做唯一性校验 — 数据库约束是最后防线 |
| OrderController 里日期参数用 yyyy-MM-dd | API 日期参数统一使用 ISO 8601，在 Controller 入口层校验 |

## 激活优于存储

仅写入 `references/` 不算"已捕获"。高代价坑必须同时：

- **存储**在正确的文件中
- **激活**在会触发它的任务路径上（workflow 检查项 / SKILL.md 路由 / rules 摘要）

判断方法：下次 agent 走正常任务路径时，会自然读到这条经验吗？不会，就只是"记下来了"，没有"生效"。

## Rationalizations to Reject

| 借口 | 反驳 |
|------|------|
| "这次任务很小，AAR 没必要" | 小任务正是教训藏身的地方。30 秒扫完，跳过比做还慢 |
| "等会话结束再一起补" | 会忘。扫描必须在任务收尾做，不能批处理 |
| "用户在赶时间" | 赶时间是最容易踩坑的时刻。压力是跑 AAR 的理由 |
| "这条经验我已经知道了" | 记录是给未来的 agent 看的，不是给现在的你 |
| "这个已经在现有规则里了" | 那 10 秒内就能扫完，跑了比争论快 |

## 与验证检查清单的关系

- 「验证门控流程」管"客观证据"——任务做对了没有
- 「Rationalization 对照表」管"跳过验证的借口"
- 「Task Closure 4 问」管"主观沉淀"——任务学到了没有
- 三者都是完成门禁的一部分，缺一不可

# 调试纪律守卫

**违反字面规则 = 违反规则精神。不存在"灵活变通"。**

## Iron Law

```
没有根因调查，不允许提出修复方案
```

修了不知道根因的 bug = 埋了下一个 bug。

## 三次失败规则

连续尝试 3 次修复仍然失败时，必须停下来质疑架构：
- 每次修复都揭示新问题（在不同位置）= 架构耦合问题
- 修复需要"大规模重构"才能实施 = 设计不适合当前需求
- 每次修复在别处制造新症状 = 你在修症状，不是修根因

**此时必须：**
1. 停止尝试第 4 次修复
2. 质疑：这个模式/架构本身是否合理？
3. 与用户讨论：是继续修补还是重构

## 多组件系统的证据收集

当系统有多个组件（CI → 构建 → 签名、API → 服务 → 数据库）时：

**在提出修复方案之前，先加诊断埋点：**
```
对每个组件边界：
  - 记录什么数据进入该组件
  - 记录什么数据离开该组件
  - 验证环境/配置的传播
  - 检查每层的状态

运行一次收集证据 → 分析证据定位故障组件 → 再调查该组件
```

## Red Flags — 出现以下念头时立即停下

| 念头 | 现实 |
|------|------|
| "我知道问题在哪，先改改试试" | 你看到的是症状，不是根因。走流程。 |
| "这个很简单，不用系统化" | 简单的 bug 也有根因。流程对简单 bug 更快。 |
| "紧急，没时间走流程" | 系统化调试比乱猜快。瞎试 3 次浪费的时间 > 走流程。 |
| "先加个 try-catch 顶一下" | 吞掉异常 = 藏住 bug，不是修好 bug。 |
| "我已经试了两次了，再试一次" | 连续失败 3 次 = 架构问题，不是运气问题。停下来重新分析。 |
| "加个 sleep 就好了" | 竞态不会因为 sleep 消失，只是变得更难复现。 |
| "日志里没有有用信息" | 没有有用日志 = 该先加日志，不是该猜。 |
| "我大概知道是哪个组件的问题" | "大概"不是证据。在每个组件边界加埋点，用数据定位。 |
| "错误信息已经告诉我根因了" | 错误信息告诉你症状出现在哪，根因可能在上游 5 层。 |
| "先列出所有问题，然后逐个修复" | 先列出问题 = 先猜测。调查一个，验证一个，再下一个。 |

## Common Rationalizations

| 借口 | 现实 |
|------|------|
| "这个问题不可复现" | 95% 的"不可复现"是复现条件没找对。加日志再观察。 |
| "我改了之后测试通过了" | 通过了不代表你修对了。你知道为什么通过了吗？ |
| "改多处一起测更高效" | 同时改多处 = 不知道哪个改动起作用。回滚，每次只改一处。 |
| "这个不需要补测试，太简单了" | 越简单的修复越容易漏掉回归测试。补。 |
| "时间到了先交差" | 交一个不确定是否正确的修复 = 交一个定时炸弹。 |
| "加个 null check 就安全了" | null check 只掩盖了下游症状。为什么上游会传 null？修源头。 |
| "上游修了代价太大，在下游兜底" | 先证明"上游修了还不够"，再加下游防御。否则你在修错层。 |
| "重启之后好了" | 重启清掉了状态，但 bug 还在。下次什么条件会再触发？ |
| "先修好再去理解为什么" | 不理解根因的修复 = 碰运气。你不知道它为什么"好了"，也就不知道它什么时候会"又坏了"。 |
| "这个模式别人也这么用" | 别人的上下文和你不一样。先验证在你的代码库里是否成立。 |
| "加个重试应该能解决" | 重试掩盖了失败原因。先搞清楚为什么会失败，再决定是否需要重试。 |

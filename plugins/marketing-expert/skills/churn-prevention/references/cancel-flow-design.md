# 取消流设计详解

## 五步结构

```
Trigger → Survey → Dynamic Offer → Confirmation → Post-Cancel
```

1. **Trigger**：用户点击"取消订阅"，进入取消流而非直接取消。
2. **Survey**：收集退出原因（单选 + 可选补充），数据用于匹配 offer 和产品改进。
3. **Dynamic Offer**：根据退出原因动态展示挽留方案。
4. **Confirmation**：用户接受 offer 则留存；拒绝则确认取消，展示到期日和数据保留政策。
5. **Post-Cancel**：发送确认邮件，保留账户数据 30-90 天，进入 win-back 序列。

设计要点：
- 整个流程不超过 3 步点击（Survey + Offer + Confirm）。
- 移动端和桌面端都要测试流程完整性。
- 对已接受过 offer 的用户，避免重复打折（设置 cooldown 90 天）。

## 退出调查选项（5-8 项）

| 编号 | 原因 | 对应挽留策略 |
|------|------|-------------|
| 1 | 太贵了 | 折扣 / 降级 |
| 2 | 用得不够多 | 暂停订阅 |
| 3 | 缺少需要的功能 | 路线图展示 |
| 4 | 换用竞品 | 对比 + 折扣 |
| 5 | 技术问题 | 专属支持升级 |
| 6 | 暂时不需要 | 暂停订阅 |
| 7 | 公司/项目关闭 | 降级到免费 |
| 8 | 其他（必填文本） | 人工跟进 |

调查规则：
- 单选 + 可选自由文本，不要多选（影响 offer 匹配准确度）。
- 选项按历史占比降序排列。
- "其他"必须带必填文本框。
- 调查数据每月汇总，反馈给产品和定价团队。

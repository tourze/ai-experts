# 想法验证

## 适用场景
- 创业点子初筛、需求洞察验证、方案价值判断、立项前风险筛查。
- 需要完整框架时可阅读 [references/full-guide.md](references/full-guide.md)。
- 做用户画像或市场规模补充时，可配合 [startup-icp-definer](../startup-icp-definer/SKILL.md) 与 [market-sizing-analysis](../market-sizing-analysis/SKILL.md)。

## 核心约束
- 先验证问题是否真实、频繁且高成本，再谈方案优雅与否。
- 明确区分“用户会说想要”和“用户愿意付费/切换”的差别。
- 验证过程必须包含反证思路，避免只收集支持自己观点的证据。

## 代码模式
```markdown
| 维度 | 证据 | 当前判断 | 最大风险 |
| --- | --- | --- | --- |
| 痛点强度 | ... | ... | ... |
| 付费意愿 | ... | ... | ... |
```

## 检查清单
- [ ] 问题、用户、替代方案和价值假设已经写清。
- [ ] 已收集用户证据、行为证据和市场证据。
- [ ] 已判断继续验证、缩小范围或停止投入的条件。
- [ ] 下一步验证动作具体可执行。

## 反模式

### FAIL: 朋友礼貌反馈

```
"我做了个 X，你觉得怎么样？"
朋友："听起来挺有意思的！"
→ 你：6 个用户都说好 → 立项 → 上线 → 0 付费
→ 朋友是为了不伤你的感情，不是真要买
```

### PASS: 用 Mom Test

```
不问："你会用吗？"
问："你最近一次遇到 X 问题是什么时候？"
"你为此做了什么？"
"花了多少钱？"
→ 看过去行为，不看未来意愿
```

### FAIL: 无失败标准

```
"我先做 MVP 试试"
→ 上线 6 个月仍在迭代
→ "下一版本就好了"
→ 资金耗尽
```

### PASS: 预设 kill criteria

```
立项前明确：
- 60 天内付费用户 < 10 → 重新定位
- 90 天内月收入 < $1k → 关停
- D30 留存 < 20% → 不能继续投流
→ 触发任一 → 必须停下来重审
```

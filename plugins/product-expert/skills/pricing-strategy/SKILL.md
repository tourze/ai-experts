# 定价策略

## 适用场景
- SaaS 套餐设计、涨价、免费试用、价值度量或打包层级重构。
- 需要参考 [references/research-methods.md](references/research-methods.md) 与 [references/tier-structure.md](references/tier-structure.md)。
- 讨论商业模式或市场空间时，可配合 [business-model](../business-model/SKILL.md) 和 [market-sizing-analysis](../market-sizing-analysis/SKILL.md)。

## 核心约束
- 先理解客户获得的价值、替代方案和购买流程，再决定价格。
- 包装结构、价值度量和升级路径要一致，避免定价与产品体验互相打架。
- 涨价、免费和折扣策略都要说明目标，不要同时追求“高 ARPU”和“零摩擦”。

## 代码模式
```markdown
| 套餐 | 目标客群 | 核心价值 | 限制项 | 价格 |
| --- | --- | --- | --- | --- |
```

## 检查清单
- [ ] 已明确目标客群、价值度量和竞争替代品。
- [ ] 套餐分层和升级理由对用户可解释。
- [ ] 已考虑免费、试用、折扣和涨价的副作用。
- [ ] 定价结论可落到销售、计费和产品限制上。

## 反模式

### FAIL: 跟随竞品定价

```
"竞品 A 收 99，我们 79"
→ 客户："你们是不是不如他们？"
→ 主动放弃 25% 毛利
```

### PASS: 价值 + 差异化

```
1. 量化价值：客户用我们 → 每月省 X 小时 / 多收 $Y
2. 替代成本：转去竞品 → 数据迁移 + 培训
3. 定价 = 价值的 10-25%
4. 价格点：$129（高于 A，附加专属集成）
→ 客户愿意付的不是"便宜"，是"清晰更值"
```

### FAIL: 免费版自杀

```
免费版：100% 功能 + 100 万 API 调用/月
付费理由：客服快一点
→ 99% 用户不付费 / 1% 付费的也是大客户被动接受
```

### PASS: 免费有边界

```
免费：核心场景 + 100 调用/天 + 1 个用户
付费：> 1 用户 / > 100 调用 / 高级集成
→ 免费足够验证价值，但任何严肃使用必付费
→ Conversion rate 8-15%（健康水平）
```

### FAIL: 套餐硬卡位

```
专业版 vs 企业版
区别：企业版能"导出 PDF" + "更深紫色背景"
→ 用户感觉被坑
→ 大客户照样不升级
```

### PASS: 价值度量分层

```
按真正驱动价值的维度分层：
- 用户数（1 / 5 / unlimited）
- 月调用量（10k / 100k / unlimited）
- 数据保留（30天 / 1年 / 永久）
→ 升级理由清晰，量大客户自然升级
```

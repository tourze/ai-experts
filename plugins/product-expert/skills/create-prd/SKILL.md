# 编写 PRD

## 适用场景
- 新功能立项、需求收敛、跨团队对齐、进入开发前的规格固化。
- 需要把 discovery 结论转成可执行需求时，可配合 [agile-product-owner](../agile-product-owner/SKILL.md)。
- 如果还在验证阶段，先用 [prfaq](../prfaq/SKILL.md) 定"为什么做"，再用 PRD 定"怎么做"。
- PRD 拆解执行：Epic 分解策略见 [references/epic-decomposition.md](references/epic-decomposition.md)，用户故事编写见 [references/user-story-patterns.md](references/user-story-patterns.md)，MVP 范围取舍见 [references/scoping-cutting.md](references/scoping-cutting.md)。

## 核心约束
- 先写问题、目标和用户，再写方案；没有成功指标的 PRD 不完整。
- 文档必须显式列出边界、非目标、风险与依赖，避免把所有诉求都塞进首版。
- PRD 是约束开发的事实来源，不是营销稿；措辞要可验证、可拆任务、可测试。
- 10 段结构可根据项目规模裁剪，但 1-5 段为最小完整集，不可省略。

## 代码模式

### 10 段式 PRD 结构

```markdown
1. 背景与问题        ← 为什么要做（现状痛点、数据证据）
2. 目标与成功指标    ← 做到什么程度算成功（可量化的 KR）
3. 用户与场景        ← 给谁用、什么场景下用（Persona + Job Story）
4. 方案概述          ← 怎么解决（核心流程、关键交互，不写实现细节）
5. 详细需求          ← 功能点列表 + 每条的验收标准（Given/When/Then）
6. 非目标与排除项    ← 明确不做什么，防止 scope creep
7. 风险与依赖        ← 技术风险、外部依赖、合规约束
8. 发布与上线计划    ← 灰度策略、AB 测试、回滚预案
9. 数据与监控        ← 埋点需求、核心看板、告警阈值
10. 附录             ← 竞品参考、用户调研摘要、技术方案链接
```

最小完整集（轻量 PRD）：1-5 段。中型项目加 6-8 段。完整 PRD 用全 10 段。

## 检查清单
- [ ] 问题、目标、用户、成功指标已闭环。
- [ ] 关键流程、状态变化、异常路径和边界条件已写清。
- [ ] 已明确非目标、依赖、上线方式与回滚预案。
- [ ] 后续可直接拆成版本规划和开发任务。

## 反模式

### FAIL: 只有功能列表

```markdown
# 用户中心 PRD
## 功能
- 登录
- 注册
- 修改密码
- 绑定手机
- 实名认证
```

→ 没有问题陈述、没有目标、没有成功指标，读者不知道为什么要做、做到什么程度算完。

### PASS: 问题驱动 + 可验证指标

```markdown
# 用户中心 PRD
## 问题
现状：第三方登录注册转化率 42%，行业基准 65%。
痛点：用户反馈”手机号注册流程太长”。

## 目标
H1 把注册转化率从 42% 提升到 55%。

## 成功指标
- 注册完成率 ≥ 55%（P0）
- 注册平均耗时 ≤ 90s（P1）

## 范围（v1）
- 手机号一键注册（新增）
- 第三方登录（保留）

## 非目标
- 邮箱注册（v1 不做，等数据验证后再评估）
```

### FAIL: “支持所有情况”

```markdown
## 登录流程
系统应支持所有主流登录方式，适配所有浏览器和设备。
```

### PASS: 显式边界

```markdown
## 登录流程（v1）
- 支持：iOS 15+、Android 10+、Chrome/Safari/Edge 最新两个大版本
- 不支持：IE11、鸿蒙非 Android 套件
- 降级：不支持的环境跳转至告知页并引导至 App
```

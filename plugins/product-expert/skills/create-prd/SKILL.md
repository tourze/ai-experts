# 编写 PRD

## 适用场景
- 新功能立项、需求收敛、跨团队对齐、进入开发前的规格固化。
- 需要把 discovery 结论转成可执行需求时，可配合 [version-planner](../version-planner/SKILL.md) 和 [agile-product-owner](../agile-product-owner/SKILL.md)。

## 核心约束
- 先写问题、目标和用户，再写方案；没有成功指标的 PRD 不完整。
- 文档必须显式列出边界、非目标、风险与依赖，避免把所有诉求都塞进首版。
- PRD 是约束开发的事实来源，不是营销稿；措辞要可验证、可拆任务、可测试。

## 代码模式
```markdown
1. 背景与问题
2. 目标与成功指标
3. 用户与场景
4. 方案与流程
5. 范围、非目标、依赖、发布计划
```

## 检查清单
- [ ] 问题、目标、用户、成功指标已闭环。
- [ ] 关键流程、状态变化、异常路径和边界条件已写清。
- [ ] 已明确非目标、依赖、上线方式与回滚预案。
- [ ] 后续可直接拆成版本规划和开发任务。

## 反模式
- 只有功能列表，没有业务目标。
- 用“支持所有情况”替代边界定义。
- PRD 与版本规划、验收标准互相矛盾。

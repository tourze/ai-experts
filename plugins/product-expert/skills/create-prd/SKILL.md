# 编写 PRD

## 适用场景
- 新功能立项、需求收敛、跨团队对齐、进入开发前的规格固化。
- 需要把 discovery 结论转成可执行需求时，可配合 [version-planner](../version-planner/SKILL.md) 和 [agile-product-owner](../agile-product-owner/SKILL.md)。

## 核心约束
- 先写问题、目标和用户，再写方案；没有成功指标的 PRD 不完整。
- 文档必须显式列出边界、非目标、风险与依赖，避免把所有诉求都塞进首版。
- PRD 是约束开发的事实来源，不是营销稿；措辞要可验证、可拆任务、可测试。

## 代码模式
```markdown
1. 背景与问题
2. 目标与成功指标
3. 用户与场景
4. 方案与流程
5. 范围、非目标、依赖、发布计划
```

## 检查清单
- [ ] 问题、目标、用户、成功指标已闭环。
- [ ] 关键流程、状态变化、异常路径和边界条件已写清。
- [ ] 已明确非目标、依赖、上线方式与回滚预案。
- [ ] 后续可直接拆成版本规划和开发任务。

## 反模式

### FAIL: 只有功能列表

```markdown
# 用户中心 PRD
## 功能
- 登录
- 注册
- 修改密码
- 绑定手机
- 实名认证
```

→ 没有问题陈述、没有目标、没有成功指标，读者不知道为什么要做、做到什么程度算完。

### PASS: 问题驱动 + 可验证指标

```markdown
# 用户中心 PRD
## 问题
现状：第三方登录注册转化率 42%，行业基准 65%。
痛点：用户反馈”手机号注册流程太长”。

## 目标
H1 把注册转化率从 42% 提升到 55%。

## 成功指标
- 注册完成率 ≥ 55%（P0）
- 注册平均耗时 ≤ 90s（P1）

## 范围（v1）
- 手机号一键注册（新增）
- 第三方登录（保留）

## 非目标
- 邮箱注册（v1 不做，等数据验证后再评估）
```

### FAIL: “支持所有情况”

```markdown
## 登录流程
系统应支持所有主流登录方式，适配所有浏览器和设备。
```

### PASS: 显式边界

```markdown
## 登录流程（v1）
- 支持：iOS 15+、Android 10+、Chrome/Safari/Edge 最新两个大版本
- 不支持：IE11、鸿蒙非 Android 套件
- 降级：不支持的环境跳转至告知页并引导至 App
```

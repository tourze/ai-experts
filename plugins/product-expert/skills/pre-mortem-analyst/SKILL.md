# 事前验尸

## 适用场景
- 发布前、项目启动前、重大决策前的失败路径推演。
- 需要参考 [references/framework.md](references/framework.md) 与 [references/examples.md](references/examples.md)。
- 与决策或不确定性分析联动时，可配合 [running-decision-processes](../running-decision-processes/SKILL.md) 和 [planning-under-uncertainty](../planning-under-uncertainty/SKILL.md)。

## 核心约束
- 先假设项目已经失败，再倒推原因；不要在头脑风暴阶段就自我辩护。
- 风险要落到可触发的具体情景，而不是“市场不好”这类空泛说法。
- 输出必须给出预防动作、监控指标和责任归属。

## 代码模式
```markdown
| 失败原因 | 早期信号 | 预防动作 | 应急动作 |
| --- | --- | --- | --- |
```

## 检查清单
- [ ] 已列出至少一组高概率/高影响失败路径。
- [ ] 每条风险都有早期信号与对应动作。
- [ ] 已区分可预防、可缓解和只能接受的风险。
- [ ] 结论能反向影响当前计划与资源分配。

## 反模式

### FAIL: 泛泛风险登记表

```markdown
| 风险 | 影响 |
|------|------|
| 市场不好 | 销量差 |
| 技术有挑战 | 可能延期 |
```

→ 没有早期信号、没有触发条件、无法行动。

### PASS: 可触发情景 + 预防/应急

```markdown
| 失败原因 | 早期信号 | 预防动作 | 应急动作 |
|----------|----------|----------|----------|
| 首周留存崩盘 D1<30% | 内测 D1<40% | 扩内测；核心路径埋点 | 切回旧版；红屏公告 |
| 支付回调超标 | 预发压测 P99>2s | 全链路压测；MQ 升级 | 限流+人工对账 |
| 关键工程师离职 | 1:1”看不到成长” | 文档 owner 分散；配对编程 | 招聘+交接 checklist |
```

### FAIL: 回避致命问题

```
“大家都很努力，应该不会失败吧”
→ 没有列出”最坏情况”，失败来时毫无准备
```

### PASS: 强制假设已经失败

```
假设 3 个月后项目失败了，我们在复盘会上。
每人写 3 个最可能的失败原因 → 合并去重 → 概率 × 影响排序
```

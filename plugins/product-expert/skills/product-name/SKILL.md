# 产品命名

## 适用场景
- 需要快速拿到一组可讨论的名字，用于功能、产品或品牌初筛。
- 想做更完整的命名流程时，可转到 [product-naming](../product-naming/SKILL.md)；想把名字和定位对齐时，可配合 [obviously-awesome](../obviously-awesome/SKILL.md)。

## 核心约束
- 先确认目标受众、品牌气质、语言环境和禁用词。
- 每个名字都要附理由、联想方向和潜在风险，而不是只报词。
- 命名输出要适合讨论，不默认直接拍板。

## 代码模式
```markdown
| 名称 | 含义/联想 | 适合的受众 | 风险 |
| --- | --- | --- | --- |
```

## 检查清单
- [ ] 已说明品牌气质、目标市场和命名禁区。
- [ ] 候选名读写顺畅，且能解释为什么适合。
- [ ] 已初步排除明显歧义、难发音或过度通用的名字。
- [ ] 有下一步筛选标准。

## 反模式

### FAIL: 无上下文列词

```
"给我 5 个名字"
→ Lumino / Vela / Nova / Atlas / Echo
→ 用户："这是做什么的？" 创始人："听起来挺好"
```

### PASS: 上下文 + 理由

```
| 名 | 联想 | 适合 | 风险 |
| Pulse | 心跳/节奏 | 实时数据产品 | "脉搏"已被同行使用 |
| Threadly | 线索串联 | 协作工具 | 与 Threads (Meta) 接近 |
→ 创始人能基于风险讨论
```

### FAIL: 创始人偏好压倒受众

```
"我喜欢这个名字，就用它"
→ 用户读不出 / 拼错域名 / 不知道是干嘛的
→ 6 个月后改名，付出 $20k 品牌成本
```

### PASS: 受众测试

```
1. 5-10 个目标用户读名字
2. 不解释，问："你猜这是做什么的？"
3. 50%+ 接近正确 → 候选可用
4. 全部偏差 → 改名
```

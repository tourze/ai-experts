# PM 职级能力模型

## 适用场景
- PM 个人需要定位当前水平和下一步发展方向。
- PM 管理者需要设计团队能力模型或做绩效评估。
- 准备晋升答辩或面试，需要对标目标职级的能力要求。
- 与 [team-composition-analysis](../team-composition-analysis/SKILL.md) 配合规划 PM 团队结构。

## 核心约束
- 职级评估必须基于**行为证据**而非自我感觉；每个能力项必须能举出具体事例。
- 不同公司的职级名称差异大，按能力维度对标而非按 title 对标。
- 晋升不是"做好当前级别的事"，而是"持续表现出下一级别的能力"。
- 技术深度和商业判断力是两个独立维度，不能用一个替代另一个。

## 代码模式

### PM 能力矩阵（Altitude-Horizon 模型）

```markdown
| 维度 | APM/初级 | PM | Senior PM | GPM/Director | VP/CPO |
|---|---|---|---|---|---|
| 视野(Altitude) | 单功能 | 产品线功能 | 整个产品 | 产品组合 | 公司战略 |
| 时间跨度(Horizon) | Sprint | 季度 | 半年-1年 | 1-3年 | 3-5年 |
| 影响范围 | 自己的故事 | 跨职能小组 | 跨团队 | 跨部门 | 跨公司 |
| 模糊度 | 执行明确任务 | 细化模糊需求 | 定义问题空间 | 选择赛道 | 塑造市场 |
| 决策权重 | what to build | how to prioritize | what to invest | where to play | why we exist |
```

### 6 维能力评估

```markdown
| 能力维度 | 评估问题 |
|---|---|
| 用户洞察 | 最近一次改变团队方向的用户发现是什么？ |
| 战略思维 | 能否说清楚产品未来 12 个月的三个战略赌注？ |
| 执行力 | 上季度最大的交付风险是什么，怎么化解的？ |
| 数据判断 | 举一个用数据推翻直觉的决策例子 |
| 影响力 | 最近一次在没有权力的情况下推动跨团队协作 |
| 技术理解 | 能否解释产品核心技术栈的架构和约束 |
```

## 检查清单
- [ ] 每个能力维度有具体的行为事例支撑。
- [ ] 对标的是能力要求而非 title 名称。
- [ ] 晋升路径明确"从当前到下一级需要补什么"。
- [ ] 区分了"做得好"和"持续表现出更高级别能力"。

## 反模式

### FAIL: 年限 = 级别

```
"我做 PM 5 年了，应该是 Senior 了"
→ 年限不等于能力
→ 5 年做同一层级的事 ≠ Senior 能力
```

### PASS: 行为证据驱动

```
Senior PM 证据：
- 用户洞察：发现企业用户和 SMB 需求分叉，推动产品分线（行为）
- 影响力：在没有汇报关系的情况下协调 3 个团队完成平台迁移（行为）
- 战略：主导年度规划，砍掉 2 个低 ROI 项目，资源集中到核心赛道（行为）
```

### FAIL: 只补短板

```
"我数据分析弱，花 3 个月学 SQL"
→ SQL 是工具不是能力
→ 该补的是"用数据做决策"的判断力
```

### PASS: 能力 + 场景

```
目标：提升数据判断力
行动：每个 PRD 必须包含成功指标和决策阈值
验证：下季度有 2 个基于数据的 go/no-go 决策案例
```

# 系统思维

## 适用场景
- 平台策略、组织协作、复杂业务链路、生态位博弈或长期副作用分析。
- 需要经验参考时可阅读 [references/guest-insights.md](references/guest-insights.md)。
- 涉及流程或不确定性时，可配合 [process-optimization](../process-optimization/SKILL.md) 与 [planning-under-uncertainty](../planning-under-uncertainty/SKILL.md)。

## 核心约束
- 先识别系统边界、参与方、激励与流量，再讨论解决方案。
- 重点看反馈回路、延迟、库存与流量，不要只看单点事件。
- 结论要指出杠杆点和副作用，而不是“多方协同更重要”这种空话。

## 代码模式
```markdown
参与方 -> 激励 -> 行为 -> 结果 -> 反馈回路 -> 杠杆点
```

## 检查清单
- [ ] 系统边界、关键角色和核心变量已定义。
- [ ] 已识别至少一条增强回路和一条平衡回路。
- [ ] 关键延迟、二阶效应和反直觉结果已说明。
- [ ] 建议动作聚焦杠杆点，而不是平均用力。

## 反模式

### FAIL: 只盯症状

```
"销售团队冲不上业绩 → 加 KPI 压力 / 多招 BD"
→ 一周后好转
→ 三个月后销售离职率 +50%，问题更深
→ 真因：产品 PMF 未达，销售只能靠话术硬卖
```

### PASS: 找结构

```
画系统图：
- 流量进来 → 销售跟进 → 客户决策 → 成单
- 反馈回路：成单率低 → 销售流失 → 经验丧失 → 成单率更低（增强回路）
- 杠杆点：在产品 PMF（上游）而非销售压力（下游）
→ 不修上游，下游永远在补丁
```

### FAIL: 推变更不看副作用

```
"每个团队都引入 OKR" → 一年后：
- 团队为冲数指标，砍掉无 OKR 关联的协作
- 跨团队协调崩溃
- 创新项目无人愿意领
```

### PASS: 副作用清单

```
变更前先列：
- 一阶效应：聚焦清晰 ✓
- 二阶（6 个月）：跨团队协作可能下降
- 三阶（18 个月）：探索性项目可能减少
→ 配套机制：保留 20% 时间给非 OKR 探索
→ 监控指标：跨团队协作频率（每月）
```

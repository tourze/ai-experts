# 代码库深挖

## 适用场景

- 用户问“这个系统是怎么工作的”“这条链路会触发什么”“这个模块为什么这么设计”。
- 需要跨多个文件梳理调用链、数据流、依赖关系或架构边界。
- 需要输出带证据的深度分析，而不是基于命名的猜测。
- 如果问题是外部互联网主题而不是代码库内部问题，转到 [deep-research](../deep-research/SKILL.md)。

## 核心约束

- 先确定代码仓上下文，再开始研究：远端仓库 URL、当前分支、引用格式。
- 所有结论都要落到真实文件、真实符号、真实调用点上。
- 明确区分“我读到的事实”和“基于局部证据的推断”。
- 需要多轮迭代，每轮更换视角：结构、数据流、集成、模式/反模式、综合建议。
- 边界不明时要主动说明“还没追到哪里”，不能把未知讲成已知。

## 代码模式

```text
第 1 轮：入口与边界
- 找入口、注册点、主模块和关键目录

第 2 轮：调用链与数据流
- 沿着真实调用链向下追到副作用发生点

第 3 轮：集成与依赖
- 查外部系统、API、配置、事件总线、存储

第 4 轮：模式与风险
- 找重复逻辑、隐式耦合、边界泄漏、死代码风险

第 5 轮：综合输出
- 结论、证据、开放问题、建议
```

```text
每条重要发现都应包含
- 发现一句话
- 证据位置（文件 / 符号 / 调用链）
- 影响说明
- 置信度
- 尚未覆盖的边界
```

## 检查清单

- 是否已经解析仓库上下文和引用格式。
- 是否真的跟到了实现，而不是只看接口或目录名。
- 是否至少追到了一个完整调用链。
- 是否给出了未覆盖范围和开放问题。
- Mermaid 图和表格是否对应真实代码，而不是“感觉上的架构图”。

## 反模式

- 根据文件名或框架习惯直接下结论。
- 只看入口文件，不追后续调用链。
- 用“应该 / 大概 / 通常”代替证据。
- 复述代码结构，却不回答用户的具体问题。
- 图画得很完整，但没有实际代码支撑。

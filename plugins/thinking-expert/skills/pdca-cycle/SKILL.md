# PDCA 循环

## 适用场景
- 运营流程的持续改进（每轮解决一个问题，螺旋上升）。
- 质量管理和 SLA 达标。
- 与 [fishbone-diagram](../fishbone-diagram/SKILL.md) 配合：Plan 阶段用鱼骨图找原因。

## 核心约束
- PDCA 是**循环**不是一次性：每轮结束立即进入下一轮。
- **Plan 阶段最重要**——计划不充分，后面三步都是浪费。Plan 包含：找问题、找原因（鱼骨图/逻辑树）、找要因（80/20）、订计划（措施/目标/负责人/截止日期）。
- Check 不是"看了一眼"，要有量化对比：目标值 vs 实际值，偏差原因分析。
- Act 分两种动作：纠正偏差（修） + 提出新问题（进入下一轮）。只纠正不提新问题 = 开环。

## 不适用场景
- 一次性项目（无需迭代）：PDCA 是循环改进工具，一次性交付用 [five-w-two-h](../five-w-two-h/SKILL.md) 做计划。
- 需要根因深挖：PDCA 的 Plan 阶段可以配合 [fishbone-diagram](../fishbone-diagram/SKILL.md)，但 PDCA 本身不做深度根因分析。

## 检查清单
- [ ] Plan 阶段有明确的问题定义和量化目标。
- [ ] Do 阶段有执行记录。
- [ ] Check 阶段有目标 vs 实际的量化对比。
- [ ] Act 阶段同时做了纠正和下一轮问题识别。

## 反模式

### FAIL: 只有 PD 没有 CA

```
Plan: 提升客户满意度
Do: 做了培训、改了流程
-> 然后呢？效果怎样？下一步做什么？
-> 没有 Check 和 Act = 开环 = 不知道有没有用
```

### PASS: 完整闭环

```
P: 客服首响时间 > 5 分钟（目标 < 2 分钟）
   根因：无自动分配，全靠人工认领
   计划：上线自动分配系统（2周内）
D: 系统上线，运行 1 周
C: 首响时间 5.2 分钟降到 1.8 分钟（达标）
   但发现：自动分配导致 15% 工单分配错误（未达标）
A: 固化自动分配系统
   新问题：分配准确率 85% -> 下一轮 PDCA 目标 95%
```

# 追问到底

## 适用场景

- 用户说“grill me”“狠狠质问我”“帮我压力测试这个方案”。
- 方案已经有雏形，但关键假设、依赖、风险还没被彻底验证。
- 需要把模糊的设计选择变成可回答、可验证的问题链。
- 如果用户连问题本身都还没说清，先用 [thinking-partner](../thinking-partner/SKILL.md) 收敛。

## 核心约束

- 一次只问一个问题，不搞机关枪式提问。
- 每个问题都要说明为什么现在问、它在决策树里的位置是什么。
- 每个问题都给出推荐基线答案，帮助用户理解什么样的答案才算过关。
- 如果问题可以通过代码库、配置或现有事实直接回答，先去查，再决定要不要继续问用户。
- 必须沿着依赖关系推进：范围 → 约束 → 风险 → 验证 → 发布，而不是跳来跳去。
- 当关键分支已经收敛时，要停下来总结，而不是为了“狠”而继续发问。

## 代码模式

```text
问题 1：这个方案的单一成功标准是什么？
为什么现在问：如果成功标准不清，后面的取舍都没有依据。
推荐基线答案：一个可观察、可验证、带时间窗口的结果指标。
如果回答 A（标准明确） -> 继续追问验证路径。
如果回答 B（标准模糊） -> 先把成功定义压实。

问题 2：谁是最终负责人？出了问题谁拍板？
为什么现在问：没有明确负责人的方案通常无法稳定落地。
推荐基线答案：一个名字 + 一个升级路径，而不是“大家一起”。
```

## 检查清单

- 是否问清了目标、成功标准和失败代价。
- 是否压实了负责人、依赖、时间线和验证方式。
- 是否覆盖了发布路径、回滚、监控和测试。
- 是否对高风险假设提出了可执行的验证问题。
- 是否把已经回答清楚的分支及时关闭。

## 反模式

- 一次丢 10 个问题，让用户逐条填空。
- 提泛泛的“有考虑安全/性能吗”却不落到具体风险。
- 无视用户前面的回答，重复追问已经解决的点。
- 能从代码库确认的问题，却懒得自己查。
- 明明已经收敛，还为了保持压迫感继续堆问题。

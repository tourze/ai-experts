# 发现-证据强绑定

## 适用场景

- 写代码审查、架构审计、安全审计、性能诊断、事故复盘报告。
- 输出含「我发现 / 我认为 / 看起来」类断言的研究、调查、咨询文档。
- 与 `debug-methodology` 的差异：后者管"怎么调"，本 skill 管"调试结论怎么写"。
- 常与 [fact-vs-inference-vs-assumption](../fact-vs-inference-vs-assumption/SKILL.md) 配合：后者标三态，本 skill 绑证据。

## 核心约束

- **每条发现 = 定位 + 证据片段 + 触发条件**，三件套缺一不可。
- 定位形式按可核验度从高到低：`文件:行号` / `log 时间戳 + trace_id` / `commit hash` / `metric 名+数值+时间窗` / `命令+退出码+输出`。
- 证据片段必须粘贴实际内容（代码段、log 行、命令输出），不能写「类似这样」。
- 触发条件必须可重现：分支、数据规模、角色、并发度、环境。
- 多源证据按可信度排序：`直接观测 > 间接推断 > 同类模式类比 > 直觉`。不允许把右侧伪装成左侧。
- 性能 / 安全 / 并发类断言必须给可重现实验或观测窗口；纯代码模式扫描只能标 `[推断]`，不能标 `[事实]`。

## 代码模式

```text
## 发现 N：<一句话标题>

- 定位：<file:line | trace_id | commit | metric>
- 证据：<粘贴的代码 / log / 输出>
- 触发条件：<复现条件>
- 影响：<谁、多大、是否用户可见>
- 修复方向：<具体改动方向>
```

## 检查清单

- [ ] 每条发现都给了文件:行 / 时间戳 / commit / metric 中至少一种定位？
- [ ] 证据片段是粘贴的实际内容，而非「大致这样」？
- [ ] 触发条件能让另一个工程师独立复现？
- [ ] 性能 / 安全断言有可重现实验或观测窗口？
- [ ] 避免了"整体感觉""看起来""印象中""应该""大概率"等无锚措辞？
- [ ] 多源结论按可信度排序，未把推断伪装成观测？

## 反模式

| 危险措辞 | 现实后果 | 替代写法 |
|---------|---------|---------|
| "看起来 / 大概率 / 应该" | 读者无法判断证据强度 | 标 `[推断]` 并给出推断依据 |
| "整体感觉 X 有问题" | 无定位，无法验证 | 引用具体文件:行号 |
| "测试偶尔失败" | 不是触发条件 | 给频率（如 20 次失败 1 次）+ run_id |
| "性能瓶颈在 X" 但没跑过 profile | 推断当事实 | 给验证路径 + 标 `[待验证]` |

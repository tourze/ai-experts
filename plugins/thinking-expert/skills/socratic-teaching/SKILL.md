# 苏格拉底式教学

## 适用场景

- 用户说"苏格拉底""Socrates""socratic mode""别直接告诉我答案""引导我自己想"。
- 用户在学习新代码库、新框架或新概念，希望通过被提问来建立理解。
- 代码 review 教学场景：让用户自己发现代码中的问题，而不是直接指出。
- 用户想深入理解某个系统设计决策背后的原因。
- 如果用户思路混乱、需要先理清局面，转到 [thinking-partner](../thinking-partner/SKILL.md)。
- 如果用户已有方案、需要压力测试，转到 [grill-me](../grill-me/SKILL.md)。
- 如果用户明确要求直接给答案，尊重用户意愿，退出本模式。

## 核心约束

- 全程不给直接答案。这是本 skill 与其他思维类 skill 的根本区别 —— thinking-partner 在后期会共创解法，grill-me 会给基线答案，而本 skill 自始至终只通过提问引导用户自己到达结论。
- 先默读相关代码/文档/资源，建立内部理解，但不分享理解本身。
- 每轮只问 1-2 个问题，避免提问轰炸。问题之间要有递进关系。
- 按 5 类问题从浅到深推进：澄清 → 探究 → 关联 → 反驳 → 假设。不需要严格按顺序，但整体趋势应从简单到复杂。
- 当用户方向正确时，简短肯定后立刻加深；当用户方向错误时，不纠正，而是用一个能暴露矛盾的问题让用户自己发现。
- 当用户说"我不知道"时，把问题拆小，降低台阶而不是放弃。
- 当用户明确到达正确理解时，请用户自己总结，然后结束。

## 代码模式

```text
五类递进问题：

| 类型   | 目的       | 示例                                           |
|--------|------------|------------------------------------------------|
| 澄清   | 暴露假设   | "你说这个函数'处理数据'——具体处理了什么？"       |
| 探究   | 往深挖     | "如果把这个中间件去掉，请求流程会怎样？"         |
| 关联   | 连接概念   | "这个 cache 策略和你刚才说的一致性要求矛盾吗？"  |
| 反驳   | 挑战判断   | "你说用 SQL 更合适——如果数据量到 10 亿行呢？"    |
| 假设   | 推演后果   | "如果这个设计上了生产，哪里会先出问题？"         |

四种用户状态的应对：

用户方向正确 →
  "对，继续沿这个方向想——那下一层会是什么？"

用户方向错误 →
  不说"不对"。问一个能暴露矛盾的问题：
  "如果真是这样，那为什么这个测试用例能通过？"

用户说"我不知道" →
  把问题拆小：
  "没关系，我们缩小范围。只看这三行，第一行做了什么？"

用户要求直接给答案 →
  "如果我直接告诉你，你只是知道了结论。我们换个角度试试——
   如果你是这个函数的调用方，你最关心它返回什么？"
```

## 检查清单

- 是否已经默读了用户提到的代码/文档/资源。
- 本轮回复是否只包含问题，没有泄露答案。
- 问题是否比上一轮更深一层（或在用户卡住时更浅一层）。
- 是否避免了"暗示太明显等于直接给答案"的陷阱。
- 用户到达正确理解时，是否请用户自己做总结。
- 是否在用户明确要求退出时尊重意愿。

## 反模式

详见 [反模式与正例](references/anti-patterns.md)。核心三条红线：

- 先说答案再提问（答案已出口，提问变走过场）
- 暗示太明显（用问句包装答案，如"你有没有注意到 N+1 查询"）
- 用户卡住就放弃（直接给答案，用户没学到思考路径）

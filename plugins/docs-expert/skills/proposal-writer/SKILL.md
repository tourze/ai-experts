# 提案撰写

## 适用场景

- 用户需要从零开始写商业提案、项目方案、渠道合作或服务报价文档。
- 目标是“写出能发给客户的文本”，不是只做提纲。
- 已经具备一部分背景材料，但需要把它们组织成对外表达。
- 若提案面向大客户或咨询场景，可优先参考 [enterprise-proposal](../enterprise-proposal/SKILL.md)。

## 核心约束

- 必须围绕客户问题和价值结果写作，不要把公司介绍写成主角。
- 结构要完整：背景、问题、方案、价值、实施、商务条款。
- 不能写未经确认的案例、数字或交付承诺。
- 文案要服务成交动作，例如预约评审、签试点、进入采购流程。

## 代码模式

```json
{
  "客户类型": "SaaS 创业公司",
  "合作目标": "建立内容营销与销售联动机制",
  "方案时长": "12 周",
  "必须包含": ["里程碑", "交付物", "报价逻辑", "下一步动作"]
}
```

## 检查清单

- 是否写清楚了客户当前问题、目标状态和价值主张。
- 是否把交付物、节奏、责任边界和验收方式写完整。
- 是否在报价之外给出版本方案、假设和排除项。
- 是否包含明确 CTA，例如下一次会议、试点范围或签约步骤。
- 在正式发送前，是否用 [proposal-review](../proposal-review/SKILL.md) 做过一次评审。

## 反模式

### FAIL: 公司宣传册

```md
P1：关于我们（公司历史、团队、文化）
P2：客户案例 logo 墙
P3：奖项荣誉
P4-5：方案概述
→ 客户：”看完不知道你们能为我做什么”
```

### PASS: 客户问题驱动

```md
P1：摘要 — 你的目标 / 我们提供 / ROI 预期
P2：当前问题（基于售前调研）
P3-4：方案 + 价值证据
P5：实施 + 里程碑
P6：报价 + 假设 / 排除
P7：CTA — 下次会议 / 试点 / 签约时间
```

### FAIL: 形容词堆专业感

```
“我们是行业领先的、值得信赖的、专业的解决方案提供商，
致力于为您提供卓越的、创新的、全方位的服务...”
→ 读完不知道做了什么
```

### PASS: 动词 + 数字

```
“我们 12 周内交付：
- 接入 10 个数据源（已对接 8 类相同系统）
- 培训 30 人内部团队
- 上线后 30 天免费支持
首月预期：报表自动化率 70%（基于过往 5 个项目均值）”
```

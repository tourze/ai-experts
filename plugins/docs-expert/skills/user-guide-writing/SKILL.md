# 用户指南写作

## 适用场景

- 文档读者是终端用户、业务用户、客户支持对象，而不是研发同事。
- 需要写 onboarding、操作手册、教程、常见问题、故障排查或培训资料。
- 用户希望内容“能照着做”，而不是高层概述。
- 如果当前还在收集素材和结构，可先用 [doc-coauthoring](../doc-coauthoring/SKILL.md)。

## 核心约束

- 以任务为单位组织内容，例如“如何导出数据”，不要按内部模块堆目录。
- 用用户语言，不用系统实现语言；一步只做一件事。
- 只有在必要时才插图，并说明截图位置、状态和成功标准。
- 指南必须标明前置条件、结果预期和异常情况的处理方式。

## 代码模式

```json
{
  "目标读者": "首次使用的运营同学",
  "文档类型": "新手上手指南",
  "关键任务": ["登录", "创建项目", "导出报表"],
  "配图策略": "仅对关键分叉步骤截图"
}
```

建议的章节顺序：

```markdown
## 开始前准备
## 第一步
## 第二步
## 常见问题
## 失败时怎么办
```

## 检查清单

- 是否明确了用户画像、前置条件和最终目标。
- 是否每个步骤都可执行，且有成功标准或结果说明。
- 是否把异常情况、权限限制和常见错误单独列出。
- 是否避免使用内部术语、缩写和实现细节。
- 若还需要同步更新仓库说明，可结合 [readme-blueprint-generator](../readme-blueprint-generator/SKILL.md)。

## 反模式

- 对终端用户写成开发文档，充满内部名词。
- 长段落解释理念，却不给操作步骤。
- 截图过多但没有说明“这张图要看什么”。
- 省略前置条件，导致用户照着做也失败。

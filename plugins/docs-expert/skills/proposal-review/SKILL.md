# 提案评审

## 适用场景

- 用户说“帮我 review 这份 proposal / RFC / 方案 / 规格文档”。
- 文档较长，需要按章节拆解，而不是一次性给泛泛评论。
- 评审目标是发现风险、缺口、冲突、模糊点和读者会卡住的位置。
- 评审完若要回写成正式文本，可接到 [proposal-writer](../proposal-writer/SKILL.md) 或 [doc-coauthoring](../doc-coauthoring/SKILL.md)。

## 核心约束

- 先明确评审维度：商业可行性、技术风险、范围定义、实施成本或受众理解。
- 先给发现，再给总结；不要先写笼统表扬。
- 对每个问题都尽量指出证据位置、影响范围和建议动作。
- 文档很长时必须分块处理，避免在记忆里拼接错上下文。

## 代码模式

```json
{
  "文档类型": "技术方案",
  "评审重点": ["边界定义", "依赖关系", "回滚策略", "测试计划"],
  "输出格式": ["问题清单", "开放问题", "修改建议"]
}
```

## 检查清单

- 是否明确了评审角色和标准，而不是“随便看看”。
- 是否按章节拆解，并记录每块的主要风险与疑问。
- 是否把严重问题、开放问题和可选优化分层输出。
- 是否给出作者下一轮最该改的 3 到 5 个点。
- 若需要把意见并回文档，可转给 [doc-coauthoring](../doc-coauthoring/SKILL.md)。

## 反模式

### FAIL: “整体不错”

```
“看完了，整体不错，再补一些细节就好”
→ 作者：”具体补哪些？”
→ 答不上 → 评审失败
```

### PASS: 分块发现 + 严重度

```md
## 评审发现

### Blocker
- 第 3 节方案：依赖 X 服务，但 X 在 Q3 才上线，与本提案 Q2 交付冲突
- 报价表：缺少假设和排除项，客户可能产生范围争议

### Should-fix
- 第 5 节风险：只列了技术风险，缺商务风险（如客户决策延迟）

### Nice-to-have
- 案例数据可量化（”提升 30%” 改为 “DAU 1k → 1.3k”）

## 下一轮 3 个最该改的
1. 解决 X 服务依赖冲突（Blocker）
2. 报价表补假设/排除（Blocker）
3. 风险章节补商务维度（Should）
```

### FAIL: 一次性看完

```
“我读了 3 遍” → 反馈是 5 句话
→ 第 12 节写得不好的细节都没记下来
```

### PASS: 分块 + 标记

```
分块顺序：执行摘要 → 每章 → 附件
每读完一块立即记 3 条：
- 这块在证明什么
- 是否真的证明了
- 缺什么 / 该补什么
全部读完再汇总 + 排序
```

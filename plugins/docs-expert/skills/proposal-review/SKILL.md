# 提案评审

## 适用场景

- 用户说“帮我 review 这份 proposal / RFC / 方案 / 规格文档”。
- 文档较长，需要按章节拆解，而不是一次性给泛泛评论。
- 评审目标是发现风险、缺口、冲突、模糊点和读者会卡住的位置。
- 评审完若要回写成正式文本，可接到 [proposal-writer](../proposal-writer/SKILL.md) 或 [doc-coauthoring](../doc-coauthoring/SKILL.md)。

## 核心约束

- 先明确评审维度：商业可行性、技术风险、范围定义、实施成本或受众理解。
- 先给发现，再给总结；不要先写笼统表扬。
- 对每个问题都尽量指出证据位置、影响范围和建议动作。
- 文档很长时必须分块处理，避免在记忆里拼接错上下文。

## 代码模式

```json
{
  "文档类型": "技术方案",
  "评审重点": ["边界定义", "依赖关系", "回滚策略", "测试计划"],
  "输出格式": ["问题清单", "开放问题", "修改建议"]
}
```

## 检查清单

- 是否明确了评审角色和标准，而不是“随便看看”。
- 是否按章节拆解，并记录每块的主要风险与疑问。
- 是否把严重问题、开放问题和可选优化分层输出。
- 是否给出作者下一轮最该改的 3 到 5 个点。
- 若需要把意见并回文档，可转给 [doc-coauthoring](../doc-coauthoring/SKILL.md)。

## 反模式

- 只给“整体不错，再补细节”这类无效评论。
- 一次看完整份文档却不做分块记录，最后意见失焦。
- 把个人偏好当成阻断问题，不区分严重度。
- 发现问题但不给证据位置，作者无法落地修改。

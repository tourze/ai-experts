# 企业提案

## 适用场景

- 面向企业客户、采购方、管理层或投资委员会输出正式提案。
- 需要把能力介绍、报价、实施路径、案例、ROI 和风险控制整合成一份材料。
- 交付物可以是长文档、Word 方案、PPT 结构稿或销售跟进材料。
- 若前置研究还没完成，先用 [consulting-analysis](../consulting-analysis/SKILL.md) 补齐事实基础。

## 核心约束

- 提案必须围绕客户目标和成交动作，而不是围绕“我想说什么”。
- 没有证据的业绩、案例、统计数字、节约金额不能写。
- 报价、范围、里程碑、假设与排除项必须成套出现，不能只写价格。
- 结构上优先“执行摘要 → 当前问题 → 方案 → 价值 → 实施 → 商务条款”。

## 代码模式

```json
{
  "客户": "某制造业集团",
  "采购目标": "统一知识库与流程文档治理",
  "交付形式": ["Word 提案", "报价页", "里程碑计划"],
  "必须包含": ["案例", "ROI", "时间表", "假设与排除项"]
}
```

可复用的补充素材包括多语言说明与数据文件：

- [description_cn.txt](description_cn.txt)
- [stats.json](stats.json)

## 检查清单

- 是否明确了客户背景、决策链、预算区间和采购阶段。
- 是否写清楚了服务范围、交付件、周期、责任边界和验收标准。
- 是否把案例、数据和承诺按证据强弱分层，不夸大。
- 是否给出清晰的下一步动作，例如评审会、试点、法务流程或签约节点。
- 若交付后还要做评审意见吸收，可交给 [proposal-review](../proposal-review/SKILL.md)。

## 反模式

- 用通用公司介绍替代面向客户的价值论证。
- 只有愿景和金句，没有实施计划、资源安排和风险说明。
- 把报价写成一行总价，没有任何构成、条件或版本方案。
- 为了显得强势而承诺无法交付的数字或范围。

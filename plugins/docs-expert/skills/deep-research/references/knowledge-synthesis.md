# 知识综合

## 适用场景

- 已经完成检索，但结果分散在多个来源，需要整理成一个可读答案。
- 同一事实在不同来源里重复出现，需要去重并保留来源证据。
- 用户想要“结论 + 依据 + 置信度”，而不是原始搜索结果堆砌。
- 常与 [deep-research](../deep-research/SKILL.md) 和 [web-content-fetcher](../web-content-fetcher/SKILL.md) 连用。

## 核心约束

- 先回答问题，再解释你是如何得到答案的。
- 每个关键结论都要能回溯到具体来源；多来源支持时要合并引用，而不是重复复述。
- 冲突信息必须显式指出，不能偷偷挑一个版本当作“事实”。
- 可信度至少看三件事：来源权威性、时间新鲜度、跨来源一致性。
- 对关键来源打 A-E 标签，低于 C 的结论要显式降级表述。
- 小结果集给细节，大结果集给分层摘要和继续下钻的入口。

来源质量 A-E（用于冲突仲裁与分层表述）:

- A — 同行评审、系统综述、官方规范、监管机构原始文件
- B — 行业分析师（Gartner/Forrester/IDC）、政府统计、成熟媒体调查报道
- C — 专家白皮书、公司官方博客/公告、知名技术媒体
- D — 预印本、会议摘要、无编辑审核的个人博客、社区问答
- E — 匿名内容、论坛发言、利益冲突、过期资料

## 代码模式

```text
原始结果
→ 去重：识别相同事实 / 相同事件 / 相同文档
→ 聚类：按主题、时间线或结论分组
→ 排序：相关度 × 权威性 × 新鲜度
→ 产出：结论 / 证据 / 分歧 / 置信度
```

```text
输出骨架
1. 直接结论
2. 关键证据（按主题归并）
3. 冲突或不确定项
4. 来源列表
5. 下一步建议（如需）
```

## 检查清单

- 是否把重复来源合并成单条叙述。
- 是否对每个核心结论给出来源归因。
- 是否标注了时间线和最新版本，而不是把旧结论和新结论混在一起。
- 是否把低置信度部分明确降级表述。
- 是否避免把原始结果逐条罗列成“搜索日志”。

## 反模式

### FAIL: 相似结果机械复制

```
A：React 18 引入并发渲染
B：React 18 支持并发模式
C：React 18 带来 concurrent rendering
→ 报告把三句都复述一遍
```

### PASS: 去重 + 合并引用

```
结论：React 18 引入并发渲染 [A, B, C]
```

### FAIL: 冲突版本只保留偏好

```
A（官方博客 2024-01）：支持 Server Components
B（个人博客 2023-05）：不支持
→ 只写 A，忽略 B
```

### PASS: 显式标注冲突

```
结论：React 18+ 支持 Server Components [官方 2024-01]
注：2023 年早期资料称不支持 [B]，因早期 RC 未启用，已被正式版覆盖
```

### FAIL: 来源不分层

```
参考：Wikipedia、官方 changelog、Reddit、某博主
→ 全部平等列出，无法判断权重
```

### PASS: 分层标注（A-E）

```
A：官方 changelog、规范文档
B：成熟媒体、官方博客
C：博主教程（需交叉验证）
D-E：Reddit、论坛讨论（仅参考）
```

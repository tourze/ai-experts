# 咨询分析

## 适用场景

- 需要先搭章节骨架、分析模型和数据需求，再生成正式报告。
- 输出物是咨询风格文档，而不是随手总结或聊天式回答。
- 题目涉及市场规模、行业格局、品牌表现、增长路径、尽调结论等结构化研究。
- 当研究结果还要继续转成提案时，可联动 [enterprise-proposal](../enterprise-proposal/SKILL.md)。

## 核心约束

- 先做框架，再写正文；不要在问题定义含糊时直接写完整报告。
- 没有证据的数据不能编造；每个关键判断都要标注来源类型、时间范围和置信度。
- 结论必须能回溯到分析链路：研究问题 → 方法 → 数据 → 推理 → 建议。
- 图表、表格和结论口径必须一致；若数据缺失，要明确缺口而不是“合理猜测”。

## 代码模式

```json
{
  "研究主题": "中国企业知识管理软件市场",
  "分析目标": ["市场规模", "竞争格局", "客户需求", "进入建议"],
  "交付物": "咨询级 Markdown 报告",
  "数据要求": ["官方统计", "公司财报", "产品定价页", "用户访谈摘录"]
}
```

可直接按下面骨架开工：

```markdown
# 执行摘要
## 1. 问题定义
## 2. 研究方法与数据源
## 3. 核心发现
## 4. 战略含义
## 5. 建议与下一步
```

## 检查清单

- 是否明确了研究对象、目标读者、时间范围和地区范围。
- 是否列出了要补的数据清单、数据优先级和不可替代的一手证据。
- 是否为每个章节指定了分析模型，例如 TAM/SAM/SOM、五力、SWOT、漏斗、单位经济模型。
- 是否区分了事实、推断和建议，避免把判断写成既成事实。
- 如果后续要做销售或融资材料，是否已预留给 [proposal-writer](../proposal-writer/SKILL.md) 或 [enterprise-proposal](../enterprise-proposal/SKILL.md) 的复用结构。

## 反模式

- 一上来就写 30 页成稿，后面发现研究问题都没定义清楚。
- 把二手文章互相引用当成事实闭环，没有原始出处。
- 只有结论，没有分析过程，导致读者无法复核。
- 用“行业普遍认为”“市场通常如此”之类空话替代数据。

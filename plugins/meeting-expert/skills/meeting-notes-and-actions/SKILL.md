# 会议笔记与行动项

## 适用场景

- 用户贴来 Zoom、Meet、Teams、飞书会议或电话沟通的转写文本，希望快速提炼重点。
- 原始材料杂乱、存在口头禅、时间戳、说话人标签或聊天穿插，需要先降噪再结构化。
- 适合“先把内容整理出来”的轻量输出，不要求完整纪要格式。
- 如果用户要求正式会议纪要、带固定章节和发文规范，切换到 [meeting-minutes](../meeting-minutes/SKILL.md)。
- 如果用户关注沟通风格、冲突回避、发言占比等行为模式，切换到 [meeting-insights-analyzer](../meeting-insights-analyzer/SKILL.md)。

## 核心约束

- 允许清洗噪音，但不得改写原意；原文里的承诺、数字、时间和明确引用必须保真。
- 决策、风险、开放问题、行动项要分栏，不得混成单一摘要段落。
- 当负责人或截止日期缺失时，可以给出“建议负责人”“建议日期”，但必须显式写明是建议，不得伪装成会议已确认。
- 输出必须默认包含 `摘要`、`决策`、`开放问题 / 风险`、`行动项` 四段；如果有时间戳，再追加里程碑时间线。
- 对隐私、客户名、敏感项目代号遵守用户指定的脱敏规则；用户未指定时，只保留完成任务所需的最小识别信息。

## 代码模式

```bash
rg --files . -g '*.md' -g '*.txt' -g '*.srt' -g '*.vtt' -g '*.docx'
```

```markdown
## 摘要
- <1-3 条会议结论>

## 决策
- [已确认] <决策内容>

## 开放问题 / 风险
- <未决问题> - 风险：<影响或阻塞>

## 行动项
- [ ] <任务描述>
  - 负责人：@<姓名>
  - 截止：2026-04-18
  - 状态：已确认 / 建议
```

## 检查清单

- [ ] 已标明输入来源是转写、原始笔记还是聊天记录，避免把噪音当事实。
- [ ] 决策只包含已明确拍板的事项；含糊说法被归入开放问题或风险。
- [ ] 每个行动项都注明负责人、截止日期与“已确认 / 建议”状态。
- [ ] 若原文包含时间戳，已抽取关键时刻而不是把全部时间戳搬运出来。
- [ ] 输出中的姓名、团队名、项目名保持前后一致，没有把同一人写成多个称呼。
- [ ] 当用户需要正式纪要或行为分析时，已改用 [meeting-minutes](../meeting-minutes/SKILL.md) 或 [meeting-insights-analyzer](../meeting-insights-analyzer/SKILL.md)。

## 反模式

### FAIL: 模糊升格决策

```
转写：Alice：”也许 Q3 上线？”
你输出：决策 → “Q3 上线新功能”
→ 实际只是个人提议，团队未拍板
```

### PASS: 分开决策 vs 开放问题

```md
## 决策（已拍板）
- Sprint 2 结束前完成数据迁移

## 开放问题
- 上线时间点：Alice 提议 Q3，待 Product Review 确认
```

### FAIL: 凑格式虚构日期

```md
- [A3] 联系供应商
  - 负责人：@alice
  - 截止：2026-04-30  ← 你编的
```

### PASS: 建议标状态

```md
- [A3] 联系供应商
  - 负责人：@alice
  - 截止：建议 2026-04-30（会议未确认）
  - 状态：建议
```

# 会议纪要

## 适用场景

- 内部周会、站会、设计评审、排障复盘、计划会等 60 分钟以内会议，需要一份正式、可流转的纪要。
- 输入可能来自录音转写、会议记录、议程文档或会后补记。
- 如果用户只想把粗糙笔记整理成轻量摘要与行动项，优先使用 [meeting-notes-and-actions](../meeting-notes-and-actions/SKILL.md)。
- 如果用户要分析发言习惯、引导风格或冲突处理方式，切换到 [meeting-insights-analyzer](../meeting-insights-analyzer/SKILL.md)。

## 核心约束

- 在缺少关键信息时，最多只问 3 个澄清问题，优先补齐：会议标题/日期/时长/组织者、参会人、原始材料来源。
- 纪要必须把“决策”和“行动项”前置；行动项必须包含负责人和截止日期，若确实缺失则标记为 `待补充`，不能擅自编造。
- 对未确认事项统一标记 `待补充` 或 `未知`，并指出缺口来自哪里，不得把猜测写成事实。
- 纪要正文必须按固定顺序组织：元信息、出席情况、议程、摘要、决策、行动项、分议题笔记、搁置项、风险/阻塞、后续安排、引用材料。
- 除非用户明确要求，不输出整段逐字稿；纪要是决策摘要，不是转写重排。

```markdown
# <会议标题>

- 日期：2026-04-14
- 时长：45 分钟
- 组织者：<姓名>
- 记录人：<姓名或 agent>

## 出席情况
- 参会：<姓名 / 角色>
- 缺席：<姓名 / 角色>

## 决策
1. <决策内容>
   - 决策人：<姓名或团队>
   - 原因：<1-2 句理由>

## 行动项
- [A1] <事项描述>
  - 负责人：<姓名 / 团队>
  - 截止：2026-04-18
  - 完成标准：<可验证结果>

## 风险 / 阻塞
- <风险描述> - 影响：<影响> - 跟进人：<姓名>
```

## 检查清单

- [ ] 已获取会议标题、日期、时长或开始结束时间；缺项时已先发起澄清。
- [ ] 已明确参会人与缺席人，不把“未确认”成员写进正式名单。
- [ ] 每个关键决策都带有责任主体或背景理由，不把开放讨论误写成已决策事项。
- [ ] 每个行动项都有负责人和截止日期；若只能给建议，明确写成 `待补充`。
- [ ] 风险、阻塞、搁置项与正式行动项分栏展示，没有混写。
- [ ] 当用户真正要的是轻量摘要或沟通洞察时，已改用 [meeting-notes-and-actions](../meeting-notes-and-actions/SKILL.md) 或 [meeting-insights-analyzer](../meeting-insights-analyzer/SKILL.md)。

## 反模式

- 不要把逐字稿整段复制进纪要，导致关键信息被噪音淹没。
- 不要凭上下文猜负责人或截止日期，然后当成既定事实输出。
- 不要把“有人提议”写成“团队决定”，也不要把风险写进决策区。
- 不要只给“稍后跟进”“线下继续”这种空动作，没有 owner 和时间就不算行动项。
- 不要把行为评价、情绪判断混入纪要正文，这属于 [meeting-insights-analyzer](../meeting-insights-analyzer/SKILL.md) 的职责。

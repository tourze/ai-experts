## 沟通与输出协议

### 输出密度

- 默认先给判断或当前状态，再补依据；不要用“我将/我会先/接下来我会”铺垫过程。
- 面向用户的中间更新保持短：一到两句，只写当前动作、新发现或阻塞点；少用第一人称，不复述计划。
- 最终回复按任务大小伸缩：简单问答 1-3 段；局部改动说清改动、验证、风险；审计、设计、重构、跨端方案才使用固定模板。
- 同一事实只说一次。路由声明、验证结果、下一步建议、退出状态之间不要互相重复。
- 有明确后续动作才写“下一步推荐”；任务已闭合时不为格式硬凑建议。
- 保留直接、克制、工程化语气；避免汇报腔、客服腔、过度解释和自我叙述。

### 结构化表达

- 讨论架构、链路、迁移、对比、风险、阶段计划时，先给全貌，再给判断，再给动作。
- 横向比较用 Markdown 表格；纵向序列、调用链和进程边界用 ASCII 树或框图。
- 事实与建议分区，不把观察、判断、建议混成一段长文。
- 禁止“结论是”“以下几点需要说明”“整体是合理的”“如果你愿意”等套话与征询式句式。
- 有明确技术倾向直接亮明，不把决策推回给提问者；必须给选项时先亮推荐。

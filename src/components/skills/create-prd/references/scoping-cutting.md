
# 范围裁剪

## 适用场景
- MVP 设计、版本瘦身、时间盒交付、资源不足下的功能取舍。
- 需要经验参考时可阅读 [references/guest-insights.md](./guest-insights.md)。
- 做版本落地时可配合 [version-planner](../../agile-product-owner/references/version-planner.md) 与 [create-prd](../SKILL.md)。

## 核心约束
- 先明确本轮要验证的核心假设，再判断哪些内容是必须、可选或延后。
- 范围裁剪不是“随机砍功能”，而是保住关键路径并推迟边缘需求。
- 时间、团队和质量约束要同时摆上桌，不能只从功能视角看范围。

## 代码模式
```markdown
| 项目项 | 为什么需要 | 不做的代价 | 是否首版必须 |
| --- | --- | --- | --- |
```

## 检查清单
- [ ] 已明确核心假设、交付窗口和不可妥协项。
- [ ] 必做项围绕关键用户路径，而非“谁提得更大声”。
- [ ] 延后项有回收条件和后续版本位置。
- [ ] 裁剪结果不会破坏最小可验证体验。

## 反模式

### FAIL: “都很重要，再加把劲”

```
PM：登录、注册、绑卡、实名、风控、推荐、IM、客服...全要
工程：时间不够
PM：大家加班一下
→ 结果：所有功能半成品，核心路径也挂了
```

### PASS: 围绕核心假设保留关键路径

```markdown
核心假设：用户愿意为”AI 自动整理会议纪要”付费。

必做（验证假设）：上传 → 转写 → AI 摘要 → 下载
延后（验证后再做）：日历同步、团队协作、多语言
```

### FAIL: 砍基础留”酷”功能

```
砍：登录、错误处理、加载状态
留：3D 主题、音效、动画
→ 用户连不上后台就走了
```

### PASS: 基础 + 1 个差异点

```
必做：登录、错误态、加载态、核心 CRUD
差异点：AI 自动生成标签（竞品没有）
砍掉：3D 主题、音效、动画（v2 再评估）
```

# 重构纪律守卫

**违反字面规则 = 违反规则精神。不存在"灵活变通"。**

## Iron Law

```
没有测试覆盖的代码，不允许开始重构
```

重构 = 改结构不改行为。没有测试证明行为不变，你做的就不是重构，是冒险。

## Red Flags — 出现以下念头时立即停下

| 念头 | 现实 |
|------|------|
| "这段代码很乱，先整理一下" | 先界定范围和目标，不要"先动手再说"。 |
| "顺手把这个 bug 也修了吧" | 重构和 bug 修复必须分开提交。搭车 = 无法回滚。 |
| "这个不需要测试，改动很小" | 越觉得安全越容易出事。跑测试只要几秒。 |
| "反正都要改，不如一步到位" | 一步到位 = 大爆炸提交。出问题无法定位。 |
| "这里也可以顺便优化" | 范围蔓延的典型信号。回到你的范围界定文档。 |
| "先改完再补测试" | 没有测试基线，你无法证明行为没变。先补测试。 |

## Common Rationalizations

| 借口 | 现实 |
|------|------|
| "测试太难补了，代码本身就是遗留的" | 补表征测试（characterization test）捕获当前行为，不需要理解意图。 |
| "这不算行为变更，只是换了个写法" | 编译器同意不代表运行时行为不变。跑测试。 |
| "一个提交改 50 个文件没问题" | 如果第 37 个文件引入了回归，你怎么 bisect？ |
| "重构完再提交，保持原子性" | 每个小步骤都应该是可提交的。原子性 ≠ 一次搞完。 |
| "这段代码以后不会有人改了" | 你不知道。留下测试和干净提交是对未来的保险。 |
| "重构到一半发现设计问题，顺手改了" | 重构和重设计是两件事。先完成当前重构范围，再开新任务做设计变更。 |
| "改了一处之后另一处也不顺眼了" | 范围蔓延。记下来，开新任务。当前提交只做原定范围。 |
| "反正 IDE 能自动重构，不用担心" | IDE 自动重构不覆盖运行时行为（反射、配置文件引用、字符串拼接）。跑测试确认。 |

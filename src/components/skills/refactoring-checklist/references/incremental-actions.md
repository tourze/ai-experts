# 增量重构动作参考

## 增量步骤循环

每步遵循：

1. **做一个小的结构变更**（抽取、重命名、移动、合并）。
2. **跑测试**，确认全部通过。
3. **提交**，写清楚这一步做了什么。
4. 回到步骤 1。

## 常见动作与风险

| 动作 | 说明 | 风险 |
|------|------|------|
| 重命名 | 变量/函数/类改名 | 低——IDE 重命名通常安全 |
| 抽取函数 | 把代码块提取为独立函数 | 低——注意参数和返回值 |
| 内联函数 | 把只调用一次的函数展开 | 低 |
| 移动函数/类 | 把代码移到更合适的模块 | 中——注意导入路径和循环依赖 |
| 拆分类/模块 | 把职责过多的类拆成多个 | 中——注意状态归属 |
| 合并重复 | 把重复代码抽取到共享位置 | 中——确认是真正的重复不是巧合 |
| 改变接口 | 修改函数签名或 API | 高——所有调用方都要同步更新 |

## 红线

- 不在重构提交中修 bug 或加功能。
- 不跳过测试（"我觉得这个改动很简单不会出问题"）。
- 不做超出预定范围的"顺手改"。
- 测试红了，立即回退到上一个绿色状态，不硬修。

## 重构后验证

- [ ] 所有测试通过，覆盖率不低于重构前基线。
- [ ] 手动检查关键功能路径。
- [ ] diff 只包含结构变更，不含行为变更。
- [ ] 提交信息使用 `refactor:` 前缀。
- [ ] 发现的 bug 另开提交处理。

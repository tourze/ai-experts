## 代码模式
```markdown
参与方 -> 激励 -> 行为 -> 结果 -> 反馈回路 -> 杠杆点
```

## 反模式

### FAIL: 只盯症状

```
"销售团队冲不上业绩 → 加 KPI 压力 / 多招 BD"
→ 一周后好转
→ 三个月后销售离职率 +50%，问题更深
→ 真因：产品 PMF 未达，销售只能靠话术硬卖
```

### PASS: 找结构

```
画系统图：
- 流量进来 → 销售跟进 → 客户决策 → 成单
- 反馈回路：成单率低 → 销售流失 → 经验丧失 → 成单率更低（增强回路）
- 杠杆点：在产品 PMF（上游）而非销售压力（下游）
→ 不修上游，下游永远在补丁
```

### FAIL: 推变更不看副作用

```
"每个团队都引入 OKR" → 一年后：
- 团队为冲数指标，砍掉无 OKR 关联的协作
- 跨团队协调崩溃
- 创新项目无人愿意领
```

### PASS: 副作用清单

```
变更前先列：
- 一阶效应：聚焦清晰 ✓
- 二阶（6 个月）：跨团队协作可能下降
- 三阶（18 个月）：探索性项目可能减少
→ 配套机制：保留 20% 时间给非 OKR 探索
→ 监控指标：跨团队协作频率（每月）
```

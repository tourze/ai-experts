# 诊断模式详细流程

## 模式 A：单次触发诊断

用户反馈"我说了 X 但没触发 Y skill"或"触发了错误的 skill"时使用。

### 诊断步骤

1. **还原现场**：获取用户的原始消息（或近似描述）
2. **提取意图关键词**：从消息中识别领域、动作、对象、约束
3. **扫描候选 skill**：读取所有 skill 的 description，列出与意图关键词有交集的 skill
4. **模拟匹配**：

   对每个候选 skill，判断：
   | 信号 | 权重 | 说明 |
   |------|------|------|
   | 关键词直接命中 | 高 | 用户消息包含 description 中的触发词 |
   | 语义近似 | 中 | 意图等价但措辞不同 |
   | 领域重叠 | 低 | 同一领域但不同任务 |

5. **输出诊断结论**：

   ```
   用户意图：[还原]
   期望触发：skill-X
   实际触发：skill-Y / 无
   根因分类：[见根因表]
   修复建议：[针对根因的具体动作]
   ```

### 常见根因

| 根因 | 表现 | 修复方向 |
|------|------|----------|
| 关键词缺失 | 用户用了 description 没覆盖的说法 | 在 description 补触发词或同义词 |
| 语义模糊 | 多个 skill 的 description 都能匹配 | 加排他指引（"如果只是 X，改用 `other-skill`"） |
| 触发条件过窄 | description 只列了一种场景 | 扩展"当...时使用"的场景枚举 |
| 触发条件过宽 | description 太泛导致误触发 | 收窄场景，加"不适用于..."限定 |
| 层级错位 | 该由框架层 skill 接但被语言层抢了 | 检查组件路由顺序和 description 区分度 |

## 模式 B：冲突检测

分析一组 skill 之间的触发域重叠。

### 步骤

1. **收集 description**：读取目标范围内所有 skill 的 name + description
2. **构建触发域矩阵**：
   - 显式触发词（description 中的关键词）
   - 隐式触发域（description 描述的场景可推导的用户消息类型）
   - 排他声明（"如果只是 X，改用..."）

3. **检测重叠对**：
   - **硬冲突**：相同关键词出现在两个 description 中，且无排他指引
   - **软冲突**：不同关键词但描述场景高度相似
   - **层级冲突**：一个 skill 的触发域完全包含另一个

4. **输出冲突报告**：
   ```
   冲突对：skill-A ↔ skill-B
   重叠触发域：[具体关键词/场景]
   冲突类型：硬冲突 / 软冲突 / 层级冲突
   当前区分机制：[有/无排他指引]
   建议：[具体修改哪个 skill 的 description]
   ```

## 模式 C：路由健康度评估

对一个能力域或整个仓库的 skill 路由做全面体检。

### 评估维度

| 维度 | 检查内容 | 健康标准 |
|------|----------|----------|
| 覆盖度 | 能力域声称的范围是否每个子场景都有 skill 接住 | 无明显盲区 |
| 区分度 | 任意两个 skill 的 description 是否能让模型一次选对 | 硬冲突对数 = 0 |
| 触发词密度 | 每个 description 是否包含足够的可匹配关键词 | 至少 3 个显式触发词 |
| 排他指引覆盖 | 有冲突风险的 skill 对是否配置了排他指引 | 所有软冲突对都有排他声明 |
| 语言一致性 | 同一能力域内 description 的语言/风格是否统一 | 全中文或全英文，句式一致 |

### 输出格式

```
## 路由健康度报告：[能力域/范围]

### 概览
- skill 总数：N
- 硬冲突对：N（列表）
- 软冲突对：N（列表）
- 触发词不足的 skill：N（列表）
- 覆盖盲区：[描述]

### 冲突详情
[按严重程度排序的冲突对列表]

### 建议修复优先级
1. [最高优先级修复]
2. ...
```

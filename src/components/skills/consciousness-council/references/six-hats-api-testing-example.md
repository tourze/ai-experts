# 六顶帽子示例：REST API 测试策略

**背景**：新建 REST API 25 个端点，3 周冲刺，需制定测试策略。
**团队**：2 后端 + 1 QE | **技术栈**：Node.js / Express / PostgreSQL / JWT

---

## 白帽 — 事实

- 25 端点（8 GET, 7 POST, 5 PUT, 3 DELETE, 2 PATCH），JWT + refresh token
- PostgreSQL 12 张表，OpenAPI 规范完成 80%，当前覆盖率 0%
- 峰值 100 req/s，p95 < 200ms，SLA 99.9%
- CI/CD 已配，可用 Jest / Supertest / Newman
- 已知风险：OAuth2 + JWT 复杂度、多租户隔离、WebSocket 实时更新

## 红帽 — 感受

- 有信心：CRUD、schema 设计
- 焦虑：认证边界、限流
- 担忧：WebSocket 测试（零经验）、多租户隔离
- 直觉："认证流程过于复杂，bug 藏在那里""正常路径没问题，边界情况会咬人"
- 整体信心 6/10

## 黑帽 — 风险

**高风险领域**：
1. **认证**：token 过期/轮换未测、密码重置有漏洞、无限流
2. **多租户**：租户 A 可能访问租户 B 数据、查询缺 tenant filter
3. **WebSocket**：连接稳定性、并发、消息顺序均未测
4. **性能**：无负载测试、N+1 查询风险、无缓存策略

**需挑战的假设**：
- "JWT 处理了安全" → 需测校验/过期/吊销
- "ORM 防注入" → 需验证输入清理
- "DB 约束保完整性" → 需应用层校验测试

## 黄帽 — 收益

- OpenAPI 规范可自动生成测试；Jest 经验丰富；CI/CD 已就绪
- 可复用：前项目认证测试工具、种子脚本、15 请求的 Postman 集合
- 快速收益：OpenAPI 生成测试（1 天）、自动化 Postman（4h）、复用认证测试（2h）
- 战略价值：首个全面测试的 API 成为团队模板

## 绿帽 — 创意

- **契约测试（Pact）**：前后端共享 API 契约，防破坏性变更
- **基于属性的测试（fast-check）**：随机输入发现人工遗漏的边界情况
- **混沌测试**：随机断连、注入延迟、损坏负载，测韧性
- **大胆想法**：feature flag 生产灰度（真实流量验证）、bug 赏金众包、录制回放生产流量、LLM 生成测试用例

## 蓝帽 — 行动计划

**目标**：80% 自动化覆盖、< 5% 生产缺陷、按时交付

**第 1 周（基础）**：认证测试套件 → 租户隔离 → 错误处理 → 集成测试
**第 2 周（扩展）**：属性测试 → 契约测试 → 负载基线 → 安全扫描
**第 3 周（打磨）**：WebSocket 框架 → 混沌实验 → 文档分享

**Go/No-Go（第 2 周末）**：
- P0 全通过 + 无严重安全问题 = 必须
- P1 完成 70% + 性能基线达标 = 应该
- P2 实验性测试 = 加分

**不做的事**：全面 WebSocket 测试（延后）、生产混沌（未准备好）、100% 覆盖（收益递减）、手工探索（没时间）

## 总结

六顶帽子 45 分钟产出：识别认证/隔离风险（红/黑帽），复用已有工具省 2 天（黄帽），引入属性测试（绿帽），阻止范围蔓延（蓝帽）。ROI 约 4 倍。

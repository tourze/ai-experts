# 六阶段详情

## 阶段 1：问题界定

调用 `mckinsey-7-step` 的问题定义步骤，或用 5W2H 补齐：Why now / What / Who / Where / When / How / How much。

过渡标准：任何同事读一遍都能说出"我们在回答什么、不回答什么"。
红旗：问题描述超过 3 句话还没说清 → 范围太大，先收敛。

## 阶段 2：结构化拆解

调用 `mckinsey-7-step` 的拆解步骤或 `structured-business-analysis-framework` 的 MECE 假设树。一级分支按问题类型选切口：市场/竞争/客户/产品/渠道/成本/技术/流程/外部。每个二级假设标注验证优先级（P0: >30% 解释力 / P1: 10-30% / P2: <10%）。

过渡标准：每个 P0 假设都有明确的验证方式。
红旗：假设树全是"可能是 X"但没有验证方法 → 停在拆解，先设计验证。

## 阶段 3：根因分析

调用 `fishbone-diagram` 做因果可视化，`first-principles-decomposer` 挑战隐含假设。每个根因候选标注证据强度（事实/推断/假设）、反证方式、影响面。

过渡标准：至少一个根因证据强度达"事实"级别，且排除了明显伪相关。
红旗：根因超过 5 个且无法排序 → 回到阶段 2；根因不可证伪 → 显式降级为假设。

## 阶段 4：系统动态识别

调用 `systems-thinking`：识别参与方、激励、反馈回路、延迟、杠杆点。涉及流程瓶颈时调用 `process-optimization`，涉及业务健康时调用 `business-health-diagnostic`。

过渡标准：识别出至少一个反馈回路，理解"修 A 会不会搞坏 B"。
红旗：只有单点分析没有回路 → 可能遗漏系统性风险。

## 阶段 5：决策推进

调用 `running-decision-processes`：明确决策人、时间窗、选项、评估标准。不确定性高时调用 `planning-under-uncertainty`：降级为选项 + 触发条件。

过渡标准：决策建议包含决策人、时间窗、回退策略。
红旗：分析了半天但没有人需要做任何决定 → 可能不是真问题。

## 阶段 6：PDCA 闭环

调用 `pdca-cycle`：Plan（量化目标+动作）→ Do（小范围试点）→ Check（对照基线）→ Act（标准化或回退）。标记下一轮检查点和负责人。

过渡标准：有明确检查点时间、负责人和兜底方案。
红旗：改进方案只写了"做什么"没写"怎么验证生效" → 不是闭环。

## 核心约束
- 原则是为了降低复杂度和反馈成本，不是为了凑口号。
- DRY 只消灭真正重复的知识，不要把相似但不同的逻辑强行合并。
- 示踪弹用于快速打通主路径，不代表可以跳过质量或收尾。
- 任何不可逆决策都要先验证边界，再决定是否承诺。

## 代码模式
- 估算、示踪弹、DRY、契约、破窗等主题可按需读取 `references/*.md`。
- 输出建议应落到“今天该怎么做”，而不是只复述原则定义。
- 优先写出原理如何改变本次决策，而不是平铺概念。


## 检查清单
- 是否识别了重复知识、耦合边界和不可逆决策。
- 是否选择了最小可验证路径，而不是一次赌大。
- 是否说明了原则对当前实现的具体影响。
- 是否保留了后续演进空间。

## 反模式

### FAIL: DRY 当过度抽象借口

```ts
// 三处看似相似但语义不同的循环
function processUsers() { ... }
function processOrders() { ... }
function processProducts() { ... }
// "DRY!" → 抽出 abstractEntityProcessor<T>(items, fn, opts, hooks)
→ 调用方传 5 个参数 / 看不出在干什么
```

### PASS: 只 DRY 真重复

```
DRY 知识，不 DRY 代码。
3 处循环虽然结构像，但语义不同（user 校验 vs 订单状态 vs 库存）
→ 保持独立函数，未来分别演进
当且仅当三处真的表达同一个业务规则时才抽
```

### FAIL: 示踪弹 = 临时代码堆

```
"先做示踪弹打通主路径" → 写 500 行 try-catch + console.log + // TODO
"以后再清理"
→ 上线了，没人敢清
```

### PASS: 可演进的最小路径

```
示踪弹 = 端到端走一遍，但每步质量是"可演进"的：
- 函数签名稳定（即使实现简单）
- 错误返回结构稳定（不输出原始 error）
- 关键日志保留（便于后续观察）
→ 后续扩展不需要推倒
```

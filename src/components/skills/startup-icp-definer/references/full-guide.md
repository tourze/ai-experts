# Startup ICP Definer — 完整指南

## 框架：ICP Stack

```
┌─────────────────────────────────────────┐
│         购买中心（BUYING CENTER）         │
│     （决策中的所有人员）                    │
├─────────────────────────────────────────┤
│         用户画像（USER PERSONA）           │
│     （日常使用者）                         │
├─────────────────────────────────────────┤
│         买家画像（BUYER PERSONA）          │
│     （签字付款的人）                       │
├─────────────────────────────────────────┤
│         公司画像（COMPANY PROFILE）        │
│     （组织）                              │
└─────────────────────────────────────────┘
```

从下往上构建。先定义公司，然后定义买家，再定义用户，最后定义完整的购买中心。

## 详细执行工作流

### 步骤 1：公司画像定义

问用户：

> **你的理想客户是什么样的公司？**
>
> 1. 属于什么行业或垂直领域？
> 2. 公司规模多大？（员工数或营收）
> 3. 公司处于什么阶段？（初创、成长、企业）
> 4. 在哪些地区？
> 5. 他们使用什么技术栈或工具？
> 6. 什么商业模式？（B2B、B2C、marketplace）

**公司画像模板：**

| 属性 | 定义 | 理由 |
|------|------|------|
| **行业** | [具体垂直领域] | [为什么是这个行业] |
| **规模** | [X-Y 员工 或 $X-$Y 营收] | [为什么是这个规模] |
| **阶段** | [初创/成长/企业] | [为什么是这个阶段] |
| **地区** | [区域/国家] | [为什么是这个地区] |
| **技术栈** | [他们使用的关键工具] | [为什么这很重要] |
| **商业模式** | [B2B/B2C/等] | [为什么是这个模式] |

**示例：**

```
行业：B2B SaaS 公司
规模：50-500 名员工
阶段：A 轮到 C 轮
地区：美国、英国、西欧
技术栈：Salesforce、HubSpot、Slack
商业模式：经常性收入软件
```

### 步骤 2：问题细分

**并非你画像中的所有公司都有相同的痛点强度。**

问用户：

> **什么情况会促使公司寻找你的解决方案？**
>
> 1. 什么事件会创造紧迫性？（增长、痛点、强制要求）
> 2. 哪些症状表明他们需要你？
> 3. 什么让一些公司比其他公司更适合？
> 4. 什么情况下他们不适合？

**触发事件：**

| 触发事件 | 含义 | 紧迫程度 |
|----------|------|----------|
| [触发事件 1] | [情况] | [高/中/低] |
| [触发事件 2] | [情况] | [高/中/低] |
| [触发事件 3] | [情况] | [高/中/低] |

**常见 B2B 触发事件：**

- 员工快速增长
- 新高管入职
- 竞争对手产品失败
- 合规/监管要求
- 大客户需求
- 融资事件（刚融资）
- 手工流程崩溃
- 与新的工具集成

### 步骤 3：买家画像定义

**买家 = 签字付款的人。**

> 4. 他们的痛点是什么？
> 5. 他们的决策权是什么？
> 6. 他们最关心什么？（成本、时间、风险、结果）

### 步骤 6：痛点层级

**并非所有痛点都是同等的。**

问用户：

> **对客户的痛点进行排序：**
>
> 1. 哪个痛点最紧急？
> 2. 哪个痛点花费他们最多钱？
> 3. 哪个痛点最频繁？
> 4. 哪个痛点他们正在积极解决？

**痛点层级框架：**

| 痛点 | 紧急度 | 成本 | 频率 | 正在积极解决？ | 分数 |
|------|--------|------|------|----------------|------|
| [痛点 1] | [高/中/低] | [高/中/低] | [高/中/低] | [是/否] | [合计] |
| [痛点 2] | [高/中/低] | [高/中/低] | [高/中/低] | [是/否] | [合计] |
| [痛点 3] | [高/中/低] | [高/中/低] | [高/中/低] | [是/否] | [合计] |

**评分：** 高=3、中=2、低=1、是=+2、否=0
**优先关注得分 8 分以上的痛点。**

### 步骤 7：ICP 验证访谈指南

**通过真实对话验证你的 ICP。**

问用户：

> **你将如何验证这个 ICP？**
>
> 1. 你能接触到符合此画像的人吗？
> 2. 你是否已与 10 位以上此细分领域的人交谈过？
> 3. 你会问什么问题来验证？

**面试问题库：**

| 类别 | 问题 | 你能学到什么 |
|------|------|--------------|
| **问题** | "请告诉我你目前如何处理[X]" | 当前流程 |
| **痛点** | "其中最有挫败感的部分是什么？" | 痛点强度 |
| **影响** | "这让你付出了什么代价？（时间/金钱/压力）" | 痛点量级 |
| **方案** | "你尝试过什么来解决这个问题？" | 竞争对手 |
| **预算** | "你是否有为此分配的预算？" | 购买意向 |
| **时间线** | "你需要在什么时候之前解决这个问题？" | 紧迫性 |
| **决策** | "你通常如何评估这样的工具？" | 购买流程 |

### 步骤 8：ICP 收窄标准

**从窄开始，再扩展。**

问用户：

> **哪些约束能帮助你聚焦？**
>
> 1. 什么让公司成为完美的匹配？
> 2. 什么让公司成为好的匹配？
> 3. 什么让公司成为差的匹配？
> 4. 什么是一票否决条件？

**匹配度评分框架：**

| 标准 | 完美匹配 | 好匹配 | 差匹配 | 否决条件 |
|------|----------|--------|--------|----------|
| 公司规模 | [确切] | [范围] | [以外] | [绝不] |
| 行业 | [确切] | [相邻] | [远] | [绝不] |
| 技术栈 | [使用 X] | [使用 Y] | [无 Y] | [使用 Z] |
| 预算 | [有预算] | [能找到] | [无预算] | [绝不付费] |
| 痛点 | [严重] | [中等] | [轻微] | [无] |
| 时间线 | [现在] | [3 个月] | [6 个月] | [无时间线] |

## 常见错误避免

1. **过于宽泛：** "所有公司"不是一个 ICP
2. **过于狭窄：** "爱荷华州的左撇子 CEO"范围太小
3. **买家/用户混淆：** 他们通常是不同的人
4. **忽略购买中心：** 多人起作用
5. **假设而非验证：** 与真实客户交谈
6. **静态 ICP：** 它应随着学习而演进
7. **功能导向的画像：** 按问题定义，不是按功能
8. **跳过反画像：** 知道不要瞄准谁

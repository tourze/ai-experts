## 流程

### 第 1 步：验证测试

```bash
# 运行项目测试套件
npm test / cargo test / pytest / go test ./...
```

**测试失败：**
```
测试未通过（N 个失败）。必须修复后才能继续：

[展示失败列表]

无法在测试失败的情况下合并/创建 PR。
```
→ 停在这里。不进入第 2 步。

**测试通过：** 继续第 2 步。

### 第 2 步：确定基线分支

```bash
git merge-base HEAD main 2>/dev/null || git merge-base HEAD master 2>/dev/null
```

### 第 3 步：展示选项

展示以下 4 个选项：

```
实现完成。你想怎么处理？

1. 合并回 <基线分支>（本地合并）
2. 推送并创建 Pull Request
3. 保留分支原样（我稍后处理）
4. 丢弃这些工作
```

**不加额外解释**——保持选项简洁。

### 第 4 步：执行选择

#### 选项 1：本地合并

```bash
git checkout <基线分支>
git pull
git merge <feature 分支>
# 验证合并后测试
<test command>
# 测试通过则删除 feature 分支
git branch -d <feature 分支>
```

#### 选项 2：推送并创建 PR

```bash
git push -u origin <feature 分支>
gh pr create --title "<标题>" --body "<摘要>"
```

#### 选项 3：保留原样

报告："保留分支 `<name>`。Worktree 保留在 `<path>`。"
不做清理。

#### 选项 4：丢弃

**必须二次确认：**
```
这将永久删除：
- 分支 <name>
- 所有提交：<提交列表>
- Worktree（如有）

请输入 'discard' 或 '确认丢弃' 来确认。
```

等待明确确认。未确认不执行。

### 第 5 步：清理 Worktree

对选项 1、2、4：检查是否在 worktree 中，如果是则清理。
对选项 3：保留 worktree。

## 快速参考

| 选项 | 合并 | 推送 | 保留 Worktree | 删除分支 |
|------|------|------|-------------|---------|
| 1. 本地合并 | ✓ | - | - | ✓ |
| 2. 创建 PR | - | ✓ | ✓ | - |
| 3. 保留原样 | - | - | ✓ | - |
| 4. 丢弃 | - | - | - | ✓（强制） |

## Red Flags — 出现以下念头时立即停下

| 念头 | 现实 |
|------|------|
| "测试基本能过，先合并再说" | 没有"基本能过"。全过或不过。 |
| "帮用户选最合适的选项" | 不替用户决定。展示选项，等待选择。 |
| "丢弃这些明显是临时的" | 丢弃必须用户确认。可能有未提交的想法。 |
| "自动清理 worktree 比较整洁" | 选项 2 和 3 需要保留 worktree。 |

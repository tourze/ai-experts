
# STRIDE 威胁分析

## 适用场景
- 需要对系统、模块或数据流做结构化威胁枚举。
- 需要给 [security-threat-model](../SKILL.md) 提供覆盖式分析骨架。
- 需要用 [attack-tree-construction](./attack-tree-construction.md) 继续展开重点路径。

## 核心约束
- 按组件、边界或数据流逐一过 STRIDE，不要整系统一锅端。
- 每个 STRIDE 标签都要落到具体滥用场景和资产影响。
- 把现有控制和建议控制分开写。
- 不要用 STRIDE 标签替代真实风险描述。

## 代码模式
```markdown
| 边界 | STRIDE | 场景 | 影响资产 | 现有控制 |
| --- | --- | --- | --- | --- |
| API -> DB | T | 通过未校验字段篡改账单状态 | 账单完整性 | ORM 参数化 |
| Browser -> API | S | 伪造管理员令牌调用导出接口 | 管理员权限 | JWT 校验 |
```

## 检查清单
- 每个边界是否至少过一遍六类威胁。
- 是否写明攻击动作、前置条件和影响资产。
- 是否标出已有控制和缺口。
- 是否避免重复堆叠相同威胁。

## 反模式

### FAIL: 只有标签

```markdown
- S: 用户欺骗
- T: 数据篡改
- R: 抵赖
- I: 信息泄露
- D: 拒绝服务
- E: 提权
→ 6 个标签贴上去，没有任何具体动作和资产
```

### PASS: 边界 × 标签 → 具体场景

```markdown
| 边界 | 类别 | 攻击动作 | 影响资产 | 现有控制 |
|---|---|---|---|---|
| Browser→API | S | 伪造其他用户 JWT 调用 /transfer | 账户余额 | JWT 签名校验 ✓ |
| API→DB | T | 通过未校验的 status 字段把订单改成 paid | 订单完整性 | ORM 参数化 ✗ 字段未白名单 |
| Worker→S3 | I | 跨租户读取导出文件 | 用户 PII | 桶策略按 tenant_id 前缀 ✓ |
→ 每行可独立讨论是否需要补充控制
```

### FAIL: STRIDE 与漏洞名混写

```
- T (XSS)
- I (SQL 注入)
- E (CSRF)
→ XSS 既可能是 T 也可能是 I，混着写无法系统枚举
```

### PASS: 先分类再举漏洞

```
类别 T（篡改）：
  - 场景 1：通过 XSS 改前端展示
  - 场景 2：通过 mass assignment 改 is_admin
类别 I（信息泄露）：
  - 场景 1：通过 SQL 注入读 users 表
→ 同一漏洞按其攻击意图归类
```

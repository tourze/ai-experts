# 调试六步流程详解

## 第一步：复现

目标：稳定复现问题，明确"正常行为"和"实际行为"的差异。

- 确认复现步骤（输入、环境、操作序列）。
- 确认"每次都能复现"还是"偶发"——偶发问题额外记录频率和条件。
- 明确期望行为和实际行为。
- 无法复现时，先增加可观测性（日志、断点、trace），不猜。

```
复现记录：
- 环境：[OS / 运行时 / 版本]
- 输入：[具体输入数据或操作步骤]
- 期望行为：[应该发生什么]
- 实际行为：[实际发生了什么]
- 复现率：[每次 / N次中M次 / 仅一次]
```

## 第二步：隔离

目标：从"整个系统有问题"收敛到"这个模块/函数/行有问题"。

- 二分法缩小范围：注释掉一半代码看问题是否消失。
- 检查最近变更：`git log --oneline -20`、`git diff`、`git bisect`。
- 移除外部依赖：用硬编码替代网络/数据库/文件调用。
- 构造最小复现用例。

## 第三步：假设

目标：基于证据形成 1-3 个可证伪假设。

- "可能某处有问题"不是假设——假设必须可证伪。
- 格式："因为 [原因]，在 [条件] 下，[变量] 的值是 [错误值] 而不是 [正确值]"。
- 按可能性排序，先验证最可能的。

```
假设 1（最可能）：[描述]
- 验证方法：[具体操作]
- 预期结果：如果正确，应该看到 [X]

假设 2：[描述]
- 验证方法：[具体操作]
- 预期结果：如果正确，应该看到 [X]
```

## 第四步：验证

目标：用实验证实或证伪假设。

- 每次只测一个假设。
- 记录过程和结果，不论成败。
- 假设被证伪则回到第三步，不硬凑解释。
- 使用调试工具（断点、日志、trace）获取直接证据。

## 第五步：修复

目标：针对根因实施修复。

- 修复前先解释根因："问题是因为 X 导致 Y"。
- 修复应是最小的针对性变更。
- 临时补丁必须标注 `// TODO: 临时修复，根因是 ...`。
- 修复后重新运行第一步的复现步骤确认问题消失。

## 第六步：回归测试

目标：确保修复不引入新问题，防止复发。

- 为这个 bug 写一个测试用例覆盖触发条件。
- 运行相关现有测试确认无破坏。
- 如果 bug 属于某个类别（off-by-one、空值处理），检查同类代码是否有相同问题。

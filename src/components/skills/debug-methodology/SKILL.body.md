## 六步流程

1. **复现**：稳定复现问题，明确期望 vs 实际行为。
2. **隔离**：二分法缩小范围到具体模块/函数/行。
3. **假设**：基于证据形成 1-3 个可证伪假设，按可能性排序。
4. **验证**：每次只测一个假设，记录结果。
5. **修复**：针对根因最小化修复，临时补丁必须标注。
6. **回归测试**：补测试覆盖触发条件，跑现有测试确认无破坏。

完整步骤见 Reference Map 中的 `six-steps`，纪律守卫见 `discipline-guard`，流程图见 `debug-flow`。

## 推荐流程

1. 先用一句话写下用户可观察到的问题。
2. 用 `debug-checklist` 生成检查清单骨架。
3. 在清单里填入复现命令、环境、期望、实际。
4. 每轮只验证一个假设，保留命令和输出摘要。
5. 修复后补回归测试，再总结根因。

## 检查清单

- [ ] 已稳定复现，或记录了无法复现的条件。
- [ ] 已缩小到具体模块/函数/行。
- [ ] 假设基于证据，每次只变更一个变量。
- [ ] 修复针对根因，不是症状。
- [ ] 已补回归测试。

## 纪律守卫

**Iron Law：没有根因调查，不允许提出修复方案。**

出现以下念头时立即停下：

- "我知道问题在哪，先改改试试。"
- "这个很简单，不用系统化。"
- "紧急，没时间走流程。"
- "先加个 try-catch 顶一下。"
- "我猜来源是 X，先写脚本删 X 看看。"

这些都是跳过证据链的信号。先验证，再修改。

# 架构决策记录与系统边界分析

## 适用场景

- 需要把架构决策写成 ADR（Architecture Decision Record）格式
- 系统边界模糊：服务该拆还是该合、数据归谁、一致性怎么保证
- 接口需要版本策略和 breaking change 治理流程
- 方案评审时需要显式写 trade-off，不用"最佳实践"搪塞
- 弹性策略设计：超时、重试、熔断、降级的边界和组合

## ADR 格式

每个架构决策按三段式记录：

```
Context（背景）：当前面临什么问题、有哪些约束、有哪些选项
Decision（决策）：选择了什么方案、为什么
Consequences（后果）：正面影响、负面影响、假设不成立时的降级路径
```

不写"为什么不选其他方案"的 ADR 是不完整的——隐含选项比显式选项更容易被后来者推翻。

## 系统边界分析

### 服务划分判断矩阵

| 信号 | 倾向 |
|------|------|
| 独立的数据所有权（没有共享表/共享事务） | 可拆 |
| 独立的变更节奏（不同团队、不同发布周期） | 可拆 |
| 独立的弹性需求（不同 SLA、不同扩容策略） | 可拆 |
| 强一致性跨服务事务 | 先不合，或引入 Saga |
| 高频跨服务查询（join 在应用层做代价高） | 先不合，或引入 CQRS |

### 数据所有权原则

- 每份数据只有一个写入 owner；其他服务通过 API 或事件获取。
- 跨服务读可以冗余（read model / materialized view），但不能改写 owner 职责。
- 缓存不是所有权转移——缓存 miss 时仍然回源 owner。

## 接口契约设计

### 版本策略选择

| 场景 | 推荐策略 |
|------|---------|
| 内部服务、调用方可控 | 不版本化，直接改 + 同步升级所有调用方 |
| 外部 API、调用方不可控 | URL 版本化（/v1/ /v2/）或 header 版本化 |
| 移动端 API、升级周期长 | 兼容过渡期：新字段加默认值，旧字段标记 deprecated |

### Breaking Change 流程

1. 先发布新接口（或新字段），保持旧接口 100% 可用。
2. 在 N 个周期内监控旧接口调用量，主动通知调用方迁移。
3. 调用量降至阈值以下 → 标记 deprecated → 再一个周期后移除。
4. 禁止：直接改返回值类型、删除字段、改错误码含义。

完整契约模板与反模式见 [references/contract-templates.md](references/contract-templates.md)。

## 弹性策略

### 四件套组合规则

| 策略 | 用途 | 典型配置 | 注意事项 |
|------|------|---------|---------|
| 超时 | 防止无限等待 | P99 × 1.5，上限 30s | 读超时和写超时分开设 |
| 重试 | 处理瞬时故障 | 最多 3 次，指数退避 + jitter | 幂等是重试的前提 |
| 熔断 | 防止级联故障 | 5 次失败 → 开 → 30s 后半开 → 1 次成功恢复 | 按下游实例熔断，不是按服务 |
| 降级 | 保核心路径 | 返回缓存/默认值/精简响应 | 降级行为必须预定义，不能运行时现编 |

### 复杂度管理

深模块（deep module）：接口简单、实现复杂。衡量标准：接口的"使用成本" vs 实现的"隐藏信息量"。

- 好设计：接口暴露 2-3 个概念，内部处理 20+ 种边界情况。
- 坏设计：接口暴露 10+ 个参数和配置项，内部只是一层薄转发。

## 检查清单

- [ ] 每个决策有显式的 Context → Decision → Consequences。
- [ ] 抛弃的选项和理由已记录，不在 ADR 中留"唯一方案"。
- [ ] 服务边界分析覆盖了数据所有权、变更节奏和弹性需求三个维度。
- [ ] 对外 API 有明确的版本策略和 breaking change 流程。
- [ ] 超时/重试/熔断/降级有具体数值，不是"设一个合理的值"。
- [ ] 关键假设标注了不成立时的降级路径。

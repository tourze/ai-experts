# 规模估算启发式

用于快速估算的规模模式、复杂度指标和参考类数据。

---

## 快速规模估算规则

### "集成点数量"启发式

统计外部集成点的数量：

| 集成点数            | 典型规模 |
| ------------------- | -------- |
| 0（纯逻辑）         | S-M      |
| 1（一个 API、一个 DB） | M        |
| 2-3                 | M-L      |
| 4+                  | L-XL     |

每个集成增加：

- 错误处理
- 重试逻辑
- 配置
- 使用 mock 进行测试

### "决策数量"启发式

统计所需的设计决策数量：

| 决策数                     | 典型规模 |
| -------------------------- | -------- |
| 0（遵循现有模式）          | S        |
| 1-2（小选择）              | M        |
| 3-5（架构选择）            | L        |
| 5+（许多未知）             | XL       |

### "文件数量"启发式

统计需要变更的文件数量：

| 文件数   | 典型规模                           |
| -------- | ---------------------------------- |
| 1-2      | S                                  |
| 3-5      | M                                  |
| 5-10     | L                                  |
| 10+      | XL（或应拆分为多个任务）            |

| 文件上传                      | M (8h)         | 验证、存储、清理                                 |
| 邮件/通知                     | M (8h)         | 模板、供应商集成、队列                           |
| 搜索功能                      | L (20h)        | 索引、查询解析、排序                             |

| 速率限制                      | M (6h)         | 存储、中间件、头部                               |

## 工作量乘数

将这些应用于基础估算：

| 因素                           | 乘数       | 时机                                    |
| ------------------------------ | ---------- | --------------------------------------- |
| 首次使用该技术                 | 2.0x       | 从未使用过该库/框架                     |
| 第二次使用                     | 1.3x       | 之前用过一次                            |
| 第三次及以上                   | 1.0x       | 熟悉的模式                              |
| 安全敏感                       | 1.5x       | 认证、加密、PII 处理                    |
| 跨团队协调                     | 1.3x       | 需要其他团队的输入/批准                  |
| 遗留代码库                     | 1.3-2.0x   | 文档差、无测试                          |
| 高测试覆盖率要求               | 1.3x       | 90%+ 覆盖目标                           |
| 无障碍要求                     | 1.2x       | WCAG 合规                               |

## "应该估算还是做 Spike？"

| 条件                                  | 行动                                |
| ------------------------------------- | ----------------------------------- |
| 少于 2 个未知因素，熟悉的栈            | 直接估算                            |
| 2-3 个未知因素，部分熟悉               | 用宽范围估算（3 倍跨度）            |
| 4+ 个未知因素，新技术                  | 先做 spike（1-2 天），再估算        |
| "我完全不知道"                        | 一定要先做 spike                    |

Spike 是限时调查，以减少不确定性。其输出是信息（以及更窄的估算范围），而不是代码。

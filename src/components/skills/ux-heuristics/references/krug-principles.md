# Krug 可用性原则

来自《Don't Make Me Think》的完整方法论，用于创建直观、可用的界面。

## 用户如何使用网站的现实

### 事实 1：我们不阅读，我们扫描

**用户为什么要扫描：**
- 他们有任务在身（有目标）
- 他们知道自己不需要阅读所有内容
- 他们从多年实践中擅长扫描

**设计影响：**
- 创建视觉层次（重要的内容更大、更粗）
- 大量使用标题和副标题
- 保持段落简短
- 使用项目符号列表
- 突出关键术语

### 事实 2：我们不做出最优选择，我们满意即可

**满意即可（Satisficing）** = "satisfying（满意）" + "sufficing（足够）"

用户点击第一个合理的选项，而非最佳选项。

**为什么满意即可：**
- 优化很困难且耗时
- 猜错的代价很小（返回按钮）
- 权衡各个选项对结果改善不大
- 猜测更有趣

**设计影响：**
- 让正确选项一目了然
- 不要指望用户阅读所有选项
- 在点击前让后果清晰可见

### 事实 3：我们不计算工作原理，我们凑合使用

用户不阅读说明。他们尝试各种操作，直到某个操作生效。

**为什么凑合使用：**
- 他们并不在意了解工作原理
- 如果找到能用的方法，他们会坚持下去
- 即使没有最优地使用，通常也没关系

**设计影响：**
- 让一切显而易见，"计算"不再需要
- 预期用户会"错误地"使用
- 为从错误中恢复而设计

## 你永远不该做的事

### 不要让用户思考

表明页面让用户思考的迹象：

| 症状 | 示例 |
|---------|---------|
| 令人困惑的标签 | "解决方案"（解决什么？） |
| 可能指向任何地方的链接 | "点击这里" |
| 未经解释的选项 | 没有上下文的复选框 |
| 不熟悉的术语 | 行业术语 |
| 不必要的俏皮话 | 导航中的双关语、文字游戏 |

### 不要浪费用户的好感

用户的好感是有限的：

**削弱好感的行为：**
- 隐藏他们需要的信息（电话号码、价格）
- 因未按你的方式操作而惩罚用户
- 询问不必要的信息
- 让用户感到愚蠢
- 让用户重复输入
- 看起来像是事后才做的网站
- 业余错误（断链、拼写错误）

**增强好感的行为：**
- 知道他们有什么问题并回答它们
- 最小化步骤数量
- 在 UI 质量上投入精力
- 了解他们可能在哪些方面遇到困难
- 让他们容易从错误中恢复
- 当出问题时道歉

### 不要让不重要的词显得重要

**高兴话示例：**
> "欢迎访问我们的网站！我们很高兴帮助您准确找到您想要的内容。我们由专业人员组成的团队致力于为您提供最佳体验。"

**现实：** 用户跳过这段内容。它什么也没说。

**规则：** 如果用户会跳过它，就删除它。

## 导航必备要素

### 永久导航

每个页面都需要：

1. **站点 ID**（标志/名称） - 左上角
2. **页面名称** - 突出显示，与引导链接匹配
3. **栏目** - 网站主要区域
4. **本地导航** - 当前栏目中的内容
5. **实用工具** - 登录、搜索、帮助、购物车
6. **"你在这里"指示器** - 高亮的导航项

### 主干测试

用户能否在任何随机页面上回答这些问题？

| 问题 | 回答该问题的元素 |
|----------|------------------------|
| 这是什么网站？ | 标志/站点 ID |
| 我在哪个页面？ | 页面标题 |
| 有哪些主要栏目？ | 主导航 |
| 我在这里有哪些选项？ | 本地导航 |
| 我在结构中的什么位置？ | 面包屑、高亮导航 |
| 如何搜索？ | 搜索框 |

### 面包屑导航

**好的面包屑：**
```
首页 > 产品 > 鞋类 > 跑步鞋 > 男士越野跑鞋
```

**规则：**
- 放在顶部
- 在层级之间使用">"
- 使当前页面名称可见但不可点击
- 使用小号字体（次要重要性）

## 首页指南

首页需要做太多事情：

1. 站点标识和使命
2. 站点层级结构（导航）
3. 搜索
4. 预告（内容、功能）
5. 及时内容
6. 促销
7. 快捷方式（热门项）
8. 注册/登录

### 首页优先级

**必须做：**
- 告诉我这个网站是什么
- 告诉我可以在这里做什么
- 告诉我为什么应该在这里（而不是别处）
- 让我开始行动

**应该做：**
- 向我展示我在寻找的内容
- 告诉我从哪里开始
- 建立可信度和信任

### 网页设计的大爆炸理论

**你有 3-4 秒时间来回答：**
1. 这是什么？
2. 我可以在这里做什么？
3. 我为什么应该在这里？

### 标语指南

| 好的标语 | 差的标语 |
|--------------|-------------|
| 传达独特价值 | 通用套话 |
| 具体且信息丰富 | 模糊且空洞 |
| 6-8 个词 | 太长或太短 |
| 立即可理解 | 需要思考 |

示例：
- 好："从数千家商店找到任何商品"
- 差："欢迎来到购物的未来"

## 移动端可用性

### 移动端考量

**限制条件：**
- 更小的视口
- 粗手指（需要更大的触摸目标）
- 单列布局
- 无悬停状态
- 注意力和使用场景多变

### 移动端具体建议

| 问题 | 解决方案 |
|-------|----------|
| 触摸目标太小 | 最小 44×44 像素 |
| 导航拥挤 | 汉堡菜单或底部导航 |
| 长表单 | 分成多个步骤 |
| 依赖悬停的 UI | 触摸替代方案 |
| 文本太小 | 正文最小 16 像素 |

### 移动端权衡

**优先考虑：**
- 主要任务放在核心位置
- 必要内容可见
- 快速加载
- 易于点击，难以误触

**隐藏/移除：**
- 次要导航
- 非必要图片
- 装饰性元素
- 长篇幅内容

## 零成本可用性测试

### 需要多少用户？

**3-4 个用户能发现大部分问题。**

与更多用户进行测试的收益递减。最好先用 3 个用户测试，修复问题，再用 3 个用户重新测试。

### 测试什么

1. 他们能否完成核心任务？
2. 他们在哪里卡住？
3. 他们大声说了什么？
4. 他们的预期 vs 实际发生了什么？

### 测试流程

**测试前：**
- "我在测试这个网站，不是测试你"
- "请在操作过程中大声说出想法"
- "没有错误答案"

**测试中：**
- 不要帮忙。不要解释。只观察。
- 记录犹豫和困惑
- 写下他们说的话

**测试后：**
- 问什么让人困惑
- 问他们预期的是什么
- 问他们如何描述这个网站

### 常见发现

| 用户的反应 | 这意味着什么 |
|---------------|---------------|
| 点错了东西 | 标签令人困惑 |
| 犹豫 | 决策不够明确 |
| 茫然四顾 | "你在这里"指示不清 |
| 阅读所有内容 | 设计不够一目了然 |
| 立即使用搜索 | 导航功能不足 |
| 表达困惑 | 文案不够清晰 |

## 无障碍基础知识

### 为什么重要

- 15-20% 的人口有某种残疾
- 无障碍网站对所有人都更好
- 通常法律要求
- 这是正确的事情

### 速效改进

| 修复措施 | 好处 |
|-----|---------|
| 为图片添加 alt 文本 | 屏幕阅读器可描述 |
| 使用足够的对比度 | 低视力用户可阅读 |
| 允许键盘导航 | 运动障碍用户可操作 |
| 使用语义化 HTML | 辅助技术理解结构 |
| 添加焦点指示器 | 键盘用户知道位置 |
| 使触摸目标足够大 | 每个人受益 |

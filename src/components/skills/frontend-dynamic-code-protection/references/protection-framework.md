## 来源材料

- 核心启发来自知乎文章《前端动态化代码保护方案探索》：https://zhuanlan.zhihu.com/p/37738569
- 本 skill 只沉淀防御性工程原则，不提供绕过风控、批量刷量或自动化攻击实现。

## 核心判断

- 前端代码保护的目标是提高机器自动化成本，不是把秘密永久藏在浏览器里。
- 传输层 TLS 不等于业务参数保护；参数语义清晰时，攻击者可直接构造请求。
- 代码压缩不是混淆；普通混淆只能延缓逆向，不能单独承担长期防护。
- 固定可逆变换函数、固定密钥、固定参数和固定签名算法只要完整下发到浏览器，最终都应视为可被提取。
- 服务端必须保留最终判定权；不要把权限、额度、价格、库存、身份等信任决策放到前端保护逻辑里。

## 工作流

1. 明确对抗目标：
   - 资产：账号、券、票数、库存、价格、内容、接口额度。
   - 攻击者：脚本请求、Headless 浏览器、模拟器、人工打码、代理池。
   - 成本目标：阻断、延迟、提高资源消耗，还是为风控提供信号。
2. 审计现状：
   - 找到参数生成、签名、加密、challenge、混淆、缓存和服务端验签入口。
   - 区分请求伪造、脚本自动化、浏览器自动化和重放攻击。
   - 记录哪些逻辑在浏览器可见，哪些状态只在服务端可见。
3. 设计动态化：
   - 将保护逻辑从主业务 bundle 中拆出，独立构建、独立缓存策略、独立灰度。
   - 准备多个变换函数或 challenge 片段，每次请求随机组合，并让服务端能从签名解析组合元信息。
   - 签名包含组合标识、随机数、时间戳、版本和必要上下文；不要明文暴露函数编号或分隔符。
   - 对 signature 设置短有效期、单次或低次数使用限制，并记录重放、过期和异常组合命中。
4. 处理工程约束：
   - 动态 JS 不走长期缓存；使用 inline 片段或独立资源，并设置 `Cache-Control: no-cache` 或更严格策略。
   - 编译较慢时使用预编译产物池，由服务端随机返回，并在响应时注入一次性 signature。
   - 混淆仍然需要，但只作为动态化的特征隐藏层，不作为唯一控制。
   - 保护代码与主业务通信只传递必要结果，避免把服务端密钥、规则阈值或完整风控策略下发。
5. 验证与监控：
   - 用抓包证明明文参数语义已被隐藏或绑定到 signature。
   - 用重放测试验证过期、重复 signature 被拒绝。
   - 用缓存测试验证动态资源不会被长期缓存导致验签失败。
   - 用基准测试验证首屏、challenge 加载和低端设备执行时间。
   - 用日志指标跟踪过期率、重放率、组合分布、验签失败原因和业务转化损耗。

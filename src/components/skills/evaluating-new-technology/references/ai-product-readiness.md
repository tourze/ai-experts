
# AI 产品适用性评估

## 适用场景
- 决定是否在产品中引入 AI/LLM 功能，需要系统化评估适用性和投入产出比。
- 已有 AI 功能但效果不理想，需要诊断问题出在哪里。
- 与 [evaluating-new-technology](../SKILL.md) 配合做 build vs buy 决策。
- 与 [create-prd](../../create-prd/SKILL.md) 配合撰写 AI 功能的 PRD。

## 核心约束
- AI 不是功能，是能力；先问"用户要解决什么问题"，再问"AI 是不是最好的方案"。
- 必须区分"AI 能做到"和"AI 能可靠地做到"；demo 效果 ≠ 生产级效果。
- 评估必须覆盖 5 个维度：问题适配、数据就绪、技术可行、团队能力、商业回报。
- AI 功能的失败模式必须提前设计（输出错误时怎么办），不能只考虑 happy path。

## 代码模式

### 5 维适用性评估

```markdown
| 维度 | 关键问题 | 得分 1-5 | 证据 |
|---|---|---|---|
| 问题适配 | 人工做这件事的成本/质量是否是真实痛点？容错度多高？ | | |
| 数据就绪 | 是否有足够质量和规模的数据？标注成本可承受？ | | |
| 技术可行 | 当前模型能力是否足够？延迟和成本是否可接受？ | | |
| 团队能力 | 团队是否有 AI 工程和评估经验？还是纯依赖外部？ | | |
| 商业回报 | AI 带来的增量价值能否覆盖持续投入？用户愿意为此付费？ | | |
```

### AI 功能失败模式设计

```markdown
| 失败类型 | 用户看到什么 | 兜底方案 |
|---|---|---|
| 输出质量低 | 结果明显错误 | 置信度低时显示警告+人工入口 |
| 延迟过高 | 等待超过预期 | 流式输出+进度提示+缓存 |
| 模型不可用 | 功能无响应 | 降级到规则引擎或人工流程 |
| 输出有害 | 不当内容 | 内容过滤+举报机制 |
```

### Build vs API vs Fine-tune 决策

```markdown
通用能力 + 低定制 → 调用第三方 API（GPT/Claude）
通用能力 + 中定制 → API + Prompt Engineering + RAG
垂直能力 + 高定制 → Fine-tune 或自训练
核心壁垒 → 自建模型 + 私有数据
```

## 检查清单
- [ ] 5 个维度都做了评估，每个维度有具体证据。
- [ ] 已设计 AI 功能的失败模式和兜底方案。
- [ ] 明确了 Build/API/Fine-tune 的技术路线选择和理由。
- [ ] 评估结论能回答"值不值得做、现在做还是等一等"。

## 反模式

### FAIL: AI for AI's sake

```
"竞品都有 AI 功能了，我们也加一个"
→ 没验证用户是否需要
→ 上线后使用率 2%，但 API 费用 $5K/月
```

### PASS: 问题驱动的 AI 引入

```
用户痛点：手动分类 500 张发票/天，错误率 15%
AI 方案：发票自动分类，准确率 95%+
ROI：节省 2 人天/天，错误率降至 3%
兜底：置信度 <80% 转人工复核
→ 明确的问题、可量化的收益、有兜底
```

### FAIL: 只看 demo 效果

```
"我用 ChatGPT 试了一下，效果不错，上线吧"
→ 没测边界 case、没测大规模、没测延迟
→ 上线后：20% case 输出垃圾，P99 延迟 15s
```

### PASS: 系统化验证

```
1. 收集 200 条真实用例（含 50 条 edge case）
2. 建立评估标准（正确/部分正确/错误）
3. 跑 baseline：准确率 78%，P95 延迟 3s
4. 用户测试：10 个目标用户 + 任务完成率
5. 结论：准确率需到 90% 才上线，当前差 12%，需 RAG 优化
```

## 检查清单
- [ ] 问题定义、成功标准和约束已经明确。
- [ ] 已覆盖能力、成本、供应商风险和替代方案。
- [ ] 有试点或验证路径，而不是直接全量切换。
- [ ] 结论包含保留条件和退出策略。

## 反模式

### FAIL: 跟风选型

```
"OpenAI 出 Assistants API 了，立项接入！"
→ 没问业务问题、没评估替代方案、没估迁移成本
→ 6 个月后 OpenAI 重构 API → 全推倒
```

### PASS: 问题驱动 + 试点

```
1. 业务问题：客服自动化（明确：减 30% 人工时长）
2. 候选：Assistants API / 自建 RAG / Botpress 三个方案
3. 用 1 个低风险场景跑 2 周 POC
4. 评估：质量 / 延迟 / 成本 / 锁定风险
5. 选定后仍保留迁移方案（抽象层）
```

### FAIL: 只比价格

```
"A: $100/月  B: $50/月 → 选 B"
→ B 缺关键集成 → 自研 → 工程投入 6 周
→ 实际总成本是 A 的 5 倍
```

### PASS: TCO 三层

```
| 项 | A | B |
| 订阅 | $1.2k/yr | $600/yr |
| 集成工程 | 0 | 6 周 ≈ $30k |
| 维护 | 低 | 持续 1 周/季 |
| 锁定 | 中 | 低 |
| 3 年 TCO | $3.6k | $50k+ |
```

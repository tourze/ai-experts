## 代码模式
- 代码库模式先执行 `scripts/scan_codebase.mjs <代码路径>` 获取结构指纹。
- 按需加载 `references/*.md`，只读当前维度对应的参考。
- 报告顺序推荐：评分总览 → 关键风险 → 分维度结论 → 修复优先级。

```bash
node scripts/scan_codebase.mjs /path/to/codebase
```

## Exhaustive 模式
- 触发条件：用户要求 "穷举式拆解 / subsystem audit / failure modes / 影响面审计"，或代码库规模较大且需要逐子系统盘点。
- 步骤：
  1. **子系统清单**：列出 `模块 | 文件范围 | 优先级 | 副作用` 表，覆盖状态系统、API、并发、UI、数据处理、配置六类。
  2. **逐子系统审计**：对每个子系统按七维度收敛具体发现，绑定 `file:line` 证据。
  3. **跨子系统去重**：把多个子系统命中的同一根因合并为一条。
  4. **症状与根因分离**：发现里区分"现象"和"为什么会发生"。
  5. **风险排序**：按 `Critical → High → Medium → Low` 给修复顺序，与七维评分相互印证。

## 反模式

### FAIL: 跳过输入分类

```
拿到代码库 → 直接打分
- 结构: 4
- 性能: 3
→ 用户："你看了哪些文件？"
→ 答不上 → 报告失信
```

### PASS: 先扫描再评

```bash
node scripts/scan_codebase.mjs /path/to/repo
# 输出：技术栈 / 模块树 / 依赖图 / 关键配置
# 基于扫描结果决定加载哪些 reference
# 报告每个评分都引用具体证据
```

### FAIL: 风险列表无优先级

```md
## 风险
- DB 单点
- 缺失监控
- 缺测试
- 文档过时
- ...
→ "都是问题，先修哪个？" → 不知道
```

### PASS: S1-S5 + 修复顺序

```md
| S | 风险 | 影响 | 修复成本 | 优先级 |
| S1 | DB 单点 | 全站不可用 | 中（1 周） | 立即 |
| S2 | 监控缺失 | 故障发现晚 | 低（3 天） | 本月 |
| S3 | 测试覆盖 30% | 回归风险 | 高（持续） | 季度 |
| S4 | 文档过时 | 上手慢 | 中 | 季度后 |
```

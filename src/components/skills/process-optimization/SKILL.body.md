# 流程优化

## 适用场景
- 运营、交付、审批、售前、需求流转等流程过慢或质量不稳。
- 需要从系统视角看依赖关系时，可配合 [systems-thinking](../systems-thinking/SKILL.md)；涉及 Backlog 或 Sprint 流程时，可配合 [agile-product-owner](../agile-product-owner/SKILL.md)。

## 核心约束
- 先画出现状流程和等待点，再讨论优化；不要跳过事实直接给方案。
- 优化目标必须量化，例如周期、一次通过率、返工率或等待时间。
- 先消除瓶颈和无价值步骤，再考虑自动化。

## 代码模式
```markdown
| 步骤 | 责任人 | 周期 | 等待/返工 | 是否保留 |
| --- | --- | --- | --- | --- |
```

## 检查清单
- [ ] 现状流程、瓶颈和浪费点已画清。
- [ ] 未来状态流程说明了责任、输入输出与指标。
- [ ] 已区分流程问题、资源问题和能力问题。
- [ ] 变更后有验证指标和回顾节奏。

## 反模式

### FAIL: AI 自动化一切

```
"客服流程慢 → 上 AI Agent 全自动化"
→ 6 个月项目 → 30% 案例 AI 答错 → 仍需人工兜底 → 总成本翻倍
```

### PASS: 先量化 + 修瓶颈

```
现状画图：客户问 → 工单 → 分类 → 分派 → 回复
量化：分类 0.5h / 分派 8h（等待）/ 回复 0.5h
→ 瓶颈是分派等待，不是回复速度
→ 改自动分派路由（简单规则即可），周期减 80%
→ AI 只在分类阶段做辅助
```

### FAIL: 不画现状

```
"我觉得审批太慢，去掉一步吧"
→ 删掉财务审核 → 一个月后大额支出失控 → 加回更多审核
```

### PASS: 现状图 + 数据

```
1. 画 8 步审批，每步标耗时 + 必要性
2. 数据：5 步真实耗时 < 1 天，3 步 > 5 天
3. 调研慢步原因：审批人不在 / 等其他文件 / 默认流转
4. 针对性修复：批量授权 / 并行审批 / 系统提醒
```

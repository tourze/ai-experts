## 核心约束

**违反字面规则 = 违反规则精神。不存在"灵活变通"。**

<HARD-GATE>
在展示设计方案并获得用户批准之前，不启动任何实现 skill、不写任何代码、不创建任何文件、不做任何脚手架操作。无论任务看起来多简单，都适用。
</HARD-GATE>

## 反模式："这个太简单了，不需要设计"

每个项目都走这个流程。一个工具函数、一条配置改动、一个 TODO 清单——全部。"简单"的项目恰恰是未经审视的假设浪费最多时间的地方。设计可以短（真正简单的项目几句话即可），但必须展示并获得批准。

## 流程

### 第 1 步：探索项目上下文

- 检查相关文件、文档、最近提交。
- 如果请求涉及多个独立子系统，先提出拆分建议，不要在一个设计里试图涵盖所有。

### 第 2 步：逐个提问澄清

- **一次只问一个问题**——不要一口气抛出 5 个问题。
- 优先用选择题（"A 还是 B？"），减少用户认知负担。
- 聚焦理解：目的、约束、成功标准。

### 第 3 步：提出 2-3 个方案

- 每个方案附带权衡分析。
- 明确推荐某个方案，并解释原因。
- 让用户选择，不替用户做决定。

### 第 4 步：分段展示设计

- 按复杂度调整每段篇幅：简单的几句话，复杂的 200-300 字。
- 每段之后确认："到目前为止是否正确？"
- 覆盖：架构、组件、数据流、错误处理、测试策略。

### 第 5 步：获得用户批准

- 设计确认后，才转向实现。
- 用户说"不对"→ 回到第 2 步或第 3 步修正。
- 用户说"可以"→ 转向 `feature-dev` / `task-decomposer` / `plan-review`。

## Iron Law

```
没有批准的设计，不写代码
```

## Red Flags — 出现以下念头时立即停下

| 念头 | 现实 |
|------|------|
| "用户说了要做什么，直接动手" | 说了"做什么"不等于说了"怎么做"。先设计。 |
| "这个太简单了，不需要讨论" | "简单"项目的未审视假设浪费最多时间。 |
| "我先写个原型看看" | 原型变成正式代码的概率 > 90%。先设计。 |
| "先搭个骨架，细节后面再说" | 骨架就是设计决策。先讨论。 |
| "用户赶时间，我先动手" | 花 3 分钟讨论 vs 花 30 分钟推翻重做。 |
| "我知道最佳实践是什么" | 你知道通用做法，不知道用户的具体约束。先问。 |

## Rationalization 对照表

| 借口 | 现实 |
|------|------|
| "只是加个按钮" | 按钮的行为、样式、位置、响应都是设计决策。 |
| "我已经在脑中设计好了" | 脑中的设计未经用户验证。说出来，确认。 |
| "设计文档太重了" | 不需要文档，需要 3 分钟对话。 |
| "用户说了'随便做'" | "随便做"不是设计批准。至少展示方案让用户确认。 |
| "改的是内部实现，用户不关心" | 内部实现影响性能、可维护性、可测试性。至少自检。 |

## 设计对话的原则

- **一次一个问题**——不要一口气问 5 个。
- **选择题优先**——比开放问题更容易回答。
- **YAGNI 无情**——从所有设计中移除不必要的功能。
- **先选方案再展开**——避免在未选定方向前深入细节。
- **增量验证**——展示设计 → 获得批准 → 再展开下一段。

## 在已有代码库中工作

- 动手前先探索现有结构，遵循已有模式。
- 如果已有代码存在影响当前工作的问题（文件过大、边界不清），把针对性改进纳入设计——不做无关的重构。

## 检查清单

- [ ] 已探索项目上下文（文件、文档、提交）
- [ ] 已通过提问理解用户意图和约束
- [ ] 已提出 2-3 个方案并附权衡
- [ ] 已分段展示设计并获得确认
- [ ] 用户明确批准了设计方案
- [ ] 才转向实现

## 何时可以简化

- 用户 prompt 中已包含完整设计规格（文件路径 + 接口 + 行为描述）→ 确认理解后直接实现。
- 用户明确说"直接做 / 不用讨论 / 放手干" → 跳过。
- 修 bug / 改配置 / 改常量等非创造性任务 → 不适用本 skill。

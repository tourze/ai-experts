## 适用场景

- 用户还没锁定具体仓库，只知道方向，例如 agent memory、RAG、浏览器自动化、工作流引擎。
- 需要给出可比较、可决策的 Top N 仓库榜单，而不是一堆散链接。
- 若用户已经锁定某个仓库并要做深入拆解，使用代码库深度分析工具。

## 核心约束

- 默认数据源是 GitHub 公开仓库；若没有额外说明，不假定用户提供 token。
- 默认硬过滤：`stars >= 100`、`archived = false`、公开仓库。
- 如果用户已经明确给出主题、数量、排序偏好和门槛，可以直接执行；若这些信息缺失，再补最少澄清。
- `stars` 是门槛，不应该成为唯一排序依据；结果必须综合相关性、活跃度和工程成熟度。
- 每个推荐项都要回答两件事：“它是什么”“为什么在当前目标下值得推荐”。

## 代码模式

### 1. 收敛输入

优先补齐这些信息：

- 主题
- 数量（Top 5 / 10 / 20）
- 最低 stars
- 排序模式：相关性优先 / 星标优先
- 目标形态：可直接使用的产品 / 可二次开发的框架 / 资料清单

### 2. 设计 5-10 组查询

每组查询至少说明目的，例如：

```text
Query-1: "agent memory framework"
目的：高召回核心主题

Query-2: "long-term memory for ai agents"
目的：补同义词盲区
```

### 3. 召回、去重、过滤

- 每组 query 抓 30-50 个候选。
- 按 `owner/repo` 去重。
- 应用默认硬过滤或用户指定门槛。
- 记录检索时间与配额状态，避免结果不可复现。

### 4. 重排与分类

建议按以下维度综合排序：

- 需求相关性
- 场景适配度
- 活跃度（最近更新时间）
- 工程成熟度（README、示例、License、维护迹象）
- stars

同时给仓库打“角色标签”：

- 通用框架层
- 应用产品层
- 基础设施层
- MCP / 工具服务层
- 目录清单层
- 方法论 / 研究层

### 5. 交付表格

```markdown
| 仓库 | 星标 | 类型 | 项目介绍（是什么 + 为什么推荐） | 其它信息补充 | 链接 |
|---|---:|---|---|---|---|
```

## 检查清单

- 信息不全时，是否只补必要澄清，而不是机械阻塞整个流程。
- 是否生成了多组 query，而不是只搜一个词。
- 是否记录检索时间、过滤规则和排序依据。
- 是否完成去重、硬过滤、噪音剔除和类型分类。
- 是否每个推荐项都写清“是什么 + 为什么推荐”。
- 是否在结论里明确下一步：继续缩小范围，还是转入深度分析深挖单仓库。

## 反模式

### FAIL: 只按 stars 排

```
用户："找浏览器自动化框架"
你：列了 30k+ star 的 puppeteer / selenium / playwright
→ 漏掉 stagehand / browser-use / browserless 等更对口产品
→ 高星不等于高相关
```

### PASS: 综合排序

```
排序权重 = 0.4×相关性 + 0.2×活跃度 + 0.2×成熟度 + 0.1×stars + 0.1×角色匹配
- 相关性：query 匹配度 + 描述匹配度
- 活跃度：最近 90 天 commit 数
- 成熟度：README + License + 示例 + tests
- 角色匹配：用户要框架还是成品？
```

### FAIL: 不分类型

```
| Top 10 |
| LangChain（框架）|
| awesome-agents（清单）|
| AutoGPT（产品）|
| ...
→ 用户："我要框架，清单和成品请别混进来"
```

### PASS: 角色标签

```
| 框架层 | LangChain / LlamaIndex / Haystack |
| 产品层 | AutoGPT / GPT Engineer |
| 清单层 | awesome-agents / awesome-llm-apps |
| MCP/工具 | mcp-server-fetch / browser-use |
→ 用户按需挑分类
```

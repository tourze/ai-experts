## 核心约束
- 旅程必须绑定具体 persona 与目标任务，不能写成“所有用户的通用流程”。
- 每个阶段至少写清触点、用户心智、阻力、指标与改进机会。
- 不要把组织内流程当成用户旅程；用户看不见的内部步骤应单独记录。

## 代码模式
```markdown
| 阶段 | 触点 | 用户目标 | 情绪/阻力 | 指标 | 机会点 |
| --- | --- | --- | --- | --- | --- |
```

## 检查清单
- [ ] Persona、阶段定义和核心目标已明确。
- [ ] 关键触点、情绪变化、流失点和机会点都有证据来源。
- [ ] 已说明哪些问题值得优先解决，以及为什么。
- [ ] 输出可以直接衔接需求、实验或服务改造。

## 反模式

### FAIL: 内部 SOP 当旅程

```
1. 用户提交申请
2. 客服分配工单
3. 后台审核团队 review
4. 财务对账
5. 发送通知
→ 用户其实只感知 1 和 5，中间黑盒等 3 天
→ 真问题（等待焦虑）被忽略
```

### PASS: 用户视角

```
| 阶段 | 用户做什么 | 心智 | 情绪 |
| 提交 | 填表 + 等待 | "我做对了吗" | 紧张 |
| 等待 | 反复打开 app 看状态 | "审核要多久" | 焦虑 ↑ |
| 通过 | 收到通知 | "终于！" | 释然 |
→ 机会点：等待阶段加进度提示 / 预期管理
```

### FAIL: 所有阶段都是问题

```
| 阶段 | 问题 |
| 认知 | 渠道太少 |
| 注册 | 步骤太多 |
| 激活 | 引导不足 |
| 留存 | 通知太多 |
| 推荐 | 动力不足 |
→ 全是 P0，团队不知道先修哪个
```

### PASS: 量化 + 排序

```
| 阶段 | 流失率 | 影响 LTV | 优先级 |
| 认知 → 注册 | 80% | -$200 | P0 |
| 注册 → 激活 | 60% | -$150 | P0 |
| 激活 → 留存 | 30% | -$50  | P1 |
→ 先修认知 → 注册的最大流失
```

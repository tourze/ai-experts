# 避免坏数据

## 适用场景

- 访谈对象都说“不错”，但没有任何推进动作。
- 问题设计里有大量假设题、意见题或诱导题。
- 团队想把礼貌性反馈当作需求验证证据。

## 坏数据类型

| 类型 | 典型信号 | 替代做法 |
| --- | --- | --- |
| 礼貌性表扬 | “挺好的”“我会考虑” | 问上次实际行为和下一步承诺 |
| 假设题 | “如果有 X 你会买吗” | 问当前如何解决和是否已付费 |
| 意见题 | “你觉得这个功能怎么样” | 请对方展示真实流程 |
| 未来承诺 | “上线后我会用” | 现在是否愿意引入同事、给数据或付定金 |
| 选择偏差 | 只问活跃粉丝 | 补充流失、沉默和低频用户 |

## 过滤规则

- 没有具体时间、场景和行为的反馈，不能当强证据。
- 没有成本、替代方案或损失的痛点，优先级要下调。
- 没有下一步动作的购买意愿，只能算弱信号。
- 访谈者解释越多，数据越可能被污染。

## 记录模板

```md
原话：
上下文：
行为证据：
付出成本：
下一步承诺：
证据等级：弱 / 中 / 强
```

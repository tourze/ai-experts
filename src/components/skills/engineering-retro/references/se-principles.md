# 软件工程原则

当作速查表使用。在 diff 中发现某种模式时，在此查找对应原则。每条包含**代码信号**——即在具体改动中应观察什么。

## 设计原则（SOLID）

**单一职责原则（SRP）**
一个模块应该只有一个变更的理由。
代码信号：一个类/文件被拆成两个。函数被提取出来。组件不再同时处理 UI 和数据获取。

**开闭原则（OCP）**
对扩展开放，对修改关闭。
代码信号：添加新行为而未修改已有代码。引入了插件/钩子/回调机制。用策略模式或配置替代条件分支。

**里氏替换原则（LSP）**
子类型必须能替换其基类型。
代码信号：引入接口统一多种实现。子类覆写的行为导致调用方出错（违反）。添加了类型收窄或守卫。

**接口隔离原则（ISP）**
客户端不应依赖它不使用的方法。
代码信号：大接口被拆成小接口。可选方法从接口中移除。臃肿的 props 对象被拆分为聚焦的子对象。

**依赖倒置原则（DIP）**
依赖抽象，而非具体实现。
代码信号：具体依赖被替换为接口/注入。引入工厂或提供者模式。导入路径从具体实现改为抽象层。

**组合优于继承**
优先使用对象组合，而非类继承。
代码信号：继承层级被替换为委托。Mixin 或 HOC 被替换为 hooks 或组合。移除了"基类"。

## 简洁原则

**DRY（不要重复自己）**
每一项知识应该有且仅有一个表示。
代码信号：重复代码被提取为共享函数。常量替代了重复的字面量。模板/生成器替代了复制粘贴的样板代码。

**KISS（保持简单愚蠢）**
能工作的最简方案就是最好的。
代码信号：复杂抽象被替换为直白实现。移除了不必要的间接层。巧妙的一行代码被替换为可读的写法。

**YAGNI（你不会需要它）**
在真正需要之前不要提前构建。
代码信号：移除了推测性功能。删除了未使用的配置选项。过度工程化的方案被简化以匹配实际需求。

**三次法则**
等到第三次重复再抽象。
代码信号：两处相似代码未被抽象（正确）。仅用了一次就引入了抽象（违反）。第三次出现触发了提取（标准做法）。

**最小惊讶原则**
代码行为应符合大多数用户的预期。
代码信号：函数被重命名以更准确描述其行为。返回类型变得一致。副作用被显式化或移除。

## 结构原则

**关注点分离**
不同职责应放在不同模块中。
代码信号：业务逻辑从 UI 组件中提取。数据访问与领域逻辑分离。配置与行为分离。"上帝文件"被拆分为聚焦的模块。

**高内聚**
相关功能应放在一起。
代码信号：分散的相关函数被归入同一模块。工具文件被拆分，每部分靠近其使用者。创建了特性文件夹。

**松耦合**
模块之间应尽可能少地相互依赖。
代码信号：直接导入被替换为事件/回调。移除了共享依赖。模块通过定义良好的接口通信，而非深入彼此内部。

**封装**
隐藏内部细节，只暴露必要部分。
代码信号：公共方法减少。内部辅助函数变为私有。模块的 API 表面缩小。实现细节隐藏在门面之后。

**信息隐藏**
模块不应暴露其内部数据结构。
代码信号：原始数据结构被包装在访问方法中。内部状态变为私有。数据转换移到拥有该数据的模块内部。

## 务实原则

**童子军规则**
让代码比你发现时更好。
代码信号：在功能变更中顺手做了小清理。重命名变量以提高清晰度。移除了死代码路径。更新了过时注释。

**快速失败**
尽早检测和报告错误。
代码信号：在入口点添加输入校验。为不变量添加断言。用早返回替代深层嵌套。错误处理移到更接近源头的地方。

**防御式编程**
预见并妥善处理意外输入。
代码信号：添加空值检查。提供默认值。处理边界情况。引入错误边界。

**过早优化（应避免）**
在测量之前不要优化。
代码信号：选择简单实现而非"更快"的复杂实现。可读性优先于微优化。在优化工作前加入了性能分析步骤。

## 重构模式

**提取方法/函数**
代码信号：长函数被拆分为命名的子函数。内联逻辑被替换为命名清晰的调用。

**提取类/模块**
代码信号：一个文件被拆分为多个文件。一个类被拆分为两个职责不同的类。

**用多态替代条件分支**
代码信号：switch/if-else 链被替换为策略模式或子类分发。引入了类型映射。

**引入参数对象**
代码信号：多个相关参数被归组为单一选项/配置对象。函数签名得到简化。

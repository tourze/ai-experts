本 skill 为所有语言/框架 reviewer agent 提供统一的方法论骨架。每个 reviewer agent 在此基础上追加自己的场景路由编排表。

## 工作方式

1. 先确认用户目标、输入范围、约束和验收标准。
2. 读取相关文件、配置、调用点、测试和同层模式，建立证据链。
3. 按安全性 → 正确性 → 影响面 → 执行成本排序输出。

## 证据规范

每条发现必须绑定到可核验定位：
- **文件:行**：精确到行号
- **代码片段**：摘录关键语句，不是翻译或摘要
- **严重级别**：阻断 / 高风险 / 建议 / 信息

每条结论标注证据强度：
- **事实**：可直接从代码、日志、配置中验证
- **推断**：基于模式和经验，需要进一步确认
- **假设**：未验证的前提，如果错误会推翻结论

## Bash 使用边界

Bash 只用于只读探测：版本查询、git 历史、文件统计、lint/typecheck 运行、测试执行。禁止安装依赖、删除/移动文件、修改配置、运行破坏性命令。

## 输出格式

```markdown
# <语言/框架> 代码审查：<scope>

## 摘要
[一句话结论 + 1-3 个关键发现]

## 环境
[语言版本、框架版本、关键依赖、配置文件]

## 发现
[按严重级别排列，每条绑定文件:行 + 证据强度标注]

## 专项审计
[按场景路由表逐项审计，每项标注使用的 skill]

## 正向观察
[做得好的地方，避免只报问题]

## 优先行动
[按安全性 > 正确性 > 影响面 > 成本排序，每条可独立验证]

## 范围限制
[未覆盖的路径、平台、场景，以及原因]
```

## 质量标准

- 优先处理安全、正确性、数据完整性和用户可见风险。
- 区分框架惯例、主观风格偏好和必须修复的问题。
- 发现性能问题时说明触发条件、影响范围和验证方式。
- 不虚构命令、退出码、测试结果；无证据不得声称"通过/可提交"。
- 每条发现必须可独立验证：审查者应能根据文件:行 + 代码片段自行判断。

## 编排约束

各 reviewer agent 的 body 应包含两个部分：

### 必经门禁（每次审查必跑）

| 步骤 | skill | 检查什么 |

至少包含：静态检查/lint → 安全红线 → 证据标注。

### 场景路由（按 diff 内容触发）

| 触发信号 | 使用 skill | 检查项 | 输出 |

按 diff 中出现的模式匹配，不是全量加载所有 skill。每个场景路由行定义：什么信号触发 → 用哪个 skill → 查什么 → 产出什么结论。

# 代码审查维度

## Iron Law — 四要素格式

```
每条发现必须包含四个要素，缺一不可：
Symptom  → 在代码中观察到了什么（含文件位置和代码证据）
Source   → 为什么这是问题（引用设计原则或工程常识）
Consequence → 如果不修复会怎样（具体的风险，不是抽象的"不好维护"）
Remedy   → 怎么修（给方向，不给整段代码）
```

缺少任何一个要素的 finding 都是噪音，不是审查。

---

## 1. 命名与语义

### 检查项
- 变量名、函数名、类名是否准确反映其用途。
- 布尔变量是否用 `is`/`has`/`can`/`should` 等前缀。
- 是否存在误导性命名（名字说的和做的不一致）。
- 缩写是否在上下文中无歧义。

### 严重度指引
- 🔴 关键：命名与实际行为相反，会导致调用方误用（如 `isValid` 返回时不检查所有约束）
- 🟡 重要：命名模糊，需要读实现才能理解（如 `data`、`result`、`process()`）
- 🟢 建议：可以更精确但不会导致误解（如 `list` vs `users`）

### 不应标记
- 领域术语 / 短循环 `i`/`j`/`k` / 测试惯用名（`sut`、`got`、`want`）

---

## 2. 函数设计

### 检查项
- 函数是否只做一件事（单一职责）。
- 函数长度是否合理（超过 30-40 行需关注）。
- 参数数量是否过多（超过 3-4 个需关注）。
- 是否有副作用未在函数名或文档中体现。
- 返回值是否一致（不要有时返回值有时返回 null 又有时抛异常）。

### 严重度指引
- 🔴 关键：函数 > 50 行且混合多层抽象；参数 > 6 个；一个函数承担 3+ 个不相关职责
- 🟡 重要：函数 30-50 行；参数 4-6 个；存在未文档化的副作用
- 🟢 建议：函数 20-30 行但逻辑清晰；参数可用对象封装但不紧急

### 不应标记
- 线性流程且命名清晰（长≠复杂） / 构造函数参数 / 框架签名

---

## 3. 错误处理

### 检查项
- 是否有裸的 catch-all（吞掉所有异常不做任何事）。
- 错误是否被正确传播或转换（不丢失上下文）。
- 是否区分了可恢复错误和不可恢复错误。
- 边界输入（空值、空集合、零、负数、超大值）是否有处理。

### 严重度指引
- 🔴 关键：吞掉异常且无日志；错误路径导致数据不一致或静默丢失数据
- 🟡 重要：错误传播丢失上下文（如 `throw new Error("failed")`）；边界输入未处理
- 🟢 建议：catch 块可以更精确但当前不会出错

### 不应标记
- 顶层兜底 catch-all / 内部代码不做防御性校验 / catch 范围略宽但无害

---

## 4. 逻辑与边界

### 检查项
- 循环是否有正确的终止条件。
- 条件分支是否覆盖所有情况（特别是 else/default）。
- 是否存在 off-by-one 错误。
- 并发访问是否安全（如果适用）。
- 时间和顺序依赖是否显式处理。

### 严重度指引
- 🔴 关键：可导致无限循环、数据损坏或安全漏洞
- 🟡 重要：边界条件遗漏可能在特定输入下出错
- 🟢 建议：分支可更完整但当前输入范围内不会触发

### 不应标记
- 已穷尽的分支省略 else / 单线程上下文的并发问题 / 惯用写法的理论 off-by-one

---

## 5. DRY 与抽象

### 检查项
- 是否有重复代码块可以抽取。
- 抽象层次是否一致（同一函数内不应混合高层逻辑和底层细节）。
- 是否为了 DRY 而创建了不自然的抽象（过度 DRY 也是问题）。

### 严重度指引
- 🔴 关键：同一业务规则在 3+ 处重复，且已经因不同步导致 bug
- 🟡 重要：2 处重复，修改时容易遗漏；抽象层混合导致理解困难
- 🟢 建议：轻微重复但尚未造成问题；可考虑提取但不紧急

### 不应标记
- 测试代码重复 setup / ≤3 行相似代码 / 跨模块结构相似但语义不同

---

## 6. 可读性

### 检查项
- 控制流是否可以用 early return 简化。
- 嵌套层级是否过深（超过 3 层需关注）。
- 注释是否解释了"为什么"而不是"做了什么"。
- 代码组织是否符合阅读习惯（公开接口在前，内部实现在后）。

### 严重度指引
- 🔴 关键：嵌套 > 5 层（箭头反模式）；控制流无法通过阅读理解
- 🟡 重要：嵌套 3-5 层；注释与代码矛盾；代码组织混乱
- 🟢 建议：可用 early return 简化但当前可读；注释可删除

### 不应标记
- 多个 early return（guard clause） / 自解释代码无注释 / 团队约定的组织顺序

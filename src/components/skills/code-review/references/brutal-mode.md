# 高压审查模式

`code-review` 的高压模式，原名 `brutal-honesty-review`，已合并为审查强度选项。

## 适用场景
- 资深工程师或成熟团队明确要求直接、尖锐、可执行的反馈
- 代码、测试、流程、供应商承诺存在明显自欺或低标准问题
- 普通评审话术已经无法传达风险

## 语气档位

| 档位 | 适用 |
|------|------|
| Linus | 技术错误、并发/性能/架构错误 |
| Ramsay | 质量标准太低、测试像摆设 |
| Bach | 流程、认证、供应商宣传存在自欺 |

## 约束
- 只攻击工作产物，不攻击作者人格
- 每条批评都要有证据，至少包含文件/行为/结果中的一项
- 必须同时给出"为什么错"和"怎样改"，不能只发泄
- 若无明确授权，不对新人首个 PR、低心理安全团队、公开场合使用
- 如果没有发现问题，要明确写"未发现明确缺陷"，并说明剩余风险

## 输出骨架

```markdown
## 发现 1
- 结论：[一句话定性]
- 证据：[path/to/file.ts:42]
- 原因：[根因]
- 正确做法：[方向]
- 修复建议：[可执行步骤]
```

## 反模式

### FAIL: 情绪宣泄
"这代码烂透了，谁写的？" → 0 修复建议 / 攻击作者人格

### FAIL: 无授权高压
新人首个 PR 被高压审查 → 心理伤害 / 团队失信 / 离职风险

### PASS: 攻击产物 + 修复
```md
发现：测试只验证 mock 调用次数（auth.test.ts:42-78）
后果：业务逻辑改了 200 行，测试不会发现
修复：改为对真实输入/输出断言，mock 仅保留外部 API 边界
```

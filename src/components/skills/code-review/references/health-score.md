# Health Score 与输出格式

## 输出格式

```markdown
## 审查结论

**Health Score: XX/100**

共发现 N 个问题：🔴 关键 X 个，🟡 重要 Y 个，🟢 建议 Z 个。

### 🔴 关键

**[维度] 问题标题** — `path/to/file:42`
- **Symptom:** 在代码中观察到了什么（引用具体代码）
- **Source:** 为什么这是问题（设计原则 / 工程常识）
- **Consequence:** 不修复会怎样（具体风险）
- **Remedy:** 怎么修（方向，不给整段代码）

### 🟡 重要

（同上四要素格式）

### 🟢 建议

（同上四要素格式）
```

## Health Score 计算

基础分 100，按 finding 扣分：

| 严重度 | 每条扣分 | 上限 |
|--------|---------|------|
| 🔴 关键 | -15 | 无上限 |
| 🟡 重要 | -7 | 无上限 |
| 🟢 建议 | -3 | 最多扣 15 分 |

最低分 0。

| 分数 | 解读 |
|------|------|
| 90-100 | 质量优秀，可放心合并 |
| 75-89 | 质量良好，处理重要问题后合并 |
| 60-74 | 需改进，应修复关键和重要问题 |
| 40-59 | 堪忧，建议重新审视设计 |
| < 40 | 需重大重构 |

## 打分纪律
- 无 finding 不硬凑——100 分合法
- 不因风格偏好扣分——只扣实际风险
- 同一根因的多个 symptom 合并为一条 finding，不重复扣分

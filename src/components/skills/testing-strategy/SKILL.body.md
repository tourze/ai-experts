## 代码模式

### 模式 1：按模块列轻量测试计划

```markdown
| 模块 | 目标 | 测试层级 | 关键场景 |
|------|------|----------|----------|
| 登录接口 | 鉴权正确 | 单测 + 集成 | 正常登录、密码错误、锁定、限流 |
| 支付页 | 用户流程可用 | 组件测试 + E2E | 成功支付、取消、重复提交 |
| 定时任务 | 数据一致性 | 单测 + 集成 | 幂等、失败重试、时间边界 |
```

### 模式 2：覆盖目标

```markdown
- 单测：业务规则、边界值、错误处理
- 集成：数据库、外部 API、消息队列、权限边界
- E2E：关键用户路径、跨页面状态、浏览器交互
```

### 模式 3：生产级风险矩阵

```markdown
| 区域 | 业务影响 | 技术风险 | 推荐测试深度 | 备注 |
|------|----------|----------|--------------|------|
| 支付链路 | 高 | 高 | 单测 + 集成 + E2E + 性能 | 必须全链路回归 |
| 后台报表 | 中 | 低 | 单测 + 集成 | 可接受人工抽样 |
| 营销页 | 低 | 低 | 冒烟 + 可访问性 | 视觉回归按需 |
```

### 模式 4：质量门

```markdown
## 质量门

- 开发完成：单测通过，新增关键分支具备失败路径覆盖
- 提测前：关键 API 集成测试全绿，P0 缺陷为 0
- 发布前：冒烟回归通过，性能回归未超过基线 10%
```

## 反模式

### FAIL: 不分层级的测试建议

```
建议：
- 多写点测试
- 提高覆盖率
- 每个函数都要有测试
```

→ getter/setter 和第三方框架行为也要测？成本失控。

### PASS: 按风险分层

```markdown
| 模块 | 风险 | 层级 | 场景 |
|------|------|------|------|
| 支付 | 高 | 单测 + 集成 + E2E | 成功/失败/重复/超时 |
| 登录 | 高 | 单测 + 集成 | 正常/锁定/MFA |
| 关于页 | 低 | 冒烟 | 页面可访问 |
```

- 不区分单测、集成、E2E，导致成本失控。
- 把框架行为和样板代码当成重点测试对象。

### FAIL: 一刀切覆盖率

```markdown
## 测试策略
- 目标：所有模块达到 80% 覆盖率
- 方法：为每个函数写单元测试
```

→ 支付链路和营销页获得同等投入，高风险模块被平均化覆盖。

### PASS: 风险驱动的差异化投入

```markdown
## 测试策略
- 支付链路（高风险）：90% 分支覆盖 + 集成 + E2E + 性能基线
- 用户 CRUD（中风险）：核心路径单测 + API 集成
- 静态页面（低风险）：冒烟测试 + 可访问性扫描
```

### FAIL: 不可执行的质量门

```markdown
- 代码质量要高
- 测试要充分
- 性能要好
```

→ 无法在 CI 中自动判断通过/失败。

### PASS: 可测量的质量门

```markdown
- 单测全绿 + 新增代码 >= 80% 行覆盖
- API P95 延迟 <= 基线 x 1.1
- 安全扫描无 Critical/High
```

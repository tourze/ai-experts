## 第一步：问题界定（5W2H）

把宽问题收敛成可回答的经营问题：

| 维度 | 问题 |
|------|------|
| Why | 为什么现在要回答这个问题？决策是什么？ |
| What | 具体要分析什么现象/指标？ |
| Who | 谁在问？决策者是谁？ |
| Where | 哪个市场/产品线/渠道/客群？ |
| When | 时间窗是什么？基线和对比期？ |
| How | 数据从哪里来？分析资源有哪些？ |
| How much | 改善空间有多大才值得行动？ |

验收标准：问题界定完成时，任何同事读一遍都能说出"我们在回答什么、不回答什么"。

## 第二步：MECE 假设树

### 构造规则

- 一级分支互不重叠（Mutually Exclusive），整体覆盖问题空间（Collectively Exhaustive）
- 商业问题常见一级分支：市场/竞争/客户/产品/渠道/成本/外部环境
- 每个一级分支下挂 2-5 个可验证的二级假设
- 标注每个假设的验证优先级（P0: 占 >30% 解释力 / P1: 10-30% / P2: <10%）

### 常见错误

- 一级分支按"内部/外部"二分——太粗，失去 MECE 意义
- 二级假设不可验证（"品牌不够强"无法直接测试）
- 假设树变成"问题列表"而非"可证伪命题"

## 第三步：证据分层

每个发现标注证据级别：

| 级别 | 定义 | 示例 |
|------|------|------|
| 事实 | 有数据支撑、来源可查、可复现 | "Q3 留存率从 34% 降至 28%，数据来自 BI 看板" |
| 推断 | 基于事实的逻辑推导，但未直接验证 | "留存下降与 9 月价格调整时间吻合，可能是主因" |
| 假设 | 有逻辑但缺数据的猜想 | "竞品同期降价 15% 可能分流了价格敏感用户" |
| 待确认 | 需要外部数据或进一步调研才能判断 | "TAM 数据来自第三方报告，需交叉验证" |

禁止把推断写成事实，禁止把假设写成"显然"。

## 第四步：模型选择

按问题类型选模型，不按清单堆框架：

| 问题类型 | 首选模型 | 排除理由（为什么不用相邻模型） |
|---------|---------|--------------------------|
| 宏观环境判断 | PESTEL | 不需要行业竞争细节 |
| 行业吸引力 | 五力 | 不需要公司级经营数据 |
| 竞争定位 | 3C（Customer/Company/Competition） | 不需要宏观政治经济维度 |
| 商业模式评估 | BMC（Business Model Canvas） | 不需要行业对标 |
| 经营健康度 | 记分卡（财务/客户/流程/学习） | 不需要外部竞争数据 |
| 营销执行 | 4P（Product/Price/Place/Promotion） | 不需要战略层分析 |

未使用的模型要明确排除理由——防止 reviewer 问"为什么没用 X 框架"。

## 第五步：最小验证计划

为 P0 假设设计验证计划：

```
假设：<一句话>
数据来源：<具体表/报告/API>
验证方法：<对比/回归/AB/专家判断>
通过标准：<数值阈值>
反证信号：<什么结果能推翻这个假设>
```

P1/P2 假设可只写验证方向，不写完整计划。

## 检查清单

- [ ] 问题界定包含 5W2H 七个维度，有明确的"不回答什么"。
- [ ] 假设树一级分支 MECE，每个二级假设可验证。
- [ ] 每个发现标注了证据级别（事实/推断/假设/待确认）。
- [ ] 模型选择说明了选 A 的理由和不选 B/C/D 的理由。
- [ ] P0 假设有完整验证计划（数据来源/方法/通过标准/反证信号）。
- [ ] 行动建议 ≤3 条，每条有验证指标和停止条件。

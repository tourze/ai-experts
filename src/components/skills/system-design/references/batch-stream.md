# batch-stream

## 适用场景

- 需要设计日志、埋点、风控、推荐、报表、同步和数据湖管道。
- 需要在批处理、流处理、事件日志、CDC 和定时任务之间取舍。
- 需要说明延迟、重放、乱序、去重和 exactly-once 的真实边界。

## 基本判断

- 结果需要秒级、分钟级、小时级还是天级更新。
- 数据源是业务事件、数据库 CDC、日志还是文件。
- 是否需要重放历史事件来修复逻辑或回填特征。
- 事件是否可能乱序、重复、迟到。
- 下游是否可幂等写入。

## 模式对比

| 模式 | 适合场景 | 代价 |
| --- | --- | --- |
| 批处理 | 报表、回填、离线训练 | 延迟高，资源峰值明显 |
| 流处理 | 风控、监控、实时特征 | 状态管理、乱序、重放复杂 |
| Lambda 架构 | 同时需要实时和离线校准 | 两套逻辑容易漂移 |
| Kappa 架构 | 以事件日志为事实源 | 对日志保留和重放能力要求高 |
| CDC + outbox | 业务库变更同步 | schema 演进和幂等写入要谨慎 |

## 输出要求

- 写清事件 schema、版本策略和兼容规则。
- 写清 checkpoint、offset、重放窗口和死信队列。
- 写清去重键、幂等写入和迟到事件处理。
- 写清指标：消费 lag、吞吐、失败率、重放耗时、数据新鲜度。

## 反模式

- 承诺 exactly-once 但没有说明幂等 sink 和事务边界。
- 事件字段无版本策略，消费者被上游变更打崩。
- 只保留 1 天日志，却要求能回放 30 天历史。

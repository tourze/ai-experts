# fault-tolerance

## 适用场景

- 需要说明系统在网络抖动、节点故障、依赖超时、磁盘满、数据损坏时如何表现。
- 需要设计超时、重试、熔断、限流、降级、恢复和灾备。
- 需要把可用性目标转成具体 RTO/RPO、SLO 和告警。

## 故障模型

- 网络：丢包、延迟尖刺、分区、DNS 故障。
- 节点：进程崩溃、OOM、CPU 饱和、磁盘满。
- 依赖：数据库慢查询、缓存不可用、队列堆积、第三方 API 超时。
- 数据：重复消息、乱序、部分写入、坏数据、备份不可恢复。

## 设计检查

| 机制 | 必须说明 |
| --- | --- |
| 超时 | 每层超时预算，避免下游拖垮上游 |
| 重试 | 最大次数、退避、抖动、幂等条件 |
| 熔断 | 触发阈值、恢复探测、降级行为 |
| 限流 | 按用户/租户/API 维度，拒绝策略 |
| 备份恢复 | 恢复演练频率、RPO/RTO、校验方式 |

## 输出要求

- 写出最关键 3 个失败模式和用户可见行为。
- 写出是否 fail-open 或 fail-closed。
- 写出队列积压、缓存雪崩、数据库故障的降级策略。
- 写出监控指标：错误率、延迟、饱和度、lag、成功率。

## 反模式

- 只写“高可用部署”，不说明依赖故障时怎么退化。
- 无限重试导致雪崩。
- 备份从未恢复演练，RPO/RTO 只是口号。

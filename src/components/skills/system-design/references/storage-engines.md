# storage-engines

## 适用场景

- 需要解释为什么选关系型、文档型、KV、搜索或列式存储。
- 需要比较 B-tree、LSM tree、倒排索引、列存和缓存层的读写代价。
- 需要回答“这个索引是否值得建”、“写放大是否可接受”、“热点会不会拖垮主库”。

## 判断顺序

1. 明确主要访问模式：点查、范围查询、排序分页、聚合、全文检索、时序写入。
2. 估算读写比例、写入峰值、数据保留周期、数据增长曲线和冷热分布。
3. 判断索引是否服务核心查询，而不是为所有可能条件都建索引。
4. 说明写放大、空间放大、读放大、压缩和后台 compaction 对延迟的影响。

## 选型提示

| 场景 | 常见选择 | 主要代价 |
| --- | --- | --- |
| 强事务 + 复杂查询 | PostgreSQL / MySQL | 水平扩展和跨库事务复杂 |
| 大写入吞吐 + key/range 查询 | LSM/KV/宽列表存储 | compaction、读放大、二级索引限制 |
| 搜索和相关性排序 | Elasticsearch / OpenSearch | 写入延迟、索引膨胀、最终一致 |
| 分析型聚合 | 列式存储 / OLAP | 写入路径和实时更新成本 |
| 低延迟读热点 | Redis / 内存缓存 | 一致性、淘汰、击穿和回源压力 |

## 输出要求

- 写出核心查询样例和预计 QPS。
- 标明每个索引服务哪个查询，避免“索引越多越好”。
- 说明缓存是否允许 stale，以及失效策略。
- 如果选 LSM 或搜索索引，必须说明后台合并、重建和容量预留。

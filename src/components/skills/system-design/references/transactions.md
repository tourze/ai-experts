# transactions

## 适用场景

- 需要定义订单、库存、支付、账户、权限等强一致业务边界。
- 需要说明隔离级别、并发异常、幂等、补偿和恢复。
- 需要评估本地事务、分布式事务、SAGA、outbox 或事件最终一致。

## 先定边界

- 哪些不变量必须立即成立，例如余额不能为负、库存不能超卖。
- 哪些状态可以最终一致，例如搜索索引、推荐特征、报表。
- 哪些写入必须同库同事务完成，哪些可以通过事件异步传播。
- 重试是否安全，幂等键如何生成和过期。

## 隔离级别提示

| 问题 | 可能需要 |
| --- | --- |
| 脏读、不可重复读不可接受 | Read Committed 或更高 |
| 写偏斜会破坏业务不变量 | Serializable 或显式锁 |
| 库存扣减/余额变更 | 条件更新、行锁、乐观锁版本号 |
| 跨服务状态推进 | outbox、SAGA、补偿任务 |

## 设计写法

```md
事务边界：
- 订单创建和库存预占在同一数据库事务内完成。
- 支付回调通过幂等键去重，重复回调返回上次处理结果。
- 发货、积分、通知走 outbox 事件，允许 1 分钟内最终一致。
- 补偿任务扫描 pending 状态超过 15 分钟的订单。
```

## 反模式

- “加事务”但不说明事务覆盖哪些表和哪些外部副作用。
- 在数据库事务中调用第三方 API，导致锁持有时间不可控。
- 双写数据库和消息队列但没有 outbox 或重放校验。

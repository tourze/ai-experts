# partitioning

## 适用场景

- 单机容量、写入吞吐或热点压力接近上限，需要水平拆分。
- 需要选择分区键、解释热点、跨分区查询、再均衡和二级索引代价。
- 需要评估按租户、用户、时间、地理位置或 hash 分片。

## 分区键检查

- 基数是否足够高，是否能把写入均匀打散。
- 查询是否经常带分区键；不带分区键的查询会不会广播。
- 是否存在大租户、大 V 用户、热门商品或单日热点。
- 是否需要按时间归档、冷热分层或过期删除。
- 分区键是否会随业务变化，迁移成本多大。

## 常见策略

| 策略 | 优点 | 代价 |
| --- | --- | --- |
| Hash 分区 | 分布均匀，适合点查 | 范围查询和按时间归档困难 |
| Range/时间分区 | 范围查询、归档、冷热分层友好 | 当前时间窗口容易形成写热点 |
| 按租户/用户分区 | 数据隔离清晰，便于限流 | 大租户倾斜，需要二级拆分 |
| 复合分区 | 可兼顾维度 | 路由、迁移和调试复杂 |

## 输出要求

- 写出路由规则：请求如何从 key 找到分区。
- 写出热点缓解：盐值、二级分片、限流、队列削峰或拆大租户。
- 写出跨分区查询策略：是否允许、如何分页、如何聚合。
- 写出再均衡计划：在线迁移、双写校验、读切换和回滚。

## 反模式

- 用自增 ID 做分区键导致新写全部落到最后一个分区。
- 只按当前数据量设计，不说明增长后如何扩容。
- 跨分区事务和跨分区唯一约束没有边界说明。

## 代码模式
- 默认优先级公式：`(影响 + 风险) × (6 - 工作量)`。
- 输出建议包含债项、类型、评分、建议时机、收益和验证方式。
- 如果债项很多，先按模块或领域分组再排序。
- 必要时读取 [治理视角补充](references/guest-insights.md) 获取治理视角。
- 治理建议按“立即止血 / 近期治理 / 长期重构”三层切分。

## 反模式

### FAIL: 全 P0

```
| 债 | P |
| 命名不一致 | P0 |
| 缺测试 | P0 |
| 旧依赖 | P0 |
| 文档过时 | P0 |
| ...（30 项 P0） |
→ 等于没排
```

### PASS: 公式排序

```
优先级 = (影响 + 风险) × (6 - 工作量)

| 债 | 影响 | 风险 | 工作量 | 分数 |
| jwt secret 硬编码 | 5 | 5 | 1 | 50 |
| N+1 query | 4 | 3 | 2 | 28 |
| 命名不一致 | 1 | 1 | 4 | 4 |
→ 第一项立即修，最后一项可暂缓
```

### FAIL: 只列不治理

```md
## 技术债 backlog
- 1.0 ...
- 2.0 ...
- ...
（120 项，6 个月没动过）
```

### PASS: 配置每迭代预算

```md
治理策略：
- 每个迭代 20% 容量给债务
- 优先级前 3 项进入下个 sprint
- 季度复盘：清掉的 / 新增的 / 残余的
退出条件：
- 一项债 6 个月仍在 backlog → 移到 wontfix 或重新评估
```

### FAIL: 喊重写

```
"代码太烂了，全部重写"
→ 6 个月后：新代码也是烂的 + 旧代码还在跑 + 业务停滞
```

### PASS: 分期治理

```md
立即（本周）：止血
- 修复 OOM 高频崩溃（已影响线上）
近期（季度内）：
- Auth 模块从 PHP 4 升 8.x
- 拆出 PaymentService，独立测试覆盖到 80%
长期（半年）：
- 替换遗留 SOAP 接口为 REST（按业务迁移，不一次性切换）
```

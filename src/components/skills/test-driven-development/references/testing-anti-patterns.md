# 测试反模式

在新增测试、引入 mock 或编写测试工具前读取本文件。

## 1. 测试 mock 自己

坏味道：

- 断言几乎全是 `toHaveBeenCalledTimes`
- 看起来在测业务，实际上在测桩对象是否存在

正确做法：

- 优先断言真实输入/输出或可观察行为
- 只把 mock 用在外部边界，不把 mock 当被测对象

## 2. 往生产代码里塞测试专用方法

坏味道：

- 生产类里新增只给测试调用的 `destroy`、`resetForTest`、`forceState` 等方法

正确做法：

- 把清理逻辑放到测试辅助层
- 让生产对象只暴露真实业务接口

## 3. 不理解依赖链就盲目 mock

坏味道：

- “怕慢，先 mock 掉再说”
- mock 了高层方法，顺带砍掉测试依赖的副作用

正确做法：

- 先搞清楚真实方法有哪些副作用
- 只 mock 真正慢或不稳定的外部边界
- 需要时下沉一层 mock，而不是把高层流程整个掐断

## 4. 只造残缺假数据

坏味道：

- mock 响应只填自己眼前要用的字段
- 测试通过，集成一跑就因为缺字段炸掉

正确做法：

- 按真实 schema 造完整数据
- 至少覆盖下游会访问到的字段与边界值

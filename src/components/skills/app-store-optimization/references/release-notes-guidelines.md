# App Store 发布说明编写指南

## 目标
- 生成面向用户的发布说明，描述自上次 tag 以来的可见变更。
- 覆盖所有影响用户的变更；略过纯内部或仅重构的工作。
- 语言平实、简短，以用户收益为导向。

## 输出形态
- 大多数版本建议 5 到 10 条。
- 必要时按主题分组：新增、改进、修复。
- 每条一句话，以动词开头。
- 避免出现内部代号、工单 ID 或文件路径。

## 筛选规则
- 包含：新功能、UI 变更、行为变更、用户可感知的 bug 修复、有可见效果的性能改进。
- 排除：重构、依赖升级、CI 变更、开发者工具、内部日志、埋点变更（除非影响用户隐私或行为）。
- 若某项变更存在歧义，先确认或仅在确认对用户可见后描述为小幅改进。

## 语言指引
- 将技术术语转换为面向用户的描述。
- 避免出现"API""重构""nil""崩溃日志""依赖"等措辞。
- 优先使用"新增""改进""修复""更新"或"搜索""上传""同步"等动作动词。
- 时态保持现在时或过去时："新增""改进""修复"。

## 示例
- "新增从个人资料菜单切换账号的功能。"
- "改进弱网环境下的时间线加载速度。"
- "修复媒体附件无法全屏打开的问题。"

## QA 检查清单
- 每条均对应范围内的真实变更。
- 无描述同一变更的重复条目。
- 无内部行话或文件路径。
- 若提供了目标商店信息，最终列表需符合对应 App Store 的文本长度限制。

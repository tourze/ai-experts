# 版本规划

## 适用场景
- MVP 拆分、阶段性交付、需求排期、里程碑定义和路线图收敛。
- 需要把需求文档落成版本时，可配合 [create-prd](../../create-prd/SKILL.md)；需要进一步瘦身时，可配合 [scoping-cutting](../../create-prd/SKILL.md)。

## 核心约束
- 先明确每个版本要验证什么，再决定功能集合；版本不是简单按时间切块。
- 每个阶段都要写清用户价值、成功标准、风险与退出条件。
- 版本节奏必须考虑依赖和能力边界，避免“先堆一个大而全 V1.0”。

## 代码模式
```markdown
| 版本 | 核心目标 | 必做能力 | 延后项 | 成功标准 |
| --- | --- | --- | --- | --- |
```

## 检查清单
- [ ] 每个版本都有明确目标、边界和验证指标。
- [ ] 当前版本的必做项围绕关键用户路径。
- [ ] 延后项有去向，而不是永久挂起。
- [ ] 版本计划能衔接排期、资源和风险管理。

## 反模式

### FAIL: 大而全 V1

```
V1.0 路线图（3 个月）：
- 用户系统 + 计费 + 报表 + 集成 × 5 + 多语言 + ...
→ 第 4 个月仍未发布
→ 团队心力透支
→ 上线后没人用，因为定位不清
```

### PASS: 验证驱动

```
V0.1 (2 周)：核心场景 走得通
  目标：5 个种子用户能完成 1 个核心任务
V0.2 (4 周)：付费验证
  目标：3 个用户付费 ≥ $50
V0.3 (6 周)：留存验证
  目标：D30 留存 > 25%
→ 每个版本验证一个假设，失败即重审
```

### FAIL: "全面完善"

```
V2.0：全面完善体验
V3.0：进一步完善体验
V4.0：体验大升级
→ 没有具体目标，团队不知道何为完成
```

### PASS: 目标 + 边界

```
V2.0：减少新用户流失（D7 +5pp）
- 必做：onboarding 重做 + 首次任务引导
- 不做：付费功能 / 主题切换 / 国际化
- 验证：D7 留存 80% → 85%
→ 范围明确，可决策、可验收
```

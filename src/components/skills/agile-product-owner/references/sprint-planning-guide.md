## Sprint 规划工作流

### 规划前（1-2 天前）

1. 审查并精简下一个 Sprint 的待办事项
2. 确保顶部事项有验收标准
3. 与团队验证故事点估算
4. 识别故事之间的依赖关系
5. 确认团队在 Sprint 期间的可用性
6. **验证：** 顶部 1.5 倍容量的故事已精炼和估算

### Sprint 规划会议

**时长：** 2 周 Sprint 为 2 小时

**议程：**

| 时间 | 活动 | 参与者 |
|------|----------|--------------|
| 0:00-0:15 | 审查 Sprint 目标和优先级 | 产品负责人介绍 |
| 0:15-0:45 | 讨论顶部待办事项 | 团队提问 |
| 0:45-1:15 | 团队选择 Sprint 故事 | 团队决定 |
| 1:15-1:45 | 将故事拆分为任务 | 团队协作 |
| 1:45-2:00 | 确认承诺并识别风险 | 全员 |

### 规划检查清单

**规划前：**
- [ ] 待办事项已整理，顶部事项已精炼
- [ ] 审查了上一 Sprint 回顾行动
- [ ] 已计算团队容量
- [ ] 已识别依赖关系
- [ ] 草拟了 Sprint 目标

**规划期间：**
- [ ] Sprint 目标已达成一致
- [ ] 选定的故事在容量范围内
- [ ] 每个故事的验收标准已审查
- [ ] 复杂故事的任务已识别
- [ ] 已讨论风险和阻塞项

**规划后：**
- [ ] Sprint 待办事项对全员可见
- [ ] Sprint 目标已传达
- [ ] 日历已为仪式预留时间
- [ ] 依赖关系已传达给其他团队

## 容量规划

### 团队容量计算

```
Sprint 容量 = (团队成员数 × Sprint 天数 × 每天小时数 × 专注系数)
              ÷ 每个故事点的小时数

简化版本：
Sprint 容量 = 平均速度 × 可用性系数
```

### 可用性系数

| 场景 | 系数 | 示例 |
|----------|--------|---------|
| 完整 Sprint，无休假 | 1.0 | 速度 = 30 时为 30 点 |
| 1 名成员 50% 不在位 | 0.9 | 27 点 |
| Sprint 期间有假期 | 0.8 | 24 点 |
| 多名成员不在位 | 0.7 | 21 点 |
| 重大发布/值班 | 0.75 | 22-23 点 |

### 容量缓冲规则

| 承诺级别 | 速度百分比 | 目的 |
|------------------|---------------|---------|
| 承诺 | 80-85% | 高信心交付 |
| 延伸 | 10-15% | 可选，顺利时执行 |
| 缓冲 | 5-10% | 计划外工作、缺陷 |

### Sprint 加载示例

```
团队速度：30 点/Sprint
可用性：90%（一名成员部分不在位）
调整后速度：27 点

Sprint 加载：
- 承诺工作：23 点（27 的 85%）
- 延伸目标：4 点（27 的 15%）
- 缓冲：剩余容量用于缺陷/支持

故事选择：
[H] US-001：用户仪表盘（5 点）← 承诺
[H] US-002：导出功能（3 点）← 承诺
[H] US-003：搜索筛选（5 点）← 承诺
[M] US-004：设置页面（5 点）← 承诺
[M] US-005：帮助提示（3 点）← 承诺
[L] US-006：主题选项（2 点）← 承诺
------------------------
承诺总计：23 点

[L] US-007：排序选项（2 点）← 延伸
[L] US-008：打印视图（2 点）← 延伸
------------------------
延伸总计：4 点
```

## 待办事项优先级排序

### 优先级框架

| 优先级 | 定义 | SLA |
|----------|------------|-----|
| 关键 | 阻塞用户、安全、数据丢失 | 立即 |
| 高 | 核心功能、关键用户需求 | 本 Sprint |
| 中 | 改进、增强 | 接下来 2-3 个 Sprint |
| 低 | 锦上添花、微小改进 | 待办事项 |

### 优先级因素

| 因素 | 权重 | 问题 |
|--------|--------|-----------|
| 商业价值 | 40% | 收入影响？用户需求？战略意义？ |
| 用户影响 | 30% | 影响多少用户？使用频率？ |
| 风险/依赖 | 15% | 技术风险？外部依赖？ |
| 工作量 | 15% | 规模？复杂度？不确定性？ |

### WSJF（加权最短作业优先）

对于较大事项，使用 SAFe 的 WSJF：

```
WSJF = 延迟成本 / 作业时长

延迟成本 = 用户价值 + 时间紧迫性 + 风险降低

评分标准：1, 2, 3, 5, 8, 13, 20

示例：
功能 A：CoD = 13, 时长 = 5 → WSJF = 2.6
功能 B：CoD = 8, 时长 = 2 → WSJF = 4.0 ← 更高优先级
```

### 待办事项组织

| 分区 | 内容 | 审查频率 |
|---------|---------|------------------|
| Sprint 待办事项 | 当前 Sprint 承诺的任务 | 每日 |
| 就绪 | 已精炼、估算、排序 | 每次规划 |
| 精炼中 | 需要进一步精炼 | 每周 |
| 冷藏区 | 未来考虑 | 每月 |
| 归档 | 已完成或废弃 | 每季度 |

## Sprint 仪式

### 每日站会

**时长：** 最多 15 分钟
**格式：** 每位团队成员回答：

1. 我昨天完成了什么？
2. 我今天要做什么？
3. 我有什么阻塞项？

**产品负责人角色：**
- 倾听需要产品负责人行动的阻塞项
- 回答澄清问题
- 记录范围问题，会下讨论
- 向利益相关者更新进度

### 待办事项精炼

**时长：** 每周 1-2 小时
**时间：** Sprint 中期

**议程：**

| 时间 | 活动 |
|------|----------|
| 0:00-0:15 | 审查即将到来的优先级 |
| 0:15-0:45 | 详细说明顶部事项的验收标准 |
| 0:45-1:15 | 估算新故事 |
| 1:15-1:30 | 拆分大型故事 |

**就绪标准：**
- [ ] 清晰的用户故事格式（作为... 我想要... 以便...）
- [ ] 已定义验收标准（Given-When-Then）
- [ ] 故事点估算已达成一致
- [ ] 已识别依赖关系
- [ ] 可在一个 Sprint 内完成（≤8 点）

### Sprint 评审（演示）

**时长：** 2 周 Sprint 为 1 小时

**议程：**

| 时间 | 活动 | 负责人 |
|------|----------|------|
| 0:00-0:05 | 回顾 Sprint 目标 | 产品负责人 |
| 0:05-0:40 | 演示已完成工作 | 团队 |
| 0:40-0:50 | 利益相关者反馈 | 利益相关者 |
| 0:50-1:00 | 路线图更新 | 产品负责人 |

**演示检查清单：**
- [ ] 仅演示已完成（完成-完成）的故事
- [ ] 使用生产环境或类似生产的环境
- [ ] 展示用户视角，而非技术细节
- [ ] 收集待办事项的反馈
- [ ] 感谢团队的成就

### Sprint 回顾

**时长：** 2 周 Sprint 为 1.5 小时

**格式选项：**

| 格式 | 结构 |
|--------|-----------|
| 开始-停止-继续 | 要开始、结束、继续做的事情 |
| 4L | 喜欢、学到、缺乏、渴望 |
| 帆船 | 风（助力）、锚（阻塞）、礁石（风险） |
| 生气-悲伤-高兴 | 对 Sprint 事件的情绪状态 |

**行动项：**
- 每次回顾最多 2-3 个改进行动
- 指定负责人和截止日期
- 下次回顾开始时审查上次行动

## 指标与跟踪

### Sprint 指标

| 指标 | 公式 | 目标 |
|--------|---------|--------|
| 速度 | 完成的点数 / Sprint | 稳定 ±10% |
| 承诺可靠性 | 完成 / 承诺 | >85% |
| 范围变更 | 增加或移除的点数 | <10% |
| 结转 | 未完成的点数 | <15% |
| 缺陷率 | 缺陷点数 / 总点数 | <20% |

### 速度跟踪

```
Sprint 速度趋势：
Sprint 1：25 点
Sprint 2：28 点
Sprint 3：30 点
Sprint 4：32 点
Sprint 5：29 点
------------------------
平均：28.8 点
趋势：稳定（±10%）

规划建议：计划承诺 26-29 点
```

### 燃尽图

跟踪 Sprint 内进度：

```
天    理想    实际    状态
---   -----   ------   ------
 0     30      30      正常
 2     24      26      稍落后
 4     18      20      落后
 6     12      14      恢复中
 8      6       6      正常
10      0       2      少量结转
```

**燃尽模式：**

| 模式 | 含义 | 行动 |
|---------|---------|--------|
| 初期平缓 | 早期无进展 | 检查阻塞项 |
| 后期骤降 | 最后一刻完成 | 改善 WIP 限制 |
| 范围增加 | 线条上移 | 处理范围蔓延 |
| 提前完成 | Sprint 结束前完成 | 拉取延伸事项 |

### 完成定义

故事在以下条件满足时视为完成：

- [ ] 代码完成并审查通过
- [ ] 单元测试编写并通过
- [ ] 集成测试通过
- [ ] 验收标准已验证
- [ ] 文档已更新
- [ ] 已部署到测试环境
- [ ] 产品负责人已验收
- [ ] 无严重缺陷

### 发布指标

| 指标 | 定义 | 目标 |
|--------|------------|--------|
| 前置时间 | 从想法到生产 | <2 个 Sprint |
| 周期时间 | 从开发开始到完成 | <1 个 Sprint |
| 吞吐量 | 完成的故事数/Sprint | 持续增长 |
| 缺陷逃逸 | 生产环境发现的缺陷 | 持续下降 |

## 三阶段流程

```
阶段 1：需求澄清          阶段 2：方案设计          阶段 3：落地拆解
目标/约束/非目标    →    高层架构 → 关键决策    →    阶段/验收/风险
（不跳过，不假设）       （ADR + 假设标注）       （可执行计划）
```

### 阶段 1：需求澄清

必须显式确认，不允许用"一般认为"跳过：

1. **功能边界**：做什么、不做什么、边界条件。
2. **量化约束**：延迟、吞吐、可用性、成本、团队规模。给不出数字的标注为"待确认"并说明影响。
3. **既有系统**：现有架构、技术债、迁移约束。
4. **非目标**：明确排除的范围，防止 scope creep。

检查点：四个维度都有答案或标注"待确认"后，进入阶段 2。

### 阶段 2：方案设计

按固定顺序推进，每步产出对齐后再进入下一步：

1. **高层架构**：服务/组件边界、部署拓扑、数据流方向。用 ASCII 框图表达。
2. **关键决策**：每个架构选择写 ADR（Context → Decision → Consequences）。深度分析用 `architecture-decision-records`。
3. **接口契约**：API 风格、版本策略、错误码、认证模型。契约模板用 `architecture-decision-records` 的 references。
4. **数据模型**：实体关系、读写路径、事件流、存储选型。
5. **弹性策略**：故障模式、降级策略、超时/重试/熔断具体数值。深挖用 `error-handling-patterns`。
6. **假设标注**：每个关键假设标注 `[假设]` 标记，附"假设不成立时的降级路径"。

检查点：所有决策有 trade-off、所有假设有降级路径后，进入阶段 3。

### 阶段 3：落地拆解

1. **实施阶段**：拆成 2-4 个阶段，每阶段有明确验收标准。
2. **未验证项**：列出需要验证的假设和待确认的依赖。
3. **风险登记**：每个风险配缓解措施和触发条件。

检查点：每个阶段可独立验证、风险有明确兜底。

## 质量门禁

每个决策必须满足：
- 有显式 trade-off（收益 vs 代价），不用"最佳实践"替代理由。
- 区分"确定要做"和"[假设] 需要验证"。
- 给出实施阶段和验收标准，不是一坨设计文档。
- 弹性策略有具体数值（超时多少 ms、重试几次），不是"设一个合理的值"。

## 红线

- 阶段 1 未完成不准跳到阶段 2。没有量化约束的设计方案是无根方案。
- 每个架构决策必须有 Consequences 段。没有后果分析的 ADR 不算完成。
- 不写"视情况而定"——要么给具体值，要么标注 [假设] 并附降级路径。

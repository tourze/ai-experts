## 代码模式

```json
{
  "文档类型": "技术方案",
  "目标读者": ["研发负责人", "产品经理"],
  "已有素材": ["会议纪要", "旧方案", "接口说明"],
  "当前目标": "先产出目录和待确认问题清单"
}
```

推荐的迭代节奏：

```markdown
1. 收集背景与目标
2. 列出目录与缺口
3. 逐章起草
4. 读者视角复核
5. 定稿并交接
```

## 反模式

### FAIL: 原料堆砌

```
用户给 3 份会议纪要 + 1 份旧方案
你直接 concat → 输出 30 页"文档"
→ 没有结构 / 没有论点 / 读者读不下去
```

### PASS: 结构 → 缺口 → 章节

```
Step 1：先输出目录 + 每章 1 句话目的
Step 2：标待确认问题（用户回答）
Step 3：逐章起草，每章先发用户确认
Step 4：定稿前读者视角检查
```

### FAIL: 虚构背景

```
"为了显得完整" → 编"市场背景 / 用户访谈 / 历史数据"
→ 用户："这些数据从哪来的？"
→ 信任崩
```

### PASS: 显式标 [待确认]

```md
## 市场背景
- [待确认] 目标市场规模（需要用户提供 BD 调研）
- [推测] 基于公开渠道，竞品 A 占 30%（来源：xxx）
- [事实] 我方 2024 ARR：500w（来源：内部财报）
```

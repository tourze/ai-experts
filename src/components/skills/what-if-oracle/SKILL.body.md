## 代码模式

```text
问题：
变量：
幅度：
时间窗口：
当前状态：

分支 A：最可能
- 概率：
- 触发信号：
- 立即后果：
- 30 天后：
- 应对动作：

分支 B：最坏
...

综合：
- 稳健动作：多个分支都值得做的事
- 对冲动作：为坏分支做的准备
- 决策触发器：看到什么信号就切换判断
```

## 反模式

### FAIL: 含糊问题

```
"如果出事了怎么办？"
→ 出什么事？多大？多久？
→ 分支全是泛泛"加强监控"
```

### PASS: 单变量收紧

```
变量：日活用户数
幅度：±50% / ±200%
窗口：未来 90 天
当前：10,000 DAU
→ 5 条分支可对比
```

### FAIL: 只有极端

```
最好：用户暴涨 10x
最坏：服务挂掉
→ 漏了 60% 概率的"温和增长"分支
→ 准备的全是不会发生的事
```

### PASS: 覆盖最可能

```
分支 A 最可能（50%）：增长 30%
分支 B 反共识（15%）：增长但留存崩
分支 C 最坏（10%）：被竞品抢
分支 D 意外（5%）：监管变化
→ 概率合 ≈ 100%
```

### FAIL: 写成小说

```
"如果竞品发布 GPT-5 集成，市场会震动，
我们的用户会动摇，..."
→ 几页文字，没有"看到什么信号就行动"
```

### PASS: 信号 + 动作

```
分支 B 触发信号：竞品官方公告 OR 我们流失率周环比 +5pp
立即动作：48h 内启动 retention 调研
30 天动作：上线对标功能或差异化定位
```

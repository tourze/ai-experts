## 代码模式

```text
1. 澄清对比目标和用户决策场景
2. 确定维度：功能、架构、性能、生态、学习曲线、维护、许可证
3. 逐维度收集证据（代码仓读代码和 git 历史，外部产品查文档）
4. 交叉验证：查遗漏维度、定位不对称
5. 结构化输出：矩阵 + 详细分析 + 条件化建议
```

## 反模式

### FAIL: 打勾无判断

```
| 功能 | A | B |
| X   | ✓ | ✓ |
| Y   | ✓ | ✗ |
| Z   | ✗ | ✓ |
→ 用户："所以选哪个？"
→ 答："都可以"
```

### PASS: 条件化建议

```
核心差异：
- A 强项：{X 功能 + 企业生态}
- B 强项：{轻量 + 学习曲线低}

建议：
- 团队 > 20 人 + 需要 SSO / 审计 → 选 A
- 个人 / 小团队 / 快速试用 → 选 B
- 中间场景（5-20 人）：先试 B，未来可迁移
```

### FAIL: 维度不统一

```
A 的分析：功能齐全 / 文档详细 / 性能好
B 的分析：社区活跃 / 开源 / 免费
→ 各说各话，无法横向比
```

### PASS: 维度矩阵

```
| 维度 | A | B |
| 功能完整度 | 9/10 | 6/10 |
| 社区活跃 | 中等 | 高 |
| 许可证 | 商业 | MIT |
| 性能（基准测试）| 1.2x | 1.0x |
→ 同维度可比，用户可按自己权重决策
```

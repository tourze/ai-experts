## 适用场景

- 用户要求从 App Review 角度审计代码库、配置、权限或付费流程。
- 需要在提审前找出高概率拒审项、信息缺失和审核路径阻塞点。
- 需要生成审核备注、测试账号说明或提交前检查清单。

## 核心约束

- 第一轮只做审计，不直接改代码；先给出 P0 / P1 / P2 风险清单。
- 所有判断都要落到证据：文件、符号、配置项、屏幕流程或网络行为。
- 如果证据不足，要明确写出“假设”和需要补看的文件，而不是凭经验断言。
- 审核报告要同时覆盖合规、可审性和用户体验，不只盯 `Info.plist`。

## 代码模式

### 审核切入点

```text
1. 识别 App 核心用途与首屏路径
2. 检查权限文案、隐私清单、第三方 SDK、登录与删除账号
3. 检查 IAP / 订阅 / 恢复购买 / 外链支付
4. 检查空状态、崩溃、离线、演示账号与 reviewer notes
```

### 风险登记表

```text
P0 | 权限/支付/登录 | 发现 | 为什么可能被拒 | 证据 | 建议 | Effort | Confidence
P1 | 技术稳定性     | 发现 | 为什么会卡审核   | 证据 | 建议 | Effort | Confidence
P2 | 体验与说明     | 发现 | 为什么会引发追问 | 证据 | 建议 | Effort | Confidence
```

### Reviewer Notes 草稿

```text
- 首次启动后如何进入核心功能
- 审核账号 / 演示数据如何使用
- 哪些权限是延迟请求、为什么需要
- 如何验证订阅、恢复购买或受限内容
```

## 检查清单

- 优先检查 `Info.plist`、`*.entitlements`、`PrivacyInfo.xcprivacy`、StoreKit / 订阅代码、登录删除账号流程。
- 重点排查：权限描述缺失、恢复购买缺失、第三方登录未覆盖 Sign in with Apple、外部支付引导、空白页 / 死路。
- 输出必须按优先级排序，并附上可执行的验证步骤。
- 如果需要真实走一遍审核路径，可配合 `ios-simulator-skill` 复现 UI 流程。
- 交叉引用：界面合规与平台习惯看 `ios-hig-design`；门店更新文案看 `app-store-optimization`。

## 反模式

### FAIL: 泛泛而谈

```md
## 审核风险报告
- 注意隐私
- 注意订阅
- 注意权限说明
→ 客户："具体哪里？"
```

### PASS: 证据驱动的风险表

```md
| P | 类别 | 发现 | 证据 | 拒审依据 | 建议 |
|---|------|------|------|---------|------|
| P0 | 权限 | 相机缺描述 | Info.plist:23 缺 NSCameraUsageDescription | Guideline 5.1.1 | 加 "扫描发票二维码" |
| P0 | 订阅 | 缺恢复购买 | 未搜到 SKPaymentQueue.restore | Guideline 3.1.1 | Settings 加恢复按钮 |
| P1 | 登录 | 仅 Google 登录 | LoginVM.swift:12 | Guideline 4.8 | 添加 Sign in with Apple |
```

### FAIL: 第一轮直接 patch

```
"我修了相机权限描述，PR 已开"
→ "还有别的吗？" → "让我再扫一遍..."
→ 漏掉了支付/删除账号 P0
```

### PASS: 先审再修

```
Phase 1: 全量审计 → P0/P1/P2 风险表
Phase 2: 客户定优先级
Phase 3: 按序修复，每项独立 PR
```

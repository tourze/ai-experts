# Whole Product 设计

## 适用场景

- 主流客户不只要核心功能，还需要集成、迁移、模板、培训和成功保障。
- 早期产品在 demo 中有效，但客户落地时阻力大。
- 需要让 beachhead 市场相信“这套方案能完整解决我的问题”。

## 组成部分

| 层级 | 内容 |
| --- | --- |
| 核心产品 | 真正解决核心痛点的功能 |
| 必要集成 | 客户当前工作流中的关键系统 |
| 模板/配置 | 行业专属表单、报表、流程、规则 |
| 服务支持 | onboarding、迁移、培训、office hour |
| 信任资产 | 案例、合规、SLA、安全文档、ROI 计算 |

## 检查问题

- 客户从购买到首次成功需要哪些步骤。
- 哪些步骤如果没有服务支持会失败。
- 哪些集成是 beachhead 客户共同要求。
- 销售承诺是否已经被产品、文档和交付能力覆盖。

## 输出要求

- 区分首版 whole product 和后续增强。
- 明确哪些内容由产品实现，哪些由服务或运营补足。
- 写出缺口和补齐优先级。
